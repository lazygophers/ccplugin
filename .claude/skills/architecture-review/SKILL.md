# 架构评审（Architecture Review）

本 Skill 提供系统化的架构评审，从宏观设计到微观实现，评估系统的可扩展性、可维护性和健壮性。

## 概览

**核心能力**：
1. **架构分析** - 分层架构、模块划分、依赖关系、接口设计
2. **设计模式** - 识别设计模式、评估应用合理性、反模式检测
3. **质量属性** - 可扩展性、可维护性、可测试性、性能、安全性
4. **架构债** - 设计债、技术债、架构腐化识别
5. **改进建议** - 重构路径、技术选型、架构演进

**评审维度**：
- **结构清晰度**：分层是否清晰、模块职责是否单一
- **耦合度**：模块间依赖是否合理、是否存在循环依赖
- **可扩展性**：是否易于添加新功能、是否遵循开闭原则
- **可测试性**：依赖注入、接口抽象、单元测试覆盖
- **性能设计**：缓存策略、异步处理、数据库优化

## 执行流程

### 阶段1：需求理解

**目标**：明确评审范围和关注点

**步骤**：
1. 使用 `AskUserQuestion` 询问以下信息（如果用户输入不明确）：
   - 评审范围：「整个系统」「特定模块」「接口设计」
   - 关注点：「可扩展性」「性能」「安全」「可维护性」「全部」
   - 项目阶段：「设计阶段」「开发阶段」「重构阶段」
   - 输出格式：「技术报告」「架构图」「改进清单」

2. 收集项目信息：
   - 项目类型（Web应用、微服务、CLI工具）
   - 技术栈（语言、框架、数据库）
   - 业务规模（用户量、数据量、QPS）
   - 团队规模（开发人数、技术水平）

### 阶段2：架构分析

**目标**：分析现有架构设计

**步骤**：
1. **分层架构分析**：
   - **表现层**：UI组件、路由、状态管理
   - **业务层**：业务逻辑、领域模型、用例
   - **数据层**：数据访问、ORM、缓存
   - **基础设施层**：日志、监控、配置

2. **模块划分分析**：
   - 模块职责是否单一（单一职责原则）
   - 模块边界是否清晰
   - 模块间通信方式（直接调用、事件、消息队列）

3. **依赖关系分析**：
   - 使用 `serena:find_referencing_symbols` 分析依赖
   - 识别循环依赖
   - 评估依赖方向（高层依赖低层，符合依赖倒置原则）

4. **接口设计分析**：
   - API设计是否符合RESTful规范
   - 接口粒度是否合理
   - 是否有版本管理
   - 错误处理是否统一

### 阶段3：设计模式评审

**目标**：评估设计模式应用

**步骤**：
1. **识别设计模式**：
   - **创建型**：Singleton、Factory、Builder、Prototype
   - **结构型**：Adapter、Decorator、Proxy、Facade
   - **行为型**：Strategy、Observer、Command、Template Method

2. **评估合理性**：
   - 模式应用是否恰当
   - 是否过度设计
   - 是否有更简单的替代方案

3. **反模式检测**：
   - **God Object**：单个类承担过多职责
   - **Spaghetti Code**：代码结构混乱
   - **Lava Flow**：死代码、过时代码
   - **Golden Hammer**：过度依赖某一技术
   - **Copy-Paste Programming**：大量重复代码

### 阶段4：质量属性评估

**目标**：评估系统的质量属性

**步骤**：
1. **可扩展性**（0-10分）：
   - 是否易于添加新功能（开闭原则）
   - 是否支持插件化
   - 是否有清晰的扩展点

2. **可维护性**（0-10分）：
   - 代码可读性（命名、注释、结构）
   - 模块化程度
   - 测试覆盖率

3. **可测试性**（0-10分）：
   - 依赖注入
   - 接口抽象
   - 单元测试覆盖率≥80%

4. **性能**（0-10分）：
   - 响应时间
   - 吞吐量
   - 资源使用

5. **安全性**（0-10分）：
   - 认证授权
   - 数据加密
   - 输入验证
   - OWASP Top 10 合规

### 阶段5：架构债识别

**目标**：识别架构层面的技术债

**步骤**：
1. **设计债**：
   - 违反SOLID原则
   - 循环依赖
   - 过度耦合

2. **技术债**：
   - 过时的技术栈
   - 缺少文档
   - 测试覆盖率低

3. **架构腐化**：
   - 分层被破坏（跨层调用）
   - 模块边界模糊
   - 新功能难以添加

4. **优先级排序**：
   ```
   优先级 = 严重度(0-10) × 影响范围(0-10) × 紧急度(0-10)
   ```

### 阶段6：生成评审报告

**目标**：提供可操作的改进建议

**步骤**：
1. **架构概览**：
   - 分层图
   - 模块依赖图
   - 数据流图

2. **质量评分**：
   - 可扩展性：X/10
   - 可维护性：X/10
   - 可测试性：X/10
   - 性能：X/10
   - 安全性：X/10
   - 综合得分：X/10

3. **问题清单**：
   | 序号 | 问题 | 类型 | 优先级 | 影响范围 | 建议方案 |
   |------|------|------|--------|----------|----------|
   | 1 | 循环依赖 | 设计债 | 900 | 核心模块 | 引入中间层 |

4. **改进建议**：
   - 短期（1-2周）：高优先级问题修复
   - 中期（1-2月）：架构重构
   - 长期（3-6月）：技术栈升级

## 设计原则

### SOLID 原则

- **S - Single Responsibility**（单一职责）：一个类只负责一个职责
- **O - Open/Closed**（开闭原则）：对扩展开放，对修改关闭
- **L - Liskov Substitution**（里氏替换）：子类可以替换父类
- **I - Interface Segregation**（接口隔离）：客户端不应依赖不需要的接口
- **D - Dependency Inversion**（依赖倒置）：依赖抽象而非具体实现

### 其他原则

- **DRY**（Don't Repeat Yourself）：避免重复代码
- **KISS**（Keep It Simple, Stupid）：保持简单
- **YAGNI**（You Aren't Gonna Need It）：不做过度设计
- **LoD**（Law of Demeter）：最少知识原则

## 最佳实践

- **自顶向下分析**：先宏观架构，再微观实现
- **结合业务场景**：架构服务业务，不能脱离业务
- **量化评估**：使用数值评分（0-10分）
- **优先级排序**：基于严重度、影响范围、紧急度
- **可操作建议**：提供具体的改进路径和时间规划
