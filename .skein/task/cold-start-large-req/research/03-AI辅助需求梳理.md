# 03 AI agent 辅助需求梳理现状 (2025-2026)

## 维度概述

经 agent-reach **dev (gh CLI)** 勘察, 2025-2026 已有多个公开 AI skill 仓把「fuzzy/vague 需求 → 可执行 spec」做成显式 skill。最完整的是 **liza-mas/liza** 仓 (vision→epic→story→spec 四层全链路, 含 anti-patterns 表 + size 门)。社区共识 (Twitter/GPT-5.6 prompt 指南讨论) 倒向「**给 agent 清晰任务契约 + 验收边界, 别管死每步流程**」—— 这正是冷启动应产出的形态。

> 数据源声明: 本维主要经 agent-reach dev 分类 (`gh api` / `gh search code`) 取 GitHub 一手仓 + WebSearch 补社区视角。Twitter 社区视角见 §条目 6。

## 矛盾点 (保留不和稀泥)

- **「轻 prompt 重契约」 vs SKEIN 重流程 (brainstorm+grill+contract)**: 社区 (GPT-5.6 指南) 说模型强了就别管死每步, 但 SKEIN grill 硬门正是「强制对抗审查」。**不冲突而是分工**: 轻契约是「产出口」, grill 是「验证口」—— 模型变强让契约本身可以更轻 (少写废话多写验收), 但 grill 作为「用户想法=PRD 写的」的最后一道对抗校验仍有价值, 只是 grill 该校验契约清晰度而非流程细节。
- **liza 四层 (vision→epic→story→spec) vs SKEIN 两层 (task→subtask)**: liza 是为「长周期 spec-driven 项目」设计的多文档体系; SKEIN 是「planning 尽快转 exec」的轻编排。直接搬 liza 四层会违反 SKEIN「探索封顶, 尽早转异步」铁律。**取舍**: 借 liza 的「技术」(SMARC / 双受众 / anti-patterns / size 门) 而非「结构」(四文档体系)。

## 对 SKEIN 的适用性标注

| 条目 | 适用性 | 说明 |
|---|---|---|
| liza SMARC + 双受众 + anti-patterns | **适配 (技术直接用, 结构不搬)** | SMARC 作 AC 校验标准; 双受众改写 prd.md 结构 (Part1 用户确认 + Part2 exec 契约); 6 类 anti-patterns 作 grill 的 cold-start 专项审查轴 |
| liza size 门 / capability 定义 | **适配 (与现有天花板合并)** | 复合嗅味文本判据 + capability≠模块 提醒, 塞进现有「复杂度天花板」表作 cold-start 行 |
| Plan/Agent 两段式 (tanure) | **适配 (SKEIN 本就有)** | SKEIN 的 planning(=Plan Mode) vs exec(=Agent Mode) 天然两段, 只是 planning 缺「模糊愿景入口」, 补这一档即可 |
| IDD intent 单一真值源 | **适配 (已内置)** | SKEIN task.json 已是单一真值源; IDD 的「意图漂移」问题在 SKEIN 由 prd.md + grill contract 抑制, 无需新增 |
| Anthropic 官方 cold-start skill | **不适用 (无)** | 官方仓无此 skill, 说明 SKEIN 自建 cold-start 档无重复造轮子之忧, 是差异化机会 |
