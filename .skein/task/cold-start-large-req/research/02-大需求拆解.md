# 02 大需求拆解模式 — epic→story→task / domain decomposition / strangler fig / vertical slice

## 维度概述

大需求拆解有两条正交轴: **纵向 (功能/能力域)** 和 **横向 (交付顺序/架构层)**。核心共识 (多源一致): 先切**纵向薄切片端到端打通** (vertical slice / walking skeleton), 而非先铺完一层再铺下一层; 对存量系统的「重构整个 X」类需求, **strangler fig 增量替换**优于大爆炸重写。liza 仓给出了 vision→epic→story→spec 的完整 AI 可执行四层拆解树, 阈值「3-8」与 SKEIN 复杂度天花板 (>8 拆多 task) **同源**。

## 矛盾点 (保留不和稀泥)

- **vertical slice (按行为纵切) vs SKEIN 的「契约 subtask 先行」(按技术契约横切)**: SKEIN 现有铁律是「先定共享契约 subtask, 下游实现 subtask 并行」(横向先定接口)。vertical slice 要求「先端到端纵切一片」。两者在 DAG 形状上冲突: vertical slice 第一个 task 是「一条完整链路」, SKEIN 第一个 task 是「一份契约定义」。**取舍**: 对「加新能力」(greenfield 段) 用 vertical slice 思想定第一个 task (先走通最薄路径); 对「重构存量」(strangler 场景) 用 SKEIN 契约先行 (先定新旧边界契约)。两种场景不同入口。
- **capability (行为切片) vs SKEIN subtask (可独立验收的工作单元)**: liza 强调 capability 是用户行为非技术模块; SKEIN subtask 定义偏「可独立验收」。对重构类需求, 按 capability 拆会得到用户可见行为, 按可验收工作单元拆可能退化成技术模块切分。需在 cold-start 档显式提醒「按用户行为拆, 勿按技术层拆」。

## 对 SKEIN 的适用性标注

| 模式 | 适用性 | 说明 |
|---|---|---|
| Epic→Story→Task 三层 | **需改造 (折叠一层)** | SKEIN 只有 task→subtask 两层。cold-start 档可把 epic 层折叠成 prd.md 的「能力域」章节标题, story 层 = SKEIN task, task 层 = SKEIN subtask。不新造 epic 机器 |
| liza size 门 (3-8 / 复合嗅味 / capability≠模块) | **适配 (直接用)** | 这三条是 cold-start 判「是否过宽/该按什么边界拆」的现成启发。复合嗅味 ("X and Y and Z") 可作自动文本判据。与现有「subtask>8 拆多 task」无缝叠加 |
| Strangler Fig | **适配 (heavy 档补强)** | 对「重构整个权限系统」这类, 在 breaking-refactor 基础上补「路由 facade + 分批绞杀」结构化建议: 第一个 task = 建 facade + 定边界契约, 后续 task 逐块替换, 末 task = 退役旧路径 |
| Vertical Slice | **适配 (greenfield 段)** | cold-start 拆完能力域后, 第一个 task 选「贯穿所有层的最薄端到端」作 walking skeleton, 验证假设; 之后再按层补全 |
| Walking Skeleton | **适配 (与复杂度天花板联动)** | cold-start 产出的第一个 task 强制是 walking skeleton (端到端最薄), 而非把所有能力域铺平。这是「模糊愿景→先验证一个闭环」的关键 |
| IDD intent-as-contract | **需改造** | 思想同源 (单一真值源 = SKEIN task.json), 但 IDD 把意图直接变可执行断言太重; 取其「意图先沉淀成契约再编码」内核 = SKEIN 的 prd.md 契约 + grill 硬门, 不引入新执行层 |
