# 01 需求发现 / 需求工程 — 从模糊愿景到结构化需求

## 维度概述

业界对「用户给的是模糊愿景/痛点而非任务」早有成熟方法族。核心共识: **先收敛「为什么/为谁/价值」(outcome), 再谈「做什么」(solution)**。适合 AI agent 引导式追问的共性是: 结构化填空模板 (Job Story / OST) + 分层逐步揭示 (vision→outcome→opportunity→solution), 而非重型多日工作坊。

## 矛盾点 (保留不和稀泥)

- **OST vs Job Story 的起点不同**: OST 从「业务 desired outcome」起 (自上而下, 偏组织视角); Job Story 从「用户情境的进展卡点」起 (自下而上, 偏用户视角)。两者并非互斥但对冷启动入口不同 — 若用户说"我们要加 AI"(组织诉求) 走 OST 起点更顺, 若用户说"用户老来抱怨 X"(用户痛点) 走 Job Story 起点更顺。**不能预设一个起点硬套**。
- **Now-Next-Later 与 SKEIN DAG 的张力**: Now-Next-Later 是「时间承诺度」分桶, SKEIN 是「依赖关系」串 DAG。Later 桶里的东西在 SKEIN 里可能根本没有 task (尚未可执行) — 直接套会破坏 SKEIN「task 必须有可拆 subtask」约束。

## 对 SKEIN 的适用性标注

| 方法 | 适用性 | 说明 |
|---|---|---|
| Job Story 模板 | **适配** (直接用) | brainstorm 第一问的填空骨架: 把"加 AI 能力"→"When 用户在 X 情境, 我想要 Y, so I can Z"。是冷启动入口的现成句式 |
| OST 四层 | **需改造** | SKEIN 无「experiments」层概念; outcome/opportunity/solution 三层可映射到 vision→epic→story, 但「experiment」需折叠进 subtask 的验收 checklist |
| said/implied/missing 三分 | **适配** (直接用) | 冷启动追问的核心技术: 让 main 显式标出用户的 missing, 逐条 Open Question 而非脑补 |
| SMARC | **适配** (直接用) | 替代/强化现有 brainstorm 的「验收基准」—— 每条 AC 须可测, 不可测 = 未定, 停在 planning。与 SKEIN「未跑 grill 禁进 exec」同向 |
| Now-Next-Later | **部分适配** | 只取「先切出 Now (可立即拆 task 的)」这一刀, 用作从模糊愿景里剥出第一个 task。Later 桶落成 prd.md 的「非目标/未来」段, 不建 task |
| Story Mapping | **需改造** | SKEIN 是 DAG 非 backlog; 但「backbone = walking skeleton 先行」的思想可直接用: 冷启动先拆出端到端最薄路径作第一个 task |
