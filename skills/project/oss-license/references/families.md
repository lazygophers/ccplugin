# 协议家族与代表协议详解

按"传染强度"从弱到强排列。传染强度 = 用了你的代码后、对方被迫开源的范围。

## 1. Permissive(宽松型)

**契约**: 几乎只要求保留版权与许可声明; 允许闭源、商用、再许可。被采用阻力最小, 生态默认首选。

| 协议 | 核心义务 | 专利条款 | 何时选 |
| --- | --- | --- | --- |
| `MIT` | 保留版权与许可声明 | **无显式专利授权** | 小项目 / 库, 生态惯例 |
| `BSD-2-Clause` | 保留声明 | 无 | 等价 MIT |
| `BSD-3-Clause` | + 禁用作者名背书 | 无 | 需防"拿作者名背书"时 |
| `Apache-2.0` | 保留声明 + NOTICE + 标注修改 | **有显式专利授权 + 专利报复终止** | 企业级 / 框架 / 要专利防御 |
| `ISC` | 等价 MIT, 措辞更简 | 无 | OpenBSD 生态默认 |
| `0BSD` / `Unlicense` | 近乎零条件(准公有) | 无 | 想连署名都不要 |

**MIT vs Apache-2.0 的关键抉择**: Apache-2.0 第 3 条授予专利许可, 且含"专利报复"——谁对该软件发起专利诉讼, 其专利授权即终止。MIT/BSD 对专利只字不提, 理论上留有"贡献者日后用专利反诉使用者"的风险。**企业项目、可能涉专利的领域选 Apache-2.0; 极简小库选 MIT/ISC。**

**Rust 范式**: `Apache-2.0 OR MIT` 双授权——既得 Apache 的专利防御, 又保留 MIT 选项以便和 GPLv2 代码组合(Apache-2.0 与 GPLv2 不兼容)。

## 2. 弱 copyleft(文件级 / 库级传染)

**契约**: 改动**被许可的那部分**须开源; 但允许被更大的(含闭源)作品链接 / 组合。

| 协议 | 传染粒度 | 要点 |
| --- | --- | --- |
| `LGPL-2.1` / `LGPL-3.0` | **库级** | 动态链接调用方可保持闭源; 修改 LGPL 库本身须开源。静态链接须让用户能替换该库 |
| `MPL-2.0` | **文件级** | 改了的 MPL 文件须开源该文件; 新增文件可任意授权。与 GPL **兼容**(含次级许可机制) |
| `EPL-2.0` | 模块级 | 含专利授权; 可指定 "Secondary License" 实现 GPL 兼容(EPL-1.0 则与 GPL 不兼容) |
| `CDDL-1.0/1.1` | 逐文件 | 类 MPLv1; **与 GPL 不兼容**(ZFS 不能进 Linux 内核的根因) |

**LGPL 静态 vs 动态链接(核心陷阱)**: 动态链接 → 闭源程序可调用、无需开源。**静态链接 → 必须提供方式让用户替换该库**(交付 .o 目标文件, 或改用动态链接), 否则触发更强义务。选 LGPL 依赖做闭源产品时, 优先动态链接。

## 3. 强 copyleft(项目级传染)

**契约**: 分发衍生作品时, **整个作品**须以同协议开源。

| 协议 | 要点 |
| --- | --- |
| `GPL-2.0-only` | 锁 v2; 与 Apache-2.0、GPLv3 **均不兼容** |
| `GPL-2.0-or-later` | 允许升级到 v3, 用以化解 v2/v3 冲突——**默认选这个** |
| `GPL-3.0` | 新增反 Tivoization + 专利条款; 与 GPLv2-only 不兼容 |
| `OSL-3.0` | 含网络分发触发; 与 GPL 不兼容 |
| `EUPL-1.2` | 欧盟官方, 23 语言等效, 覆盖 SaaS; 有独特附录兼容机制(见 compatibility.md) |
| `CECILL-2.1` | 法国法律框架, **显式声明 GPL 兼容** |

**GPLv2 vs GPLv3 为何不兼容**: GPLv2 含"不得附加额外限制"条款; GPLv3 新增的专利与反 Tivoization 条款, 在 GPLv2 眼中即"额外限制"。`-or-later` 变体允许升版本来化解。

## 4. 网络 copyleft

| 协议 | 要点 |
| --- | --- |
| `AGPL-3.0` | 在 GPL-3.0 基础上堵 "SaaS 漏洞": **通过网络提供服务也视同分发**, 须向用户提供源码 |

**为何存在**: GPL 仅在**分发二进制**时触发开源义务。云厂商把软件跑成 SaaS、不分发二进制即可绕开。AGPL 第 13 条把"网络交互提供服务"也定义为触发点——这是开源项目对抗"云厂商白嫖"的主要合规武器(Redis 2025、Elastic 2024 回归开源均选 AGPL)。

## 5. 源码可见 / 非 OSI(source-available)

> ⚠️ 这些**都不是开源协议**(OSI 未认证), 不是自由软件。源码可见 ≠ 开源。宣传时不可称"开源"。

| 协议 | 机制 |
| --- | --- |
| `BUSL-1.1`(Business Source License) | 加非竞争限制; N 年后(默认 4 年)自动转为指定 Change License(常为 Apache/MPL) |
| `SSPL-1.0` | AGPL + 额外: 提供服务须开源整个服务**管理栈**。OSI 2019 审查中被撤回 |
| `Elastic-2.0` | 禁作为托管服务转售; 禁绕过授权密钥 |
| `FSL-1.1-MIT` / `FSL-1.1-Apache-2.0` | 类 BSL 但非竞争期仅 **2 年**, 到期转 MIT/Apache |
| `Confluent-Community` | 禁作为 SaaS 提供 |
| `PolyForm-*` | 模块化限制系列(非商业 / 小企业 / 防竞争) |
| Commons Clause | **附加条款, 非独立协议**; 包在 MIT/Apache 上加"禁销售", 一旦附加即令原协议**不再开源** |

商业动机与 relicensing 案例史见 `references/commercial.md`。
