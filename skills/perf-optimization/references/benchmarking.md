# 基准测试与压测(防优化器作弊 + 防协调遗漏)

两类最常见的"测了个寂寞":① 微基准被编译器/JIT 优化扭曲;② 压测因协调遗漏低估尾延迟。

## 微基准:防优化器作弊

**核心铁律:被测的结果和输入都必须被"消费",否则编译器/JIT 合法删掉它**(死代码消除 DCE)。再加:预热到稳态、做统计而非单点。

| 语言 | 防优化 API | 说明 |
|------|-----------|------|
| Java | `Blackhole.consume(x)`(JMH) | 消费结果防 DCE;`@State` 管理共享状态防伪共享 |
| Go | `for b.Loop()`(1.24+) | 自动 `KeepAlive` 防 DCE + 自动管理计时;旧写法用包级 `var sink T` 接收结果 |
| Rust | `std::hint::black_box(x)` | 阻止编译器优化掉被测代码;criterion 已内置 + 完整统计(预热/采样/离群检测) |
| C++ | `benchmark::DoNotOptimize(x)` + `ClobberMemory()` | DoNotOptimize 保护结果(注意输入也要保护防常量折叠如 `10+20→30`) |

常见陷阱:
- **死代码消除(DCE)**:结果没被消费 → 编译器删掉计算,benchmark 假快 8-12×。
- **常量折叠**:常量入参在编译期算完,运行时啥也没干。
- **JIT 预热**:稳态前测量,多种执行模式被平均成无意义数。
- **`volatile` 是粗糙手段**:能挡部分优化但语义不精确,优先用专用 API。

## 压测:防协调遗漏(Coordinated Omission)

**问题**:闭环压测器(等上个请求返回才发下个)在系统卡顿 5s 时也停发请求,结果只记到 1 个坏样本,p99 报 1ms 却完全失真。测量系统无意中与被测系统"协同",漏测离群值。

**修正**:用**开环负载生成**(恒定速率发请求,不管前一个是否完成):wrk2 先行实践,k6/Vegeta/autocannon 跟进。

**度量**:用 HDR Histogram 捕获完整分布,报 p99/p999/p9999/p100,而非均值。"Outliers are the norm"——大规模系统里磁盘抖动/上下文切换/cache miss/偶发锁竞争在 1/1000 概率上累积,足够多次必然出现。
