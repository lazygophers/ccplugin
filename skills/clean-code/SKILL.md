# Clean Code 代码整洁顾问

> 基于 Robert C. Martin《代码整洁之道》(Clean Code) 全书 17 章 + 附录 A + 66 条坏味道/启发式清单 (ch17)。
> 一手语料：github.com/glen9527/Clean-Code-zh（中英双语译本，调研时间 2026-07-01）。

## 何时用 / 何时不用

**用**：
- 贴一段代码问「有什么问题 / 怎么改 / 帮我重构」→ **Mode B 审查**
- 描述需求要「帮我写 X」/ 给草稿求整洁化 → **Mode A 顾问**
- 命名纠结、函数过长、注释该不该写、参数过多 → 查「规则体系」对应章节

**不用**（走别的 skill）：
- 架构选型 / 系统设计 → `architecture-design`
- 性能优化 / 瓶颈分析 → `perf-optimization`
- 重写 / 现代化遗留系统 → code-modernization 系列

## 核心哲学（5 条，全书立论）

1. **童子军军规**：检入代码比检出时更干净 (ch1.6)。不必大重构，每次小改善即可。
2. **读 >> 写**。「我们是作者」——代码被读的次数远多于被写 (ch1.5)。为「读得顺」优化，非为「写得快」。
3. **简单设计四原则**（Kent Beck，按重要性递减，ch12）：
   1. 跑通所有测试
   2. 无重复
   3. 表达意图
   4. 最少类与方法（这条是刹车，防过度抽象）
4. **整洁代码只做一件事**（Bjarne Stroustrup 定义，ch1.3.5）：优雅、逻辑直接（bug 难藏）、最小依赖、完整错误处理、性能近最优。
5. **混乱复利**：糟糕代码「引诱」人越改越烂 (ch1.3)；整洁是阻断混乱复利的唯一壁垒。

### 命名 (ch2 · N1-N7)

| 规则 | 一句话 | 编号 |
|------|--------|------|
| 名副其实 | 名到即意，无需注释补全 | 2.2 |
| 避免误导 | 别用 `accountList` 若非 List 类型；别用相近拼写的名 | 2.3 / N4 |
| 有意义区分 | `a1/a2`、`ProductInfo/ProductData` 是无意义区分 | 2.4 |
| 读得出来 | `genymdhms` 不行，`generationTimestamp` 行 | 2.5 |
| 可搜索 | 单字母名只在短作用域；长名配长作用域 (N5) | 2.6 / N5 |
| 避免编码 | 拒匈牙利记号 / 成员前缀 `m_` (N6)；接口别加 `I` 前缀 | 2.7 / N6 |
| 一概念一词 | `fetch/get/retrieve` 选一个不混用 | 2.12 |
| 不双关 | `add` 表「加值」就别用于「加项」（用 `append`） | 2.13 |
| 解决方案领域名优先 | CS 术语可用（`AccountVisitor`）；问题领域名次之 | 2.14 / 2.15 |
| 名描述副作用 | `getName` 不该藏连接副作用 (N7) | N7 |
| 别扮可爱 | `HolyHandGrenade` 不如 `DeleteItems` | 2.11 |

### 函数 (ch3 · F1-F4, G30/G34)

- **短小** (3.1)：20 行内为佳，缩进 ≤ 2 层。函数超长 = 抽象层混乱信号。
- **只做一件事** (3.2 / G30)：函数内若能再分「区段」→ 它做了多件事，拆。
- **一层抽象** (3.3 / G34)：不混高层概念与实现细节。自顶向下读如散文（Stepdown Rule）。
- **switch 多态替代** (3.4 / G23)：switch 散落 = 违反 OCP，用多态收口。
- **参数** (3.6 / F1)：
  - 理想 0 > 1 > 2；3 个以上用参数对象包装 (3.6.5)
  - **禁 flag 参数** (3.6.2 / F3)：布尔入参 = 函数做两件事，拆成两个函数
  - 一元函数用动词+名词 `writeField(name)`；二元用动词 `assertExpectedEqualsActual`
- **无副作用** (3.7 / F2)：`checkPassword` 不该顺带 `initializeSession`。输出参数 (F2) 是副作用气味。
- **命令查询分离** (3.8)：`set` 改状态 / `get` 读状态，不混。函数要么做事要么回答事，不同时。
- **异常优于错误码** (3.9)：抽 try/catch 成独立函数（错误处理本身即「一件事」）。
- **去重复** (3.10 / G5)：DRY 是简单设计第 2 原则。

### 注释 (ch4 · C1-C5)

> 注释不能美化烂代码 (4.1)。能用代码表达就别注释。

**好注释** (4.3)：法律信息 / 提供信息 / 解释意图 / 阐释 / 警示后果 / TODO / 放大意义 / 公共 API Javadoc。

**坏注释** (4.4)：
- 喃喃自语 (4.4.1)、多余 (C3: `i++; // increment i`)、误导、循规式、日志式、废话 (4.4.6)、位置标记 (4.4.9)、归属署名、**注释掉的代码** (C5: 别留，源码控制会记)
- 不恰当信息 (C1: 变更历史、作者、SPR 号不进注释——那些归 issue 跟踪系统)
- 过期注释 (C2) 比没注释更糟
- **能用函数或变量替代就别注释** (4.4.8)

### 格式 (ch5)

- **垂直开放** (5.2.2)：概念间空行分隔
- **垂直靠近** (5.2.3)：相关代码紧贴，别隔空行
- **垂直距离** (5.2.4)：变量近声明处；私有变量类顶；私有函数紧跟首次调用处；概念相关者靠近
- **横向密度** (5.3.1)：空格隔运算符与参数
- **缩进** (5.3.3)：永不违反，违反 = bug 温床
- **团队规则 > 个人风格** (5.4)：统一比「正确」重要

### 数据、对象、类、系统 (ch6 · ch10 · ch11)

- **数据抽象** (6.1)：隐藏实现，别只加 getter/setter 暴露内部
- **对象 / 数据反对称** (6.2 / G6)：对象藏数据露行为；数据结构露数据无行为。别造「混杂体」。
- **得墨忒耳律** (6.3 / G36)：避免 `a.getDept().getManager().getName()` 火车失事；模块不该知它操作对象的内部
- **DTO** (6.4)：纯数据传送对象 OK；Active Record 是混合体，慎用
- **类** (ch10)：单一职责（SRP）、高内聚、低耦合、为扩展开放 / 为修改关闭（OCP）；类应短，像函数一样
- **系统** (ch11)：构造与使用分离；依赖注入；延迟决策到需要时；决策局部化

### 错误处理与边界 (ch7 · ch8)

**错误处理**：
- 异常而非返回码 (7.1)
- 先写 try-catch-finally (7.2)：先定错误边界再填正常路径
- 不可控异常优先 (7.3)：受检异常违 OCP
- 异常带上下文 (7.4)：操作 + 失败类型 + 堆栈
- 依调用者需要定义异常类 (7.5)
- **特例模式** (7.6)：定义常规流，把异常路径封装成特例对象
- **不返回 null** (7.7)：返 Optional / 空集合；null 强迫每个调用方防御
- **不传 null** (7.8)

**边界** (ch8)：
- 第三方代码用 adapter 包裹，污染不外溢
- 学习性测试锁版本（> 免费，升级时跑得通）
- 边界处封装「尚不存在的代码」用接口占位

### 测试 (ch9 · T1-T9)

**F.I.R.S.T 原则**：
- **F**ast 快：慢测试会被跳过
- **I**ndependent 独立：测试间无依赖
- **R**epeatable 可重复：任何环境都能跑
- **S**elf-validating 自验证：布尔结果，不需人工读日志
- **T**imely 及时：先于或同时写产品代码

**其他**：
- 单一断言优先（概念清晰）；Given-When-Then 结构
- 测边界条件 (T5) / bug 附近穷尽测 (T6) / 失败模式揭示问题 (T7) / 覆盖率模式揭示遗漏 (T8)
- 别跳过平凡测试 (T3)；被忽略测试 = 对歧义点的提问 (T4)
- 覆盖率工具必用 (T2)；**测试要快** (T9)
- 测试代码与产品代码**同等整洁**（脏测试 = 测试本身成负担）
- 测试充分性 (T1)：覆盖所有失败路径而非「百分比覆盖率」

### 并发 (ch13 · 附录 A)

- **并发分离原则**：并发代码与主逻辑解耦，分离线程
- 限定临界区；数据封装；线程尽量不可变
- 避免死锁：固定锁顺序
- 并发缺陷难复现 → 用学习性测试与压力测试锁行为

## 双模式工作流

收到代码或需求后，先判模式：

| 用户输入特征 | 模式 | 入口 |
|------|------|------|
| 贴现成代码 + 「有什么问题 / 审查 / review / 重构」 | **Mode B 审查** | 见下 |
| 描述需求 / 给草稿 / 「帮我写 / 整洁化」 | **Mode A 顾问** | 见下 |
| 单点疑问（命名、参数数、要不要注释） | 直接查「规则体系」 | — |

### Mode A：写代码顾问

1. **先写测试用例**（定义「做对」的形状）——遵循 FIRST + 单概念
2. **最简实现跑通**（绿）——不追求一步到位
3. **重构**：套规则体系逐项整洁化，顺序固定：
   - 命名 → 函数粒度（拆/参数对象/去 flag）→ 去重复（G5）→ 命令查询分离 → 去注释（能用代码替代的删）→ 格式
4. **输出**：整洁代码 + 关键决策的规则依据。例：
   > 「参数包成 `SearchCriteria` 对象 (ch3.6.5)，因为 4 参数 > 上限；flag `isQuick` 拆成 `quickSearch()` / `fullSearch()` 两个函数 (F3)。」

### Mode B：代码审查

1. **逐行扫**，按 ch17 坏味道清单匹配违规，标编号（G5 / N4 / F3...）
2. **按严重度排序**：
   - 🔴 **必修**：违反 SRP / 重复 (G5) / 误导命名 / 返回 null (7.7) / 死代码 (G9) / 注释掉的代码 (C5) / 特性依恋 (G14)
   - 🟡 **应改**：长函数 / 深缩进 / 注释可被代码替代 / 边界行为错 (G3) / 命令查询混用
   - 🟢 **可选**：格式 / 命名口味 / 空行间距
3. **每条给三件**：违规编号 → 问题定位（行号/片段）→ 改写示例
4. **输出格式**：

## 反模式与例外（别教条）

- **不过度抽象**：YAGNI。简单设计第 4 原则「最少类与方法」是刹车——别为「可能将来」造接口/工厂。
- **规则服务可读性，非反之**：若遵守某规则反而更难读，破规则并留注释解释。
- **性能热点可破规则**：明确标注「此处为性能让步」+ 注释说明。但热点需先测后断（perf-optimization），勿臆测。
- **领域语言优先**：领域有约定术语时从领域，即使与通用规则冲突（N3 标准术语优先）。
- **原型 / 一次性脚本 / 即将废弃代码** 不必求整洁——童子军军规针对要留的代码。
- **测试亦有度**：穷举测一切 = T1 反面（insufficient 才是问题），覆盖率 100% 不等于无 bug。

## 诚实边界

- **语言偏向**：原书以 Java 为示例语言。规则适用于多数面向对象语言；函数式语言（Haskell/纯 Scala）、系统语言（Rust/C）部分规则需按语言机制翻译——尤其错误处理 (ch7)、并发 (ch13)，当代语言有更优原生（`Result`/`Option`/所有权/`async`）。
- **时代**：原著 2008 年。`Optional`、记录类、不可变数据、纯函数等当代语言原生支持，本书未覆盖；可作补充而非替代。
- **不替架构判断**：整洁是代码层。系统架构（ch11 仅入门，依赖注入/延迟决策）需另参 `architecture-design`。
- **不替性能判断**：用 `perf-optimization`。
- **66 条坏味道非穷尽**：是高频项，非完整 Design Smells 体系（参 Fowler《Refactoring》）。
- **不预测作者立场**：本 skill 只引原书已写规则，不揣测 Uncle Bob 对当代新议题（AI 生成代码、微服务）的态度。
- **案例章 (ch14-16)** 为逐步重构演示，本 skill 提炼规则不照搬其长示例；需查原书案例时引章节号。
