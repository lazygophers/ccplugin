# 进阶: 全局功能 / 架构范式 / 边界审视 / 老化重构 / 共识确认

> 正交分解之外的五个进阶主题。前四步工作流(SKILL.md)是「怎么分」, 本文是「分不动时怎么办 / 旧系统怎么救 / 怎么积累范式」。

## A. 全局性功能 / 横切关注点

- **定义**: 难以剥离成独立模块的功能, 特征是「每加一个新功能都要动它」——如存盘/读盘、Undo/Redo、宏录制。
- **真·全局 vs 伪·全局(判据)**: 鉴权、日志看似全局, 但能在**统一入口处理**或**只提供辅助函数** → 不算全局功能, 别特殊对待。只有需求本身交织的真全局功能才需特殊处理。
- **处理范式: 反向依赖**。常规思路是「为全局功能建框架, 注入核心系统」; 更好的做法是反其道——**抽象核心系统的接口(如 IO DOM), 让全局功能反向依赖这些接口**。这样核心系统受到的伤害值最低。
- **Undo/Redo 洞见**: 不要在「问题发生的地方」解决。Undo/Redo、镜像、异步存盘本质都与「数据多版本」相关 → 引入**数据层(DataLayer)**, 多版本能力一并解决 Undo/Redo、快照、异步。
- 信条: **「全局性功能难, 但不代表可以放弃正交分解。不能放过任何一块难啃的骨头。」**

## B. 架构范式: 设计场景优于设计模式

- **范式 ≠ 设计模式**。架构师真正的武器库是「**一个个业务只读、接口稳定、易于组合的模块 + 组合的方法论**」, 不是抽象原则。
- **「设计场景」优于「设计模式」**: 设计场景有清晰的问题域定义, 是实实在在的通用子系统; 设计模式脱离问题域谈就是空的。**架构师成熟度的核心标志 = 能把业务场景分解为多个「通用设计场景」的组合。**
- 「积木」哲学: 软件可「搭积木」搭出来; 工程效率的真正来源 = 业务分解能力 + 历史积累的可复用模块。
- 经典原则可作索引(是地图不是武器, 熟读它们不足以成为优秀架构师): ISP 接口隔离 / DIP 依赖倒置 / ADP 无环依赖 / CARP 组合优于继承 / HCLC 高内聚低耦合 / COC 惯例优于配置 / CQS 命令查询分离 / SOC 关注点分离。

## C. 边界要「不断重新审视」

- **核心方法**: 在业务分解中反复审视模块接口, 发现其中「过度的/多余的」约束, 把它提高到足够通用、普适的场景来看。
- **可操作流程(强判据)**: 架构设计时, **过一遍所有用户故事(User Story)确认架构适应性**。示范: 先漏了分页文档 → 补 View DOM; 又漏了剪贴板 → 把 `*os.File` 提升为 `io.Reader/Writer`; 入口参数从 `string` 提升到通用介质类型。**「如果做需求分析时没把这些需求关联性找到, 那就不是一次合格的需求分析」。**
- 边界不是一次划定永久不变; 重新正交分解是常态。

## D. 架构老化与重构

- **老化的标志(可操作诊断)**: 添加功能越来越难、迭代效率降低、解决一个问题却生出好几个新问题、功能代码逸出框架散落各处「散发臭味」。
- **老系统加新功能(三模块切分)**: 任何周边系统 B 与核心系统 A, 理想切成三块——`A`、`B(与 A 无关部分)`、`A↔B 桥接代码(与 A 相关部分)`。桥接代码尽可能小且独立 →「保护投资, 未来重构成本最小化」。
- **局部优化两手法**:
  1. **局部重构/重写**: 彻底移除该功能代码重写。前提是**你已对这块业务足够了解**(维护过一阵), 且必须清干净老代码。
  2. **依赖优化**(做「代码搬运工」, 不改业务逻辑): 把散落的修改集中成 `doXXX_yyyy` 命名的函数(故意取丑名, 标记「此处待重新考虑边界」)。**有多少个 `doXXX_yyyy` 就有多少对核心系统的伤害**。同一处若多个功能都加了 `doXXX_yyyy` → 信号: 这里该提供**事件机制**让它们监听 → 核心更稳定, 逼近 OCP。
- **重构不要盲目**: 应基于「伤害值」客观判断, 而非主观「看不顺眼就重构」。
- **核心系统重构的顺序(可操作六步)**:
  1. 不要一上来重构核心系统(牵一发动全身、风险高)。
  2. 最高优先级 = **先确定核心系统的边界(使用界面/接口)**, 最好不改实现就能把使用界面调整到期望样子。
  3. 对所有周边模块做依赖优化 → 初步确定核心系统该暴露的**事件集合**。
  4. 把核心系统 DOM 接口抽象出来: 先做「实现依赖 → 接口依赖」转变(周边依赖接口而非实现), 此时即可 mock 核心对接周边。
  5. **重新设计核心系统接口**——难点是业务理解要真有长进 + 对周边切换新接口的成本有充分预计(过渡期不能太长)。
  6. 接口改造完成后, 核心与每个周边完全独立, 可单独优化甚至重写核心(「要做的事很收敛, 不影响大局」)。
- 信条: **「架构的功夫全在平常。架构能力靠日常一点一滴反思打磨, 不能指望某天突飞猛进。」** 重构比全新业务的架构更难——是架构设计 + 资源规划 + 阶段规划 + 持久战韧性的综合工程。

## E. 架构 = 共识确认过程

- **「架构过程是团队共识形成与确认的过程, 共识必须精确、无歧义」**。不精确的架构(只有架构图)→ 风险被掩盖到最后联调才爆, 动作走形变成打补丁。
- **「连接性代码越少越好」**: 把两个子业务连起来的代码若多, 应抽象出更大范畴的新业务系统 → 焦点始终在业务上。
- **「抓住稳定的东西比追逐变化更重要」**: 业务语义是稳定的, 抽象符合业务自然语义的接口, 远比抽象泛需求的框架容易。
