# UI/UX 设计原则（无关平台）

所有视觉 / 交互产出都受这些原则约束。跨 HTML / App / CLI / TUI 通用。信息架构见 [information-architecture.md](information-architecture.md)，交互见 [interaction-design.md](interaction-design.md)，可用性见 [usability-a11y.md](usability-a11y.md)。

## 视觉原则

### 层级（Hierarchy）

引导视线从最重要到次要。工具（从强到弱）：

1. **尺寸**：差距 ≥1.5× 才读得出
2. **留白**：相关贴近、不相关拉开（格式塔接近性）
3. **对比**：色 / 字重 / 背景差异
4. **位置**：左上 > 右下（阅读起点 > 终点）

自检：眯眼看，焦点是否落点在你想要的位置？多个等权元素互相打架——让一个赢。

### 对比（Contrast）

- 文字与背景对比 ≥ 4.5:1（正文）/ ≥ 3:1（大字）
- 不只靠颜色传信息（色盲），加图标 / 文字 / 纹理冗余
- 重点用对比突出，别处处强调（处处强调 = 无强调）

### 对齐（Alignment）

- 元素沿统一参考线对齐（左 / 网格 / 基线）
- 对齐产生秩序感，不对齐显乱
- 禁随机偏移，每个位置可解释

### 接近性（Proximity）

- 相关元素贴近成组，不相关拉开距离
- 组比元素先被识别（格式塔）
- 间距尺度见各媒介 layout 文件

### 一致性（Consistency）

- 同一动作 / 概念在全局用同一表现（按钮、图标、色、文案）
- token 化：值取自固定阶，禁魔法数
- 内部一致 > 跟随外部潮流

### 留白（Whitespace）

- 留白是设计元素，不是「空」
- 信息塞满 = 无焦点
- 段落级留白分隔区块，元素级留白分组

## 交互原则

### 反馈（Feedback）

每个用户动作都有即时反馈：

- 视觉：hover / press / focus 态变化
- 状态：loading / success / error 明确
- 时效：<100ms 感知即时，>1s 需进度提示

无反馈 = 用户怀疑「是不是没点上」。

### 容错与防错（Error Prevention & Tolerance）

- 防错：危险操作要确认；约束输入（格式、范围）
- 容错：错误可恢复（撤销）、数据不丢
- 错误信息说人话：什么错 + 为什么 + 怎么修（见 [interaction-design.md](interaction-design.md)）

### 可控（User Control）

- 用户可撤销 / 取消 / 返回
- 状态可见：用户随时知道在哪、能做什么
- 不强制流程，允许跳过 / 返回

### 可识别优于回忆（Recognition over Recall）

- 选项可见（菜单 / 工具栏）优于要求记忆命令
- 常用操作显式可见，长尾可藏（命令面板 / 搜索）

### 效率（Efficiency）

- 新手有引导，专家有快捷键
- 高频操作可批量化 / 一键化
- 别为新手友好牺牲专家效率（两者并存）

## 原则冲突时的优先级

| 场景 | 让位 |
|------|------|
| 一致性 vs 平台习惯 | 平台习惯赢（用户预期） |
| 美观 vs 可用 | 可用赢 |
| 信息密度 vs 可读 | 可读赢，密度靠交互补（展开 / 分页） |
| 创新 vs 一致 | 一致赢，创新限制在非关键路径 |

## 自检清单

- [ ] 视觉层级焦点唯一
- [ ] 对比达标（含无障碍）
- [ ] 对齐统一、接近性正确
- [ ] 一致性（token 化）
- [ ] 每个动作有反馈
- [ ] 危险操作防错、错误可恢复
- [ ] 用户可控（撤销 / 返回）
