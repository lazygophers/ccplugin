# 八个心智模型(详解)+ 决策启发式

> 正交分解架构方法论的核心心智模型。每个含定义、应用方式、**失效边界**。

## 模型 1: 正交分解(Orthogonal Decomposition)

- **定义**: 架构 = 业务的正交分解。把业务系统拆成相互独立、互不耦合的模块, 每个模块有自己的业务(「模块」泛指: 函数/类/接口/包/子系统/网络服务/桌面程序)。降复杂度的根本手段是分解, 标准是「正交」——改 A 不影响 B。
- **乘法不是加法**: 正交分解是**做乘法**——用互相正交、零相关的模块「乘」出业务场景; 而继承是做加法(靠叠加能力把一个模块改造成另一个)。「组合优于继承」的本质就是「乘法优于加法」。
- **应用**: 拿到任何系统, 第一步不是设计模块, 而是问「哪些子业务彼此正交」。三步曲(需求分析→概要设计→详细设计)都是同一件事——把正交分解从模糊推进到精确无歧义。
- **信念与边界**: 「任何功能都可正交分解; 暂时找不到方法, 只是还没透彻理解需求」——这是工作信念, **不是已证事实**。失效边界: 子业务间存在本质不可消除的耦合(强一致事务、共享状态)时, 强行正交化会制造虚假边界; 横切关注点(日志/鉴权)用「全局性功能」专门处理(见 `decomposition-and-refactoring.md` §A), 不塞核心。

## 模型 2: 最小核心 + 正交周边(Minimal Core + Orthogonal Periphery)

- **定义**: 应对业务变化最好的结构 = 「最小化的核心系统 + 多个彼此正交的周边系统」。核心构成业务的**最小功能集**, 复杂度通过「往周边加功能」而非「改核心」增长。
- **核心纯洁性纪律(重要)**: **核心系统一定要最小化、要稳定; 坚持不往核心系统加新功能, 业务架构就不会有臭味。保持核心系统的纯洁性比什么都重要。** 新功能若早期没规划、后期却被界定为核心, 必须认真评估它对既有架构的破坏性。
- **周边内聚**: 为降低一个功能的负面影响, 其相关代码首先要尽可能内聚——代码量小可不独立成模块, 但**一定要写到独立文件**里。
- **失效边界**: 核心边界划错(把易变的放进核心)会令核心频繁改动, 放大模型 3 的伤害。核心「最小化」不能小到不完备(配接口的最小依赖原则)。

## 模型 3: 修改影响面量化(Modification Blast Radius)★含精确公式

- **定义**: 架构好坏可量化——衡量「新增/变更一个功能时需要修改的处数与代码量」。改动越分散、改的行数越多, 架构越差。
- **核心系统伤害值公式**: `Σ_(每一处修改) log₂(修改行数 + 1)`。
  - 同一周边功能相邻的代码行算**一处**; 不同周边功能即使相邻也算**多处**。
  - 含义: **修改处数越多伤害越大**; 鼓励每处尽量压到只改 1 行, 更多代码推到周边模块自己那里。这是「最重要的测量公式」——核心越干净, 加新功能越容易。
- **模块耦合度公式**(同时适用接口与实现): 对所依赖模块 A 的耦合度 = `Σ_(每个依赖符号) log₂(符号出现次数 + 1)`(「符号」= 被引用的类型/全局变量/全局或成员函数)。模块总耦合度 = `Σ_A (耦合度_A × 不成熟度系数_A)`(系数: 完全成熟不再变 = 0, 规格未定剧烈变动 = 1 → **鼓励依赖成熟模块**)。
- **三条约束(量化时必须意识到)**:
  1. 把参数类型改成 `interface{}`/`object` **不能降低耦合度**——实际使用时所有可能类型都计入依赖。
  2. 该值**无物理含义**, 只能对比「相同功能」的两个方案; 不同功能的系统不可比。
  3. 公式只覆盖**静态依赖**(调用接口的数量), 未覆盖**动态依赖**(运行时调用次数)——这是盲区。
- **应用**: 评审时做思想实验——「加一个典型新功能, 要碰几个文件、改几行核心代码?」理想 = 核心 0 改动(或 ≤1 行)+ 新增一个独立周边文件。
- **失效边界**: 纯追求「核心零改动」可能催生过度抽象的插件框架(YAGNI 风险)。该度量衡量「应对预期变化」的成本, 对未预见的根本性需求变更无能为力(那属重构, 见 §D)。

## 模型 4: 接口 > 数据结构 > 架构图

- **定义**: 「比框架(架构图)更重要的是数据结构, 比数据结构更重要的是接口」。
  - 数据结构 > 架构图: 业务数据结构是架构实现机制的灵魂, 是比框架更重要的共识。
  - 接口 > 数据结构: **架构是团队共识确认的过程, 共识必须精确无歧义**; 架构图不精确, 接口(用代码表达)才精确。架构图只是辅助说明模块接口之间的关联。
- **反模式**: 「用实现替代业务」——定义了数据结构却不抽象业务逻辑, 直接让使用方操作成员变量或堆一堆 get/set; 用 ORM 时直接操作数据结构而不定义业务接口。
- **验证手段(可操作)**: 为证明接口有效, 在**概要设计阶段就用真实代码(非伪代码)把所有用户故事串一遍**——「代码即文档, 代码是理解一致性更强的文档」。这把联调风险前置, 是管理工程最大风险的手段。
- **失效边界**: 探索期/需求极不确定时, 过早冻结接口会锁死错误抽象。接口稳定性以「业务理解已收敛」为前提。

## 模型 5: 开闭原则应对变化(OCP)

- **哲学**: OCP 推崇**模块业务的确定性**。可以改 Bug, 但不要随意调整模块的业务范畴(加/减功能都不鼓励)。
- **「只读」模块**: 与其修改模块的业务, 不如实现一个新业务——OCP 鼓励写「只读」的业务模块, 一经设计不可改, 要改就废弃它、转而实现新模块(类比 Git 版本管理、容器化服务治理, 都靠「只读」设计降低治理难度)。
- **不限于 OOP(纠偏)**: OCP 是「信息技术架构」的基本原则。CPU 即范例——指令稳定但指令序列可变; 中断机制就是 CPU 引入的回调函数。
- **两层落地**:
  1. 模块业务要稳定 → 遵循「只读」设计, 要变就归档放弃(「每个模块都应该是可完成的」)。
  2. 把业务变化点抽离: 简单的用**回调函数/接口**交给其他模块; 复杂的引入**插件机制**分解为「最小核心 + 正交周边」。回调/接口本质是事件监听, 是插件机制的特例。
- **与 SRP 的关系**: SRP 强调「每个模块只做一个业务」, OCP 强调「把变化点抽离外包」——同一问题的两个面。
- **失效边界**: OCP 依赖「变化点预判准确」。预判错误会让扩展点变成无人使用的复杂度负债。故需配模型 6。

## 模型 6: 预判什么不会发生(反过度设计的缰绳)

- **过度设计的精确定义**: **「不会发生的事情你考虑了、并且为它做足了准备, 就是过度设计」**。
- **「如无必要, 勿增实体」**: 实现依赖大部分情况应**直接依赖组件、不抽象**。大量抽象 = 可配置性更好但学习成本更高; 简单又不失扩展性的平衡完全依赖你对业务的理解。
- **何时才该抽象依赖(可操作三判据)**: 仅在以下三种情况——
  1. 需要提供多种选择时(如 Logger 把决策权交使用方);
  2. 需要解除一个庞大外部系统依赖时(测试 mock);
  3. 依赖的外部系统是可选组件时(默认装 mock, 用户不关心就当它不存在)。
  - 反例: 为「MySQL/MongoDB 自由切换」而抽象数据库依赖, 绝大多数是过度设计——选数据库本应是谨慎严谨的行为。
- **应用**: 留扩展点之前, 先逐条写下「哪些维度未来一定不变」, 钉死进核心简化设计; 只为「理性可判断会变」的维度留接口。
- **失效边界**: 判断「什么不会变」是经验性、概率性的, 在新领域/快速演化领域误判率高; 成熟、约束清晰的领域最可靠。

## 模型 7: 需求分析——三要素 + 真伪需求

- **需求分析要回答的问题清单**(比「三要素」更全):
  1. 面向的**核心用户人群**是谁?
  2. 用户**原始需求**是什么? 最核心的几个问题是哪些?
  3. 已有哪些玩家/上下游? 之前用户怎么解决? 我们的替换方案是什么?
  4. 我们创造的**价值点**、用户最关注的**核心指标**是什么?
  5. 需求的**变化点与稳定点**分别在哪?
- **稳定点 / 变化点**: 稳定点往往是系统核心能力/核心价值(→ 核心); 变化点要做开放性设计(→ 周边/接口)。**关键约束: 必须有明确的参考坐标系——站在「我们要设计的产品」角度判断**(设计计算机时外设是变化点, 设计显示器时问题域完全变了)。
- **真伪需求判据**: 用户反馈需求时往往**已带着自己的解决方案**(二次加工的需求)。要**刨根问底还原到不带任何技术假设的「根源需求」**——一个根源需求可由很多技术方案满足。难点是「小绿点」需求: 不做可能丢客户, 做了产品又被放大成「四不像」。
- **收敛性判据**: 「产品功能必须是收敛的、可完成的」。某子类需求若**呈发散无法收敛**, 团队必须坐下来反复推敲响应姿势。
- **失效边界**: 三要素/问题清单是脚手架, 不替代领域知识; 对高度模糊、用户自己都说不清的探索性需求, 需先用原型/实验暴露真问题再套用。

## 模型 8: 全栈构建——基础架构 + 业务架构

- **定义**: 架构师能力底座 = 三大技能(理需求 / 读代码 / 抽象系统); 终极视野是「在心中重建整个系统」, 从底层硬件到顶层业务每层都是决策。
- **武器库的全栈结构**(四层): 基础平台(硬件体系/编程语言/操作系统)→ 桌面开发平台(窗口/GDI/浏览器·小程序 + MVC)→ 服务端开发平台(负载均衡 + 存储中间件)→ 服务治理平台(容器/编排生态)。**「基础架构 + 业务架构 = 你设计的软件的全部」**——别只盯业务架构而忘了基础架构选型。
- **应用**: 做业务架构时也要能向下追问「这层抽象在底层如何落地」; 上层决策的合理性根植于对底层机制的理解。
- **失效边界**: 全栈纵深成本极高, 对多数工程师是「理想态」而非准入门槛; 是修炼方向, 非每个具体决策的必经步骤。

## 决策启发式(如果 X → 则 Y)

1. **要降复杂度** → 先正交分解、再设计模块(做乘法); 目标是「改 A 不影响 B」, 不按技术分层切。(模型1)
2. **新功能要改核心多处** → 坏设计信号; 用伤害值 `Σ log₂(改行数+1)` 度量, 把功能代码内聚到独立文件/周边, 核心改动压到 ≤1 行。(模型3)
3. **面对变化点** → 优先「新增只读业务模块」而非「改现有模块」; 简单变化点用回调/接口, 复杂的上插件。(模型5)
4. **留扩展点前犹豫** → 先答「这维度未来一定不变吗」; 只在「多选择/解除重依赖/可选组件」三种情况才抽象依赖, 否则直接依赖。(模型6)
5. **评估是否想清楚** → 别看架构图, 看有没有用真实代码把所有用户故事串通。(模型4)
6. **看到一堆 get/set 或外部直接操作成员变量** → 警惕「用实现替代业务抽象」; 接口必须反映业务逻辑。(模型4反模式)
7. **接口拿不准** → 分两种语义: 使用界面走 KISS+符合惯例, 环境依赖走最小依赖原则(见 `interface-design.md`)。
8. **判断需求该不该做** → 还原成「根源需求」(剥掉用户自带的技术方案); 发散无法收敛的需求要重新推敲。(模型7)
9. **依赖谁** → 先评「模块规格的成熟度」, 鼓励依赖成熟模块(不成熟度系数低)。(模型3/模型6)
10. **架构已老化(改动处处掣肘、散发臭味)** → 基于伤害值客观判断是否重构(非主观厌恶); 核心系统重构走六步(见 `decomposition-and-refactoring.md` §D)。(模型3)
