# 接口设计

> 正交分解方法论中的接口设计准则。接口是模块正交性的落地载体, 也是团队精确共识的载体。

## 表达优先级: 接口 > 数据结构 > 架构图

「比框架(架构图)更重要的是数据结构, 比数据结构更重要的是接口。」

- **架构图**: 不精确、有歧义, **只能辅助说明**模块接口之间的关联, 不能当团队共识。
- **数据结构**: 业务数据结构是架构实现机制的灵魂; 从共识角度比框架图更重要。
- **接口**: 最高。架构是**团队共识确认的过程, 共识必须精确无歧义**; 接口(用代码表达)才精确。接口 = 需求的流程分解, 体现「用户地图」; 是业务的抽象, 也是模块与使用方的耦合契约。

**实操结论**: 评审「设计是否完成」, 不看 PPT 架构图, 看是否产出了**精确的接口定义 + 核心数据结构**。在概要设计阶段就应**用真实代码(非伪代码)把所有用户故事串一遍**, 以证明接口有效——把联调风险前置。

## 两种接口语义, 两套准则(核心框架)

接口按语义分两种, 各有准则:

### 1. 模块的使用界面(规格 / 公开类·函数原型)→ KISS 原则

- KISS = Keep It Simple, Stupid, 「让傻子也能看懂」, 追求**简单、自然、符合惯例**。
- **符合语言与社区的惯例**: 若某类业务在语言中已有约定俗成的接口, 尽可能沿用相同语义, 避免「惊异(surprise)」。
- **「简单」不是外观简洁, 而是业务语义表达的准确无歧义**。判据: SAX/事件模型 vs DOM 模型——事件模型看似简洁高效, 但心智负担大(大量缓存冗余代码)、不体现业务、必须配文档才能用; DOM 模型虽冗长, 却能脱离文档零心智负担使用, 才是该追寻的接口描述方式。

### 2. 模块的环境依赖(模块间契约 / 对依赖环境的抽象)→ 最小依赖原则(LKP)

- 最小依赖 / 最少知识原则(Least Knowledge Principle): 尽可能**发现并消除模块中多余的依赖**。
- 环境依赖再分两种:
  - **使用界面依赖**(用接口时自然涉及的): 对参数做泛化抽象以适应更广场景。例: 把 `*os.File` 换成 `io.Reader/io.Writer` 获得通用性、支持剪贴板/内存介质。
  - **实现依赖**(当前实现方案才引入的, 换实现就可能消失): 大部分情况应直接依赖组件、不抽象(见 mental-models 模型6「何时才抽象依赖」三判据)。
- **陷阱**: 形式上更泛化 ≠ 实际更通用。`interface{}` 参数若内部靠反射做了强假设, 反而是「场景收紧」; 用更恰当的通用接口才是「恰如其分的描述」。

## 四准则助记(实质映射)

「最小完备 / 自然正交 / 易用 / 隐藏实现」是常用助记口诀, 实质映射到上面两套准则:

| 助记 | 实质 |
| --- | --- |
| **易用 / 自然** | 使用界面侧的 KISS + 符合惯例 + 业务语义无歧义 |
| **最小完备 / 隐藏实现** | 环境依赖侧的最小依赖原则(LKP)+ 契约式设计(只暴露业务、不暴露实现) |
| **自然正交** | 接口之间彼此独立、不重叠职责——模块正交性在接口上的体现 |

## 接口反模式(禁)

| 反模式 | 病因 | 纠正 |
| --- | --- | --- |
| 定义一堆 `get/set`, 让外部直接读写成员变量 | 「用实现替代业务抽象」; 接口反映内存布局而非业务 | 接口方法应是**业务动作**(如 `placeOrder`), 不是字段存取 |
| 接口跟着实现走(先写类再抽接口) | 实现细节渗进契约, 换实现就破坏接口 | 接口跟着**业务**走; 先想清业务抽象再定接口 |
| 用 `interface{}`/反射「假泛化」 | 形式泛化但内部强假设 = 场景收紧 + 耦合度未降 | 用恰当的通用接口(如 `io.Reader`)做「恰如其分的描述」 |
| 为臆想的切换(MySQL↔MongoDB)抽象依赖 | 不会发生的泛化 = 过度设计 | 只在「多选择/解除重依赖/可选组件」三情况抽象 |
| 用架构图当接口契约去对齐团队 | 架构图有歧义, 共识是假的 | 用代码/接口签名 + 真实代码串用户故事做精确共识 |

## 接口与正交分解的关系

接口是正交分解的**落地载体**: 模块之间的正交性, 最终体现为接口的正交性。设计时:

1. 正交分解出子业务(模型1) → 每个子业务一个模块
2. 模块的**稳定点**藏进实现, **变化点**通过接口/回调暴露(模型5)
3. 模块间只通过接口耦合, 不共享内部状态 → 保证「改 A 不影响 B」
4. 用「修改影响面」自检(模型3): 加新功能时, 理想是新增一个实现了既有接口的周边模块, 核心接口与核心代码不动
