# Scoring Matrix — 通用维度评分表

> 配套主 SKILL.md Phase 1「基线评分」。主文件讲 4 阶段流程；validation-gate.md 讲验证命令；本文件只讲**打分标准**：6 个通用维度怎么打分、plugin/skill/agent 三类的差异化映射、权重表、诚实边界。
> 维度软对齐 darwin-skill 9 维 rubric 精简（禁 import 禁调脚本，仅借鉴共性）。

## 1. 6 个通用维度（每维 1-10 分）

打分给**可操作的判定锚点**，禁「好/差」抽象判定。每维附 1-3 分（差）/ 4-7 分（中）/ 8-10 分（好）判定线。

### D1 触发准确（trigger-accuracy）
组件能否被**该触发的信号**触发、**不该触发的**不触发。
- 1-3：description / argument-hint 缺失或与 body 语义错位；该触发不触发 / 大量误触发。
- 4-7：能触发主路径，边缘信号漂移（模糊 prompt 路由不稳）。
- 8-10：should-trigger 全命中 + should-not-trigger 全不命中（validation-gate.md §2 测试集判定）。

### D2 结构清晰（structure-clarity）
阶段 / 章节 / 步骤的层级与导航是否能让执行者 30 秒内定位到下一步。
- 1-3：无章节 / 平铺一锅炖 / 硬规混在散文里难找。
- 4-7：有章节但层级混乱（同级标题深浅不一 / CHECKPOINT 不醒目）。
- 8-10：硬规置顶、阶段编号一致、CHECKPOINT 醒目、路由表/失败模式表结构化。

### D3 失败模式覆盖（failure-coverage）
显式列出**触发了怎么办**（触发 → 一线修复 → 兜底），非只讲 happy path。
- 1-3：无失败模式段。
- 4-7：列了常见失败但缺兜底或归因不完整。
- 8-10：触发 → 一线修复 → 仍失败兜底 三列齐全 + 反例段（命中即流程错误）。

### D4 可执行具体性（actionable-specificity）
指令是否落到「具体命令 / 具体路径 / 具体判定线」，禁「视情况 / 适当优化」。
- 1-3：通篇原则口号，无可执行步骤。
- 4-7：有步骤但部分留模糊（「跑测试」「看输出」未指定命令 / 判定线）。
- 8-10：每步有可复制命令或可判定条件（行号引用 / 阈值 / 文件路径）。

### D5 冗余度（redundancy）— 低分好，反向计分
重复段落 / AI 腔废话 / 为凑分注水膨胀。
- 1-3（冗余严重）：同一段话出现 ≥2 次、整段客套、注水凑分。
- 4-7：偶有重复但可接受。
- 8-10（精简）：无重复、无废话、文件大小 ≤ 原始（改后 ≤ 原 ×1.5 硬护栏见 validation-gate.md §3）。

### D6 可验证性（verifiability）
改前 / 改后能否用同一组 prompt / 命令客观对比，非「我觉得更好」。
- 1-3：无验证路径，纯主观。
- 4-7：能跑质量门但无 before/after 对比集。
- 8-10：有 should-trigger / should-not-trigger / edge 三段式测试集 + 独立子 agent 盲评（validation-gate.md §2）。

## 2. 三类目标映射（同维度 × 组件类型差异化判定）

| 维度 | plugin | skill | agent |
|---|---|---|---|
| D1 触发准确 | marketplace 标题 / description 命中预期；hook 触发条件正确 | frontmatter `description` + `argument-hint` 与 body 语义对齐 | `name` / `description` / tools 路由能让上层正确选用 |
| D2 结构清晰 | manifest + 各组件 README 导航清晰 | frontmatter → 硬规 → 阶段 → 资源 层级一致 | agent.md 系统提示结构清晰，角色/工具/边界分段 |
| D3 失败模式覆盖 | hook 失败 / 接线断 / 版本不匹配 兜底 | 触发漂移 / 误触发 / 端点抖动 兜底 | 工具不可用 / 权限缺失 / 上下文超限 兜底 |
| D4 可执行具体性 | manifest 路径 + 接线**双向核对**（路径 → 文件 / 文件 → 路径） | references 路径可达 + 每步可复制命令 | tools / skills 引用正确（路径存在 + 语义匹配） |
| D5 冗余度 | manifest 与各组件 README 不重复说明 | SKILL.md 与 references 去重（主文件讲流程，ref 讲细节） | agent.md 不重复所引 skill 的内容 |
| D6 可验证性 | 各组件单独可测 + 整体接线可测 | 三段式测试集 + 独立子 agent 盲评 | 触发上层调用 + 输出符合角色预期 |

> D4 的三类差异是核心：plugin 看「接线双向核对」，skill 看「references 路径可达」，agent 看「tools/skills 引用正确」——同一维度，判定对象不同。

## 3. 权重表（总分 100）

| 维度 | 权重 | 为什么这个权重 |
|---|---|---|
| D1 触发准确 | **25%** | 触发错则全错，最高优先（该来时不来 / 不该来时来 是最致命退化） |
| D4 可执行具体性 | **20%** | optimize-any 核心是「可验证改动」，具体性决定能否客观判 Δ |
| D3 失败模式覆盖 | **15%** | 区分「能用」与「可靠」，但非致命维度 |
| D2 结构清晰 | **15%** | 影响执行效率，但结构可由内容补回 |
| D6 可验证性 | **15%** | 决定能否套 validation-gate 循环，但依赖外部 harness 可用性 |
| D5 冗余度 | **10%** | 反向护栏（防膨胀凑分），低权重但作硬门（> 原 ×1.5 拒提交） |

**总分公式**：`Σ(Di 分数 × Wi) / 10 = 0-100`（Di 1-10，Wi 百分比，归一到 100）。

**ratchet 判定**：改后总分 **严格 >** 改前才留（呼应 validation-gate.md §4 + 主文件硬规 2）。相等或更低 → `git revert HEAD`。

## 4. 诚实边界（fine-grained 不可信）

- **fine-grained 分数差异不可信**：LLM-as-judge 对细粒度排序准确率 ~46%（darwin 实证，乐观偏差 + 位置偏差）。**禁用 +1 / +2 的细微差异作 merge 依据**。
- **gross 信号可信**：总分 Δ > 5（方向性改善）作粗判可接受；Δ < 5 不构成「严格更好」证据，需独立子 agent 复评或换真实 harness 实测。
- **重要决策必须人审**：破坏性变更（触发词 / 大范围重写 / merge）禁「分数涨了就 merge」，走主文件硬规 2 的 before/after 对比 + 用户确认。
- **实测 > 静态打分**：本表是静态评分，validation-gate.md 的质量门 + before/after 实测**优先于**本表分数。两者冲突以实测为准。
