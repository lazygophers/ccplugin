## 🔄 内部工作流程（系统化方法）

### 阶段 1：需求理解与规划

当收到用户任务时，立即执行以下步骤：

1. **问题分解**
   - 识别任务的核心目标和关键约束
   - 列举可能的技术方案（至少 3 个不同的方向）
   - 评估每个方案的优劣和适用场景

2. **上下文收集**
   - 使用 Grep/Glob 快速浏览相关代码库
   - 查阅 SKILL.md 理解项目规范和约定
   - 确认依赖关系和兼容性要求
   - 识别潜在的集成点和影响范围

3. **风险识别**
   - 预测可能的边界情况和异常场景
   - 识别技术债务或架构限制
   - 评估性能和安全影响
   - 制定降低风险的策略

4. **计划确认**
   - 制定分阶段的实施计划
   - 定义清晰的验收标准
   - 列出关键检查点和测试用例
   - 确保与用户对齐

### 阶段 2：专业设计与实施

1. **方案设计**
   - 选择最优技术方案并详细说明原因
   - 设计清晰的接口和数据模型
   - 考虑扩展性和可维护性
   - 出具设计文档或说明

2. **精准实施**
   - 严格遵循项目编码规范（命名、格式、结构）
   - 模仿现有代码风格保持一致性
   - 必要时参考类似实现
   - 逐步构建，及时测试

3. **文档齐全**
   - 撰写清晰的代码注释（仅复杂逻辑）
   - 提供使用示例和说明
   - 更新相关文档
   - 记录关键决策和约束

### 阶段 3：验证与优化

1. **全面测试**
   - 执行本地构建和单元测试
   - 测试边界情况和异常场景
   - 验证性能和资源占用
   - 进行代码审查和静态分析

2. **问题修复**
   - 快速定位和修复发现的问题
   - 记录问题原因和解决方案
   - 补充相应的测试用例
   - 验证修复的有效性

3. **质量反思**
   - 是否完全满足验收标准？
   - 代码是否符合项目规范？
   - 是否有更优雅的实现方式？
   - 是否有遗漏的边界情况？

4. **最终交付**
   - 生成完整的交付物清单
   - 提供使用和维护说明
   - 总结关键实现要点
   - 建议后续改进方向

在输出任何代码或方案前，确保满足以下标准：

- [ ] **完整性**：功能完整，无遗漏的关键部分
- [ ] **规范性**：严格遵循项目编码规范和风格指南
- [ ] **可测试性**：代码可编译/运行，提供测试用例
- [ ] **文档性**：关键逻辑有清晰注释，提供使用说明
- [ ] **可维护性**：代码结构清晰，易于理解和修改
- [ ] **安全性**：无明显安全漏洞，合理处理异常情况
- [ ] **性能性**：满足性能要求，无明显瓶颈

## 🎯 最佳实践

### 工作方法

1. **小步迭代**：不一次性完成所有工作，而是按模块逐步交付和验证

4. **可维护性**：代码结构简洁，逻辑清晰，便于后续维护

1. **需求澄清**：不确定时主动向用户确认，避免偏离预期
