# 接口设计(许式伟式)

> 出自《架构课》lesson 59《少谈框架, 多谈业务》、lesson 63《接口设计的准则》及方法论篇。原话经第三方笔记转载, 见 `sourcing.md`。

## 表达优先级: 接口 > 数据结构 > 架构图

「比框架(架构图)更重要的是数据结构, 比数据结构更重要的是接口。」

- **架构图**: 不精确、有歧义, **只能做辅助说明**, 不能当团队共识。
- **数据结构**: 「业务数据结构是架构实现机制的灵魂」; 从共识角度比框架图更重要。
- **接口**: 最高。接口 = 需求的流程分解, 体现「用户地图」; 是业务的抽象, 也是模块与使用方的耦合契约。

**实操结论**: 评审「设计是否完成」, 不看 PPT 架构图, 看是否产出了**精确的接口定义 + 核心数据结构**。许式伟主张系统设计阶段也要产出代码(伪代码/接口签名), 因为只有代码是精确无歧义的。并要求架构师**以伪代码或流程图把所有用户故事过一遍**, 以证明接口有效。

## 接口设计四准则

| 准则 | 含义 | 自检问题 |
| --- | --- | --- |
| **最小完备** | 接口要最小, 但要完备——能覆盖业务所需, 不多不少 | 有没有冗余方法? 有没有缺失导致调用方要绕路? |
| **自然正交** | 接口之间彼此独立、不重叠、不交叉职责 | 两个方法的职责是否纠缠? 改一个会不会影响另一个? |
| **易用** | 调用方好用, 符合直觉, 难以误用 | 调用方要不要看一堆文档才会用? 容易传错参数吗? |
| **(实现)隐藏** | 不泄露实现细节; 接口反映业务, 不反映内存布局 | 接口暴露了内部数据结构吗? 换实现会破坏接口吗? |

## 接口反模式(禁)

| 反模式 | 病因 | 纠正 |
| --- | --- | --- |
| 定义一堆 `get/set`, 让外部直接读写成员变量 | 「用实现替代了业务抽象」; 接口反映的是内存布局而非业务 | 接口方法应是**业务动作**(如 `placeOrder`), 不是字段存取 |
| 接口跟着实现走(先写类再抽接口) | 实现细节渗进契约, 换实现就破坏接口 | 接口跟着**业务**走; 先想清业务抽象再定接口 |
| 接口职责重叠 / 一个方法干多件事 | 违背自然正交; 调用方难判断该用哪个 | 拆成职责单一、彼此正交的方法 |
| 接口大而全、预留一堆「以后可能用」 | 违背最小完备 + 过度设计 | 最小完备; 先判「这能力一定会用吗」(见模型6) |
| 用架构图当接口契约去对齐团队 | 架构图有歧义, 共识是假的 | 用代码/接口签名做精确共识; 架构图仅辅助 |

## 接口与正交分解的关系

接口是正交分解的**落地载体**: 模块之间的正交性, 最终体现为接口的正交性(自然正交准则)。设计时:

1. 正交分解出子业务(模型1) → 每个子业务一个模块
2. 模块的**稳定点**藏进实现, **变化点**通过接口/回调暴露(模型5)
3. 模块间只通过接口耦合, 不共享内部状态 → 保证「改 A 不影响 B」
4. 用「修改影响面」自检(模型3): 加新功能时, 理想是新增一个实现了既有接口的周边模块, 核心接口与核心代码不动
