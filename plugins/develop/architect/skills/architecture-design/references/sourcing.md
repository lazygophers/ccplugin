## 课程实际规模

课程正文为 **00-77 讲**(约 77 讲 + 加餐 + 结束语), 六大话题: 基础平台篇 / 桌面开发篇 / 服务端开发篇 / 服务治理篇 / **架构思维篇(57-67, 方法论高潮)** / 软件工程篇。本插件的心智模型主要出自「架构思维篇」。交付形态有「58 讲」「84 讲音频重交付」等不同口径, 属迭代差异。

## 存疑点(不回避)

1. **❌「DICE」术语查无实据**: 若有人提及许式伟用「DICE 复杂度」衡量模块复杂度——经多轮检索**未找到任何证据**, 全部命中无关概念(气候经济模型/IoT 安全/电路仿真)。许式伟衡量复杂度实际用的是**「修改影响面 / 修改处数」**(模型3, lesson 58)。**本插件不采用 DICE 说法。**

2. **需求分析篇细节存疑**: 社区流传的「场景六要素 / 付费意愿判真伪 / 痛点痒点爽点 / 马斯洛」等, 与许式伟原文的精确归属未证实, 可能是笔记作者用通用产品框架补充。可确证的只有「用户—问题—核心系统」三要素与「稳定点/变化点」。本插件模型7 只用可确证部分。

3.「正交」非自创: 概念源自数学; 许的贡献是把它系统化为架构方法论的核心操作标准, 而非发明该词。

4. **七牛自身架构无逐字一手拆解**: 课程实战用虚拟的「画图程序」载体, 未找到许式伟就七牛对象存储架构的逐字技术拆解一手公开材料。涉及七牛的具体例子均为【推断】。

5. **批评证据偏薄**(本方法论最大短板, 见下)。

## 已知批评(不只展示优点)

- **理论像「科学雏形」**: 豆瓣笔记指软件架构「当前还不是科学, 没有一个从头到尾的逻辑链条」, 课中「判断好的架构的公式尚显粗糙, 只能说有一个科学的雏形」。
- **门槛过高**: 「不是科学就无法从最简单的 case 讲起, 更无法验证」; 基础知识要求高; 「需要深入的时候深入」这类说法「有说漂亮话的嫌疑」。→ 本插件刻意把方法落到可操作判据(三要素/四准则/影响面度量)以对冲此批评。
- **务虚观感**: 强调「掌控全局」「在心中重建世界」, 部分读者感到偏宏观而非实操。
- **史观争议**: 批评者认为信息产业当前结构是「很多商业公司一系列阴差阳错的结果」, 而许式伟把历史偶然当成可推导的必然。

> 最大短板: 最尖锐批评集中在一篇有反爬的豆瓣笔记, 仅取得摘要级内容; 知乎讨论(按调研规则排除作引用源)可能有更系统的负面意见, 未纳入。

## 表达风格(供 agent 语气参考, 不强制模仿)

高确定性断言式(爱用「一定会」「将会」「最正确」「非常坚定地相信」, 极少「也许」); 长期主义/面向未来框架(「下一个十年」「跨越历史周期」); 先抽象世界观、后下沉细节; 信仰式表达(「任何功能都是可以正交分解的, 即使我目前还没有找到方法」); 爱用类比与「造物主」隐喻。

> ⚠️ 用于架构设计工具时, **重判断逻辑、轻语气模仿**——这套方法的价值在框架, 不在腔调。高确定性语气是双刃剑(被批「漂亮话」), agent 输出应保留「失效边界」与不确定标注, 不要照搬「信仰式断言」。
