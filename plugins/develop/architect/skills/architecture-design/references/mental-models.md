# 八个心智模型(详解)+ 决策启发式

> 提炼自许式伟《架构课》(主要为「五、架构思维篇」lesson 57-67)。引号内为课程原话, 但多经第三方笔记转载, 逐字精度见 `sourcing.md`。

## 模型 1: 正交分解(Orthogonal Decomposition)

- **定义**: 架构的本质是把业务系统正交分解为相互独立、互不耦合的模块。降复杂度的根本手段是分解, 分解质量标准是「正交」——彼此独立、改 A 不影响 B。
- **原话**: 「架构就是对业务系统的正交分解。」「业务分解就是最小化的核心系统, 加上多个正交分解的周边系统。」
- **应用**: 拿到任何系统, 第一步不是设计模块, 而是问「哪些子业务彼此正交」。三步曲(需求分析→概要设计→详细设计)都指向把正交分解从模糊推进到精确无歧义。
- **失效边界**: 子业务间存在本质不可消除的耦合(强一致事务、共享状态)时, 强行正交化会制造虚假边界。横切关注点(日志/鉴权)用「全局性功能的架构设计」专门处理, 不塞核心。
- **注**: 「正交」概念借自数学, 非许自创; 他的贡献是把它系统化为架构方法论的核心操作标准。

## 模型 2: 最小核心 + 正交周边(Minimal Core + Orthogonal Periphery)

- **定义**: 任何复杂系统都应组织为「一个尽可能小且稳定的核心 + N 个彼此正交、可插拔的周边」; 复杂度通过「往周边加功能」而非「改核心」增长。
- **原话**: 「核心系统一定要最小化, 要稳定……核心系统构成了业务的最小功能集, 而后通过不断增加新的周边功能, 演变成功能强大的复杂系统。」
- **应用**: 识别「业务最小功能集」作核心, 其余下沉为周边/插件。回调与接口是「插件机制的特例」。
- **失效边界**: 核心边界划错(把易变的放进核心)会令核心频繁改动, 放大模型 3 的伤害。核心「最小化」不能小到不完备(配「最小完备」接口准则)。

## 模型 3: 修改影响面量化(Modification Blast Radius)

- **定义**: 架构好坏可量化——衡量标准是「新增/变更一个功能时需要修改的处数与代码量」。改动越分散、改的行数越多, 架构越差。
- **原话**: 「修改处数越多, 伤害越大。对于每一处修改, 鼓励尽可能减少到只修改一行, 更多代码放到周边模块自己那里去……核心系统越干净, 增加新功能越容易。」(lesson 58)
- **应用**: 评审时做思想实验——「加一个典型新功能, 要碰几个文件、改几行核心代码?」理想 = 核心 0 改动(或 ≤1 行)+ 新增一个独立周边文件。功能相关代码要尽可能内聚, 至少写到独立文件。这是开闭原则的可操作度量。
- **失效边界**: 纯追求「核心零改动」可能催生过度抽象的插件框架(YAGNI 风险)。该度量衡量「应对预期变化」的成本, 对未预见的根本性需求变更无能为力(那属重构, lesson 66)。

## 模型 4: 接口 > 数据结构 > 架构图

- **定义**: 架构表达优先级上, 最重要的不是漂亮的架构图, 而是数据结构; 比数据结构更重要的是接口——接口是「业务的抽象」与「团队共识」的精确载体。
- **原话**: 「比框架(架构图)更重要的是数据结构, 比数据结构更重要的是接口。」「接口是需求的流程分解, 即可以体现用户地图。接口是业务的抽象, 同时也是它与使用方的耦合方式。」
- **反模式原话**: 「一些架构师能够想清楚实现, 但是想不清楚业务。他们用实现替代对业务系统的抽象」——定义数据结构却不抽象业务逻辑, 直接让使用方操作成员变量或定义一堆 get/set。
- **应用**: 评审设计完成度时, 不看 PPT 架构图, 看是否产出精确的接口定义与核心数据结构(他主张系统设计阶段也要产出代码, 因架构图不精确)。
- **失效边界**: 探索期/需求极不确定时, 过早冻结接口会锁死错误抽象。接口稳定性以「业务理解已收敛」为前提。

## 模型 5: 开闭原则应对变化——「与其改模块, 不如加新业务」

- **定义**: 软件工程是「关于变化的学问」; 应对变化不是修改既有模块, 而是通过开放扩展点新增业务模块(OCP)。
- **原话**: 「与其修改模块的业务, 不如实现一个新业务。只要业务的分解一直被正确执行, 实现一个新的业务模块……是一件极其轻松的事情。」「模块的业务变化点, 简单一点的, 通过回调函数或者接口开放出去。」
- **应用**: 设计每个模块时显式区分稳定点与变化点, 把变化点用回调/接口/插件暴露, 核心只承载稳定点。新需求来时优先「新增周边」而非「改核心」。
- **失效边界**: OCP 依赖「变化点预判准确」。预判错误(在不会变的地方留扩展点、真会变的地方写死)会让扩展点变成无人使用的复杂度负债。故需配模型 6。

## 模型 6: 预判「什么一定不会发生」(反过度设计的缰绳)

- **定义**: 架构师必须预判需求的未来变化, 但最关键的不是预测会发生什么, 而是判定**什么一定不会发生**——这是防止过度设计的唯一刹车。
- **原话**: 「预测什么不会发生最为重要, 只有做到这一点, 才能真正防止架构的过度设计, 把简单的事情复杂化。」(他亦援引《程序员修炼之道》:「任何大到需要占卜的任务, 都会很快超出理性猜测的范畴。」)
- **应用**: 留扩展点之前, 先逐条写下「哪些维度未来一定不变」, 钉死进核心简化设计; 只为「理性可判断会变」的维度留接口。
- **失效边界**: 判断「什么不会变」是经验性、概率性的, 在新领域/快速演化领域(早期 AI 产品)误判率高。模型在成熟、约束清晰的领域最可靠。

## 模型 7: 需求三要素——什么人 / 什么场景 / 什么问题

- **定义**: 架构始于需求, 真需求 = 「什么样的人, 在什么样的场景下, 遇到了什么样的问题」; 剥离表层诉求, 识别人性本质层的真需求与伪需求。
- **原话(可确证部分)**: 需求分析三要素「用户(面向什么人群)/ 问题(要解决什么)/ 核心系统(如何解决)」, 并区分「稳定点(核心需求, 底层)与变化点(扩展需求, 上层)」。
- **应用**: 把需求结构化为「用户—场景—问题」三元组, 再把问题映射到「核心系统(最小功能集)+ 扩展接口」。稳定点放核心, 变化点放周边——衔接模型 2、5。
- **失效边界**: ⚠️ lesson 17/18 正文受付费墙限制, 社区流传的「场景六要素/付费意愿/痛点痒点/马斯洛」等细节**与原文精确对应存疑**(可能是笔记作者用通用产品框架补充, 非许原创)。可确证的只有「用户—问题—核心系统」三要素与「稳定点/变化点」。

## 模型 8: 信息世界全栈构建

- **定义**: 架构师的终极能力是「窥见整个信息世界的来龙去脉」——从冯·诺依曼体系经操作系统、编程语言到顶层业务架构, 每层都是决策; 架构的本质是「构建和创造」。
- **原话**: 「架构的本质, 不正是构建和创造么? 这种透过层层代码的业务层级, 窥见整个信息世界的来龙去脉的能力, 才是架构师的终极能力。」成为优秀架构师的关键是「掌控全局——对系统的全貌了然于胸, 在自己心中重新构建出整个世界」。
- **应用**: 这是课程的元结构(自底向上从汇编/CPU 讲起); 做业务架构时也要能向下追问「这层抽象在底层如何落地」。
- **失效边界**: 全栈纵深成本极高, 对多数工程师是「理想态」而非准入门槛。是世界观/修炼方向, 非每个具体设计决策的必经步骤。

## 决策启发式(如果 X → 则 Y)

1. **要降复杂度** → 先正交分解、再设计模块; 目标是「改 A 不影响 B」, 不按技术分层切。(模型1)
2. **新功能要改核心多处** → 坏设计信号; 把功能代码内聚到独立文件/周边, 核心改动压到 ≤1 行。(模型3)
3. **面对变化点** → 优先「新增业务模块」而非「改现有模块」; 简单变化点用回调/接口, 复杂的上插件。(模型5)
4. **留扩展点前犹豫** → 先答「这维度未来一定不变吗」; 能判定不变就钉死简化, 不能判定才留口。(模型6)
5. **评估是否想清楚** → 别看架构图, 看有没有精确的接口定义和核心数据结构(必要时直接写代码)。(模型4)
6. **看到一堆 get/set 或外部直接操作成员变量** → 警惕「用实现替代业务抽象」; 接口必须反映业务逻辑。(模型4反模式)
7. **接口拿不准** → 过四准则: 最小完备 / 自然正交 / 易用 / 隐藏实现。(lesson 63)
8. **判断需求该不该做** → 还原成「什么人/什么场景/什么问题」, 剥到人性本质层; 「更快的马」不是真需求(「更快到达」才是)。(模型7)
9. **一次性任务 vs 长期系统** → 切到「工程师思维」: 用最小化编码解决更大范围问题, 系统化考虑有效性、异常、防御, 而非只让 demo 跑通。(lesson 48)
10. **架构已老化(改动处处掣肘)** → 进入重构而非继续打补丁; 「不断重新审视边界」, 边界不是一次划定永久不变。(lesson 60/66)
