# Design 编排

`design.md` = 技术架构。覆盖边界、契约、数据流、取舍、兼容性、回滚形状。**必须标注每块工作的执行层 + 资源边界**, 让后续 dispatch 无需重新决策。

## 执行层标注

每模块在表中标 `main / sub-agent / agent-team`, 标注准则见 `layer-selection.md`。同一 deliverable 内多模块若执行层不同, design 必须说明协调点 (谁等谁, 在什么交付物上交接)。

## 资源边界

每模块独占资源必须列, 资源类型见 `shared-resources.md`。**冲突即串行** — design 阶段就写明顺序, 不要丢给 dispatch 时临时判定。

## 跨层契约

设计涉及 ≥ 2 层 (e.g. controller + service) 必须:
- 列每层契约边界 (输入 schema / 输出 schema / 副作用)
- 跨层调用走显式接口, 禁直接读对方实现字段
- design 给出 grep 验证命令防止层间漏写

## 写 design 的硬规

1. **每个模块必须可独立测试**, 设计阶段就给出 mock / stub 的方法; 否则模块切分有问题。

2. **数据流必须有方向**, 禁循环依赖; 若必要给出打破循环的方案 (event / DI / interface)。

3. **接口契约必须可机器校验**, 给出 schema / 类型 / OpenAPI; 禁纯文字描述。

4. **回滚形状必须存在**, 不可回滚的设计 = 高风险设计, 必须标 P0 风险 + 缓解方案。

5. **取舍必须留痕**, 选 A 不选 B 的理由写下来, 防止未来重审时无据可依。
