# 结构化根因复盘协议 (root-cause protocol)

check 反复不过时的跨维度定位。逐维度问「是不是这里」, 从**外层 (需求) 到内层 (测试)** 扫一遍, 收敛到 root cause。**不许只查一个维度就下结论** —— 反复不过往往是维度错配 (在实现层狂改, 根因其实在需求层)。

## 5 维度 checklist (逐项 ✅/❌ 输出 trace)

```
break-loop 根因 trace
═══════════════════
① 需求理解偏差: ✅/❌ (证据: ..)
   探针: checker 验的验收标准, 是不是 planning 时就理解错了? 契约本身写错了?
② 方案设计缺陷: ✅/❌ (证据: ..)
   探针: 实现按 design 走了, 但 design 的路子本身走不通 / 有结构漏洞?
③ 实现 bug: ✅/❌ (证据: ..)
   探针: 需求方案都对, 就是代码写错 (逻辑/边界/拼写/漏改调用点)?
④ 环境或依赖: ✅/❌ (证据: ..)
   探针: 代码没错, 是 node/python 版本、缺依赖、脏缓存、worktree 状态、平台差异 (darwin)?
⑤ 测试本身错误: ✅/❌ (证据: ..)
   探针: 被测代码是对的, 是断言写错 / 用例过时 / mock 失真 / 环境未初始化?
─────────────────
root cause: <哪个维度 + 一句话根因>
可修层: <exec 可修 (③④⑤) | 超 exec 范围 (①②)>
```

- **证据必须具体**: file:line + 报错原文 + 为什么归到这个维度。凭空断言不算定位。
- **收敛判据**: 找到能解释「为什么连修 2 轮都不过」的那个维度才算收敛。若表层维度 (③) 已改多轮仍不过, 强制上溯 ①②。
- 多维度同时 ✅ 时, 取**最外层**为 root cause (需求错则实现再对也没用)。

## 根因 → 出口映射

| root cause 维度 | 可修层 | 出口 |
|---|---|---|
| ③ 实现 bug | exec | 带根因回 exec 定点重修 |
| ⑤ 测试本身错 | exec | 带根因回 exec 改测试 (先确认不是在掩盖真 bug) |
| ④ 环境/依赖 | exec | 带根因回 exec 修环境 (装依赖/清缓存/对齐版本) |
| ② 方案设计缺陷 | 超 exec | STOP 转人工, 建议退回 planning 改 design |
| ① 需求理解偏差 | 超 exec | STOP 转人工, 建议退回 planning 重对齐验收标准/契约 |

## 预防措施 (定位后必给)

每个 root cause 配两句:

- **本次修**: 具体怎么改 (给 implementer 的定向指令, 或给用户的方案建议)。
- **同类防**: 下次同类任务怎么避免踩同一坑 → 若可写成可验证契约, 即 sediment 判定门正向项②的输入。
