# 结构化根因复盘协议 (root-cause protocol)

skein-check 定点修复 **≥2 轮仍 FAIL**、第 3 轮跳出调试循环时执行本协议。**症状级修补已失败**, 此刻禁再盲改 —— 先做跨维度根因定位, 再决定回 exec 重修还是转人工。

> STOP 再改一轮的冲动: 连修 2 轮不过说明问题**不在你改的那层**。第 3 轮继续同层定点改 = 无限循环。必须先复盘 root cause。

**触发**: 定点修复 1-2 轮仍 FAIL → 仍走 check 正常循环, 不进本协议; 第 3 轮仍 FAIL → STOP 定点循环, 走本协议 (main 亲做, 复盘需跨维度语义判断)。

**流程**: ① 跨维度根因定位 (下节 5 维 checklist) → ② 预防措施 (本次怎么修 + 同类怎么防) → ③ 可复用教训判定 (走 skein-memory sediment 判定门, 见「沉淀」节) → ④ 出口二选一 (见「出口」节)。

## 跨维度定位

check 反复不过时的跨维度定位。逐维度问「是不是这里」, 从**外层 (需求) 到内层 (测试)** 扫一遍, 收敛到 root cause。**不许只查一个维度就下结论** —— 反复不过往往是维度错配 (在实现层狂改, 根因其实在需求层)。

## 5 维度探针

逐维度问「是不是这里」, 每维度定位必附证据 (file:line + 报错原文 + 归因理由):

- ① 需求理解偏差 — checker 验的验收标准, planning 时就理解错了? 契约本身写错了?
- ② 方案设计缺陷 — 实现按 design 走了, 但 design 路子本身走不通 / 有结构漏洞?
- ③ 实现 bug — 需求方案都对, 就是代码写错 (逻辑/边界/拼写/漏改调用点)?
- ④ 环境或依赖 — 代码没错, 是版本/缺依赖/脏缓存/worktree 状态/平台差异 (darwin)?
- ⑤ 测试本身错误 — 被测代码对, 是断言写错 / 用例过时 / mock 失真 / 环境未初始化?

收敛后给: root cause (哪个维度 + 一句话根因) + 可修层 (exec 可修 ③④⑤ / 超 exec 范围 ①②)。

- **收敛判据**: 找到能解释「为什么连修 2 轮都不过」的那个维度才算收敛。表层维度 (③) 已改多轮仍不过, 强制上溯 ①②。
- 多维度同时命中, 取**最外层**为 root cause (需求错则实现再对也没用)。

## 根因 → 出口映射

| root cause 维度 | 可修层 | 出口 |
|---|---|---|
| ③ 实现 bug | exec | 带根因回 exec 定点重修 |
| ⑤ 测试本身错 | exec | 带根因回 exec 改测试 (先确认不是在掩盖真 bug) |
| ④ 环境/依赖 | exec | 带根因回 exec 修环境 (装依赖/清缓存/对齐版本) |
| ② 方案设计缺陷 | 超 exec | STOP 转人工, 建议退回 planning 改 design |
| ① 需求理解偏差 | 超 exec | STOP 转人工, 建议退回 planning 重对齐验收标准/契约 |

## 预防措施 (定位后必给)

每个 root cause 配两句:

- **本次修**: 具体怎么改 (给 implementer 的定向指令, 或给用户的方案建议)。
- **同类防**: 下次同类任务怎么避免踩同一坑 → 若可写成可验证契约, 即 sediment 判定门正向项②的输入。

## 沉淀 (复用 skein-memory, 禁新造)

复盘完的根因**不自动沉淀**, 走既有判定门:

- 判定 → 分层 → 审批 → 写盘四步, 完全复用 [skein-memory](../../skein-memory/SKILL.md) 的 [sediment-workflow](../../skein-memory/references/sediment-workflow.md), 命令是同一个 `memory.py sediment`。
- **不新造沉淀机制**。命中判定门正向项②「踩坑 ≥2 轮 (根因可写可验证契约)」正是本场景 —— 反复不过 ≥2 轮本身就是踩坑证据, 但仍须过「排除」项 (一次性 bug / 本 task 私有细节 / 已有规则覆盖 → 跳过)。
- 沉淀正文写**根因契约** (可验证的 MUST/禁), 非流水账。

## 出口 (二选一)

| 出口 | 条件 | 动作 |
|---|---|---|
| **回 exec 重修** | 根因定位明确 (实现 bug / 测试本身错 / 环境可修) | 带**根因 + 预防措施**派 `skein-implementer` 定点重修, 回 skein-check 重验。**根因驱动的修复不计入原 3 轮上限** (是新一轮定向修, 非盲改) |
| **STOP 转人工** | 根因是需求理解偏差 / 方案设计缺陷 (超出 exec 层可修范围) | STOP 回传用户: 附**根因报告** (哪个维度 + 证据 + 建议退回 planning / 改方案), 由用户裁定 |

## 反例

| 禁 | 改为 |
|---|---|
| 第 3 轮继续同层盲改 | STOP 定点循环, 先跨维度定位 root cause |
| 只查实现层 (③) 就下结论 | 5 维度全扫, 表层多轮不过强制上溯 ①② |
| 根因无证据凭空断言 | 每维度给 file:line + 报错原文 + 归因理由 |
| 需求/设计错却硬留 exec 重修 | ①② 超 exec 范围 → STOP 转人工退回 planning |
| 复盘完自动写规则 | 走 skein-memory sediment 判定门 (判定→审批→写盘) |
| 一次性 bug 硬凑成沉淀契约 | 过「排除」项, 一次性 → drop 跳过 |
| 新造一套沉淀机制 | 复用 memory.py sediment, 禁另起炉灶 |
| STOP 只丢一句"过不了" | 附根因报告 (维度 + 证据 + 退回建议) 交用户裁 |
