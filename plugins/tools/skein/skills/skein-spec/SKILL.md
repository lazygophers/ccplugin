# skein-spec — 破坏式重构执行

**破坏式** = 不保后向兼容, 重写而非贴补丁: 直接改签名/契约、删旧路径、统一所有调用站点。适用于「旧设计已烂、修补比重写更贵」的场景。

> ⚠️ 破坏式 ≠ 乱删。前提是**任务显式授权**破坏 (用户要 / PRD 定了「重写」)。默认仍走加固式修补, 不主动破坏。

## 何时破坏式 (vs 修补)

| 特征 | 走破坏式 |
|---|---|
| 旧契约被多处将就, 加参数越加越脏 | ✅ 改契约 + 全站点改齐 |
| 要删的旧路径无外部消费者 (可 grep 验) | ✅ 直接删, 不留 deprecated 垫片 |
| 用户/PRD 明确「重写, 不用管旧的」 | ✅ |
| 有外部/线上消费者依赖旧契约 | ❌ 退回加固式 / 走版本迁移 |

## 铁律

- 🔴 **全站点一次性改齐** — 改契约必同轮改**所有**调用站点 (先 grep 穷举, 再逐个改)。留一个没改 = 破坏式失败 (半破坏最糟)。
- 🔴 **禁留兼容垫片** — 破坏式的意义就是删掉旧路径; 留 `@deprecated` / shim / 双写 = 没破坏, 退回修补式即可。
- 🔴 **改前查爆炸半径** — 有 GitNexus 则 `gitnexus_impact({target, direction:"upstream"})` 查全部上游; 否则 grep 调用站点。HIGH/CRITICAL 风险先报用户。
- 载体照旧: 派 subagent 在 task worktree 内改 (main 默认禁写源码), check 必覆盖被改契约的全部站点。

## 流程

1. **定破坏面**: 要改的契约 / 要删的路径 + grep 全部调用站点清单。
2. **查半径 + 报风险**: impact/grep 查全部上游。🔴 CHECKPOINT: HIGH/CRITICAL 风险 MUST 停下报用户; 用户否决 → 退回加固式修补, 禁擅自续改。
3. **一次性改齐**: 契约 + 全站点 + 删旧路径, 同一 task worktree 内。
4. **check**: 编译/类型/测试全过 (破坏式最易漏站点, check 是安全网)。
5. sediment 归 **skein-memory** (若沉淀「XX 契约已重写」这类跨任务契约)。
