# order-create-api — 详细设计

> planning 工件 (skein-plan 写)。**AI 可读写**, 不受 guard 拦截。架构/数据流/取舍/选型在此 (不含调度图, 调度归 task.json)。

## 架构 / 数据流

`POST /orders` 单事务闭环:

```
请求 → [参数校验] → [库存扣减 (原子)] → [订单落库 (幂等)] → [发 MQ 创建事件]
```

- 校验层拦非法请求 (缺字段 / 数量 ≤ 0), 早失败返回 400。
- 库存扣减走原子 `UPDATE ... WHERE stock >= qty`, 影响行 0 即库存不足返回 409。
- 订单落库用幂等键唯一约束, 冲突时返回既有订单号 (不重复落单)。
- 落单成功后异步发一次 MQ 创建事件, 供下游发货 / 通知消费; 发送失败可重试。

## 关键取舍

- **库存扣减先于落库**: 库存是稀缺资源, 先占再落单, 避免超卖; 落库失败则补偿回滚库存。
- **幂等靠唯一约束而非应用层查重**: 并发下应用层「先查后插」有竞态, 交给 DB 唯一约束兜底。
- **MQ 事件异步**: 落单主链路不阻塞在下游消费, 事件失败独立重试。

## 技术选型

- 事务边界: 库存 + 订单同库同事务; MQ 发送在事务提交后 (避免发了事件但订单回滚)。
- 幂等键来源: 客户端传入 `idempotency-key` header。
