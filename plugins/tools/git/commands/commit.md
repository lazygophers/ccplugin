# 提交变更

按照规范提交代码变更到 Git 仓库。

## 执行流程

### 1. 检查工作区状态

```bash
git status
```

查看未提交的文件和变更类型。

### 2. 添加文件到暂存区

```bash
# 如果暂存区为空，添加所有文件
git add .

# 或选择性添加
git add <file1> <file2>
```

### 3. 生成提交信息

根据变更内容生成符合规范的提交信息：
- 必须严格遵守 `Skills(commit)` 规范
- 使用规范的类型前缀（feat/fix/docs/refactor/test 等）
- 提交信息简洁明确，说明变更内容和原因

### 4. 执行提交

```bash
git commit -m "<描述信息>"
```

## 执行检查清单

### 提交前检查

- [ ] 运行 `git status` 确认工作区状态
- [ ] 确认变更文件列表符合预期
- [ ] 确认没有意外的文件变更（如敏感信息、临时文件）
- [ ] 确认代码已通过本地测试
- [ ] 确认代码已通过 lint 检查

### 暂存区检查

- [ ] 使用 `git add` 添加需要提交的文件
- [ ] 确认暂存区包含所有相关变更
- [ ] 确认没有遗漏关键文件
- [ ] 使用 `git diff --cached` 检查暂存的变更内容

### 提交信息检查

- [ ] 提交信息符合 `Skills(commit)` 规范
- [ ] 使用正确的类型前缀（feat/fix/docs/style/refactor/perf/test/chore）
- [ ] 提交信息清晰描述变更内容
- [ ] 提交信息长度适中（标题 ≤ 50 字符）
- [ ] 必要时包含详细说明（body）和关联的 issue/PR 编号

### 提交后检查

- [ ] 使用 `git log -1` 确认提交成功
- [ ] 确认提交信息正确
- [ ] 确认提交包含了预期的文件变更
- [ ] 确认没有敏感信息泄露

## 工具调用说明

### Bash 命令

| 命令 | 用途 | 调用时机 |
|------|------|---------|
| `git status` | 查看工作区状态 | 提交前必须执行 |
| `git add` | 添加文件到暂存区 | 暂存区为空时 |
| `git diff --cached` | 查看暂存的变更 | 确认变更内容时 |
| `git commit` | 提交变更 | 暂存区准备完毕后 |
| `git log -1` | 查看最新提交 | 提交后验证 |

### Skills 引用

| Skill | 用途 | 必需性 |
|------|------|--------|
| `Skills(commit)` | 提交信息规范 | **必需**，必须严格遵守 |
| `Skills(git:core)` | Git 操作规范 | 推荐 |

## 提交信息规范

遵循 `Skills(commit)` 规范：

```
<type>(<scope>): <subject>

<body>

<footer>
```

### 类型（type）

- `feat`: 新功能
- `fix`: 修复 bug
- `docs`: 文档变更
- `style`: 代码格式（不影响功能）
- `refactor`: 重构（不改变功能）
- `perf`: 性能优化
- `test`: 测试相关
- `chore`: 构建、工具、依赖等

### 示例

```bash
# 新功能
git commit -m "feat(auth): 添加用户登录功能"

# 修复 bug
git commit -m "fix(api): 修复用户信息接口返回错误"

# 文档更新
git commit -m "docs(readme): 更新安装说明"

# 代码格式
git commit -m "style(lint): 修复 eslint 格式问题"
```

## 注意事项

### 必须遵守

1. **严格遵守 Skills(commit) 规范** - 提交信息格式必须符合规范
2. **提交前必须测试** - 确保代码可以正常运行
3. **原子提交** - 每次提交只做一件事
4. **避免大批量提交** - 大量变更应拆分为多个提交

### 避免的错误

- ❌ 提交信息模糊不清（如 "update", "fix bug"）
- ❌ 提交包含无关变更
- ❌ 提交包含临时文件或敏感信息
- ❌ 提交未经测试的代码
- ❌ 提交破坏构建的代码

### 最佳实践

- ✅ 提交前运行完整测试
- ✅ 使用有意义的提交信息
- ✅ 每个提交保持独立可回退
- ✅ 提交信息使用祈使语气（"添加"而非"添加了"）
- ✅ 关联相关 issue 编号
