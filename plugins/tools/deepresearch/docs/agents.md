# Agent 系统（2026架构）

Deep Research 插件提供4个专业Agent，统一使用opus模型，职责明确无重叠。

## Agent 概览

| Agent | 统一职责 | 适用场景 | 模型 |
|-------|---------|---------|------|
| **code-analyst** | 代码质量+技术债+性能 | 代码审查、重构规划、性能优化 | opus |
| **research-strategist** | 概念研究+技术选型 | 技术学习、框架选型、方案对比 | opus |
| **project-assessor** | 项目评估+依赖安全 | GitHub研究、安全审计 | opus |
| **architecture-advisor** | 架构评审+设计建议 | 架构评估、系统设计 | opus |

---

## code-analyst（代码分析师）

### 统一职责

代码质量、技术债务和性能的一站式分析专家。

**合并来源**（旧架构）：
- code-review（代码质量分析）
- technical-debt（技术债识别）
- performance-analysis（性能分析）

### 核心能力

- **代码质量分析**：静态分析、代码规范、最佳实践验证
- **技术债识别**：债务分类（设计债/测试债/文档债/架构债）、优先级排序
- **性能分析**：瓶颈定位、优化建议、基准测试

### 触发关键词

`代码质量` `代码审查` `技术债` `重构` `性能瓶颈` `性能优化`

### 使用示例

```bash
# 代码质量分析
分析 ./src 目录的代码质量和技术债

# 性能优化
定位 API 服务的性能瓶颈并提供优化建议

# 重构规划
评估项目技术债并给出重构优先级
```

### 分析维度

**质量维度**：
- 可读性（命名、注释、代码组织）
- 可维护性（模块化、耦合度、复杂度）
- 可测试性（依赖注入、接口抽象）
- 安全性（注入漏洞、敏感信息泄露）

**技术债维度**：
- 设计债（违反设计原则）
- 测试债（测试覆盖率低）
- 文档债（缺少文档）
- 架构债（技术栈过时）

**性能维度**：
- 响应时间（API延迟）
- 吞吐量（QPS、并发）
- 资源使用（CPU、内存、I/O）
- 瓶颈定位（慢查询、低效算法）

### 优先级评分

```
优先级 = 严重度(0-10) × 影响范围(0-10) × 紧急度(0-10)
```

## research-strategist（研究策略师）

### 统一职责

技术概念研究和方案对比选型专家。

**合并来源**（旧架构）：
- concept-explanation（概念解释）
- tech-selection（技术选型）

### 核心能力

- **技术概念研究**：分层讲解（初学者→专家）、深度理解、最佳实践
- **方案对比选型**：多维度对比、决策支持、风险评估、趋势预测

### 触发关键词

`什么是` `如何` `概念解释` `技术选型` `方案对比` `vs` `选择框架`

### 使用示例

```bash
# 概念学习
解释微服务架构的核心概念和最佳实践

# 技术选型
对比 React vs Vue vs Svelte 在企业项目中的优劣

# 方案对比
GraphQL 和 REST 分别适合什么场景
```

### 对比维度

| 维度 | 权重 | 说明 |
|------|------|------|
| 性能 | 25% | 响应时间、吞吐量、资源占用 |
| 生态 | 20% | 社区活跃度、插件数量、文档质量 |
| 学习曲线 | 15% | 上手难度、概念复杂度 |
| 成熟度 | 15% | 版本稳定性、生产案例 |
| 可维护性 | 15% | 代码质量、架构设计 |
| 扩展性 | 10% | 插件系统、定制能力 |

### 输出格式

- **概念学习**：博客文章（分层讲解+代码示例）
- **快速决策**：对比表格（多维度评分）
- **深度分析**：技术报告（详细对比+决策建议）

## project-assessor（项目评估师）

### 统一职责

GitHub/GitLab项目评估和依赖安全审计专家。

**合并来源**（旧架构）：
- project-evaluation（GitHub项目评估）
- dependency-audit（依赖安全审计）

### 核心能力

- **项目综合评估**：健康度评分、质量分析、社区评估
- **依赖安全审计**：CVE漏洞检测、许可证合规、供应链风险评估

### 触发关键词

`评估项目` `GitHub项目` `开源库` `依赖安全` `安全审计` `CVE漏洞` `github.com`

### 使用示例

```bash
# 项目评估
评估 facebook/react 的项目质量和安全风险

# 安全审计
审计项目的依赖包安全性和许可证合规

# 对比分析
对比 React 和 Vue 两个项目的健康度
```

### 评估指标

**项目健康度**（0-10分）：
```
健康分 = 活跃度(40%) + 质量(30%) + 社区(20%) + 维护(10%)
```

**安全评分**（0-10分）：
```
安全分 = 无高危漏洞(40%) + 依赖安全(30%) + 更新及时(20%) + 许可合规(10%)
```

### GitHub MCP工具

使用9个GitHub MCP工具进行深度分析：
- get_repository - 仓库基本信息
- list_issues - Issue列表和趋势
- list_pull_requests - PR统计
- get_file_contents - 代码分析
- search_repositories - 仓库搜索

## architecture-advisor（架构顾问）

### 统一职责

系统架构评审和设计建议专家。

**来源**（旧架构）：
- architecture-review（保留核心）

### 核心能力

- **架构评审**：设计文档审查、架构模式识别、原则验证
- **设计建议**：架构改进、重构方案、演进路径规划
- **最佳实践**：SOLID原则、DDD、Clean Architecture

### 触发关键词

`架构评审` `架构评估` `设计评审` `架构建议` `重构方案` `架构演进`

### 使用示例

```bash
# 架构评审
评审 ./docs/architecture.md 的架构设计

# 重构建议
设计从单体到微服务的演进路径

# 模式识别
识别当前系统使用的架构模式并给出改进建议
```

### 评审维度

**可扩展性评分**（0-10）：
```
扩展性 = 水平扩展(40%) + 功能扩展(30%) + 技术扩展(20%) + 性能扩展(10%)
```

**可维护性评分**（0-10）：
```
维护性 = 模块化(35%) + 低耦合(30%) + 文档(20%) + 测试(15%)
```

**可靠性评分**（0-10）：
```
可靠性 = 容错(40%) + 监控(25%) + 降级(20%) + 恢复(15%)
```

### 架构原则验证

**SOLID原则**：
- Single Responsibility（单一职责）
- Open/Closed（开闭原则）
- Liskov Substitution（里氏替换）
- Interface Segregation（接口隔离）
- Dependency Inversion（依赖倒置）

**其他原则**：
- DRY（Don't Repeat Yourself）
- KISS（Keep It Simple, Stupid）
- YAGNI（You Aren't Gonna Need It）

### 架构决策记录（ADR）

推荐使用ADR记录关键决策：

```markdown
# ADR-001: 采用微服务架构

## 状态
已接受

## 上下文
单体架构无法支撑业务快速增长...

## 决策
采用微服务架构...

## 后果
正面：独立部署、技术栈灵活...
负面：复杂度增加、运维成本...
```

## 自动Agent匹配

deep-research Skill 会根据任务特征自动选择Agent：

```python
if "代码" in task or "质量" in task or "性能" in task:
    agent = "code-analyst"
elif "选型" in task or "对比" in task or "vs" in task:
    agent = "research-strategist"
elif "github.com" in task or "依赖" in task or "安全" in task:
    agent = "project-assessor"
elif "架构" in task or "设计" in task or "重构" in task:
    agent = "architecture-advisor"
```
