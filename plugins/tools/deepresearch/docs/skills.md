# Skill 系统（2026架构）

Deep Research 插件提供6个Skill：1个用户入口 + 5个核心能力。

## Skill 概览

| Skill | 核心职责 | 用户可调用 | 关键特性 |
|-------|---------|-----------|---------|
| **deep-research** | 研究工作流入口 | ✅ 是 | Task-Centric，自动编排 |
| **dgot-engine** | DGoT动态图思维引擎 | ❌ 否 | 早停+裁剪，-50%成本 |
| **agentic-retriever** | 智能检索器 | ❌ 否 | Agentic RAG，99%精度 |
| **source-validator** | 来源验证器 | ❌ 否 | A-E评级，链式验证 |
| **knowledge-synthesizer** | 知识合成器 | ❌ 否 | 8种输出格式 |
| **code-inspector** | 代码检查器 | ❌ 否 | 本地+GitHub统一接口 |

---

## deep-research（研究工作流）

### 用户入口

唯一的用户可调用Skill（`user-invocable: true`），替代旧的command系统。

### Task-Centric 架构

基于任务驱动而非Agent驱动：
1. **单次需求收集**：智能结构化提问（5-8个维度）
2. **自动Agent匹配**：基于任务特征自动选择
3. **DGoT核心驱动**：动态生成研究路径并优化
4. **智能检索**：Agentic RAG动态策略
5. **质量验证**：A-E评级过滤
6. **知识合成**：8种格式输出

### 使用示例

```bash
# 代码质量分析
分析 ./src 的代码质量和技术债

# 技术选型
对比 React vs Vue vs Svelte

# 项目评估
评估 facebook/react 的质量和安全

# 架构评审
评审 ./docs/architecture.md 设计
```

## dgot-engine（DGoT动态图思维引擎）

### 核心职责

研究路径优化和动态资源分配，降低43-56%研究成本。

**合并来源**（旧架构）：got-controller + research-executor

### 核心操作

1. **Generate(k)**：生成k个并行路径（k=3~8）
2. **Score**：质量评分（权威性40% + 时效性25% + 相关性20% + 准确性15%）
3. **KeepBestN(n)**：动态阈值裁剪（≥8.5分优先保留，<7.0分丢弃）
4. **EarlyStop**：早停机制（3个≥8.5分路径或5个≥7.5分路径）
5. **Refine(n)**：路径深化
6. **Aggregate(n)**：结果聚合

### DGoT vs 传统GoT

**传统GoT问题**：固定k值、无早停、资源分配不均

**DGoT解决方案**：
- 动态阈值：质量达标提前终止
- 早停机制：平均减少30%迭代
- 智能裁剪：平均减少20%无效路径
- **总体节省**：43-56%成本

### FoT框架集成

- 超参数调优：自动调整k值和阈值
- Prompt优化：根据任务类型优化
- 智能缓存：缓存高质量路径结果

## agentic-retriever（智能检索器）

### 核心职责

基于Agentic RAG和GraphRAG技术，多源并行检索高质量信息。

**合并来源**（旧架构）：content-retriever + explorer

### Agentic RAG

嵌入自主AI代理，动态调整检索策略：
- **深度优先**：单一主题深入挖掘
- **广度优先**：多角度快速覆盖
- **平衡搜索**：深度+广度结合（推荐）

### GraphRAG接口

知识图谱+向量搜索，精度高达99%：
- 向量搜索：基于embedding的相似度匹配
- 知识图谱：结构化关系推理
- 混合检索：向量+图谱结合

### 多源并行检索（8+渠道）

- **学术源**：PubMed、IEEE、ACM、arXiv、Google Scholar
- **技术源**：GitHub、Stack Overflow、技术博客
- **商业源**：Gartner、IDC、McKinsey
- **新闻源**：技术媒体、行业报告
- **政府源**：标准文档、法规政策

### 智能过滤与反思

**四维评分**：
```
总分 = 相关性×0.35 + 质量×0.30 + 时效性×0.20 + 重要性×0.15
```

**反思机制**：
- 结果<5条：放宽相关性阈值（0.7→0.6）
- 质量<7.0分：扩展检索源
- 时效性差：优先最近1年内容

## source-validator（来源验证器）

### 核心职责

A-E级质量评分和链式验证，确保信息可信度。

**来源**（旧架构）：citation-validator（精简版）

### A-E评级系统

| 级别 | 分数 | 特征 | 示例 |
|------|------|------|------|
| **A级** | 9.0-10 | 同行评审论文、权威机构报告 | Nature、Science、IEEE、WHO |
| **B级** | 7.0-8.9 | 行业报告、官方文档、专家博客 | Gartner、React Docs、Martin Fowler |
| **C级** | 5.0-6.9 | 专业博客、案例研究、会议记录 | 技术博客、白皮书 |
| **D级** | 3.0-4.9 | 个人博客、营销材料、预印本 | 个人网站、产品宣传 |
| **E级** | 0-2.9 | 传言、纯观点、过时信息 | 自动丢弃 |

### 链式验证

1. **交叉验证**：同一事实需要≥2个独立来源（B级以上）确认
2. **时效性检查**：技术类<2年，学术类<5年
3. **利益冲突检测**：标注赞助商和利益相关方
4. **事实核查**：验证原始数据源

### 质量指标

```
总分 = 权威性(40%) + 时效性(25%) + 准确性(20%) + 独立性(15%)
```

## knowledge-synthesizer（知识合成器）

### 核心职责

整合多源发现为结构化报告，适配不同受众。

**来源**（旧架构）：synthesizer（精简版）

### 8种输出格式

| 格式 | 受众 | 长度 | 用途 |
|------|------|------|------|
| **执行摘要** | 高层决策者 | 1-2页 | 快速汇报、决策支持 |
| **技术报告** | 技术专家 | 5-15页 | 深度分析、技术评审 |
| **学术论文** | 研究人员 | 8-20页 | 学术研究、理论探讨 |
| **演示文稿** | 团队成员 | 10-20页PPT | 会议分享、团队汇报 |
| **博客文章** | 普通读者 | 1500-3000字 | 知识分享、技术传播 |
| **对比表格** | 决策者 | 1-2页 | 快速决策、方案对比 |
| **清单报告** | 审计员 | 2-5页 | 质量检查、合规审计 |
| **llms.txt** | AI系统 | 可变 | API文档、项目概览 |

### 合成流程

1. **信息汇总**：收集所有检索和分析结果
2. **主题识别**：识别核心主题和关键发现
3. **结构组织**：根据输出格式组织内容
4. **引用标注**：标注所有来源（A-E评级）
5. **质量检查**：逻辑连贯性、引用准确性

### 受众适配

- **高层决策者**：简洁明了（≤2页）、避免术语、强调业务价值
- **技术专家**：技术准确（5-15页）、深度分析、实现细节
- **普通读者**：通俗易懂（1500-3000字）、比喻说明、实用性强

## code-inspector（代码检查器）

### 核心职责

提供本地代码库和GitHub远程仓库的统一分析接口。

**合并来源**（旧架构）：local-code-analysis + github-analysis

### 统一接口

**本地分析**：分析路径、深度、关注点（质量/债务/性能）

**GitHub分析**：仓库名、分支、关注点（健康度/安全/社区）

### 本地分析能力

**静态分析**：
- 圈复杂度（McCabe Complexity）
- 代码重复检测
- 认知复杂度

**依赖分析**：
- 依赖关系图
- 循环依赖检测
- CVE漏洞扫描

**测试覆盖率**：
- 行覆盖率（Line Coverage）
- 分支覆盖率（Branch Coverage）
- 目标：≥80%

### GitHub分析能力

**项目健康度**（0-10分）：
```
健康分 = 活跃度(40%) + 质量(30%) + 社区(20%) + 维护(10%)
```

**GitHub MCP工具**（9个）：
- get_repository - 仓库基本信息
- list_issues - Issue列表
- list_pull_requests - PR列表
- get_file_contents - 文件内容
- search_repositories - 仓库搜索

### 质量指标

**代码质量评分**（0-10）：
```
质量分 = 可读性(30%) + 可维护性(30%) + 可测试性(20%) + 安全性(20%)
```
