# 使用指南（2026架构）

Deep Research 插件的使用方法和示例。

## 重要变更

**2026架构移除了Command系统**，改用Skill作为用户入口：

| 项目 | 旧架构 | 新架构（2026） |
|------|--------|---------------|
| 入口类型 | Command | Skill |
| 调用方式 | `/deep-research` | 直接描述任务 |
| Agent选择 | 手动选择 | 自动匹配 |
| 需求收集 | 多次交互 | 单次智能收集 |

---

## 使用方法

### 基本语法

直接描述你的研究需求，系统会自动：
1. 智能需求收集（单次结构化提问）
2. 自动Agent匹配（基于任务特征）
3. DGoT核心驱动（动态路径优化）
4. 生成研究报告（8种格式）

**无需指定命令或Agent**，直接说明需求即可。

---

## 使用场景

### 1. 代码质量分析

**触发Agent**：code-analyst

```bash
# 基本分析
分析 ./src 目录的代码质量和技术债

# 性能分析
定位 API 服务的性能瓶颈并提供优化建议

# 重构规划
评估项目技术债并给出重构优先级排序
```

**自动流程**：
1. 需求收集：分析范围、输出格式
2. Agent匹配：code-analyst（检测到"代码质量"关键词）
3. 代码检查：code-inspector静态分析
4. 报告生成：技术报告（质量+债务+性能）

---

### 2. 技术选型研究

**触发Agent**：research-strategist

```bash
# 框架对比
对比 React vs Vue vs Svelte 在企业项目中的优劣

# 技术决策
GraphQL 和 REST 分别适合什么场景

# 概念学习
解释微服务架构的核心概念和最佳实践
```

**自动流程**：
1. 需求收集：对比对象、使用场景
2. Agent匹配：research-strategist（检测到"对比"/"vs"关键词）
3. 多维对比：性能/生态/学习曲线等6个维度
4. 报告生成：对比表格（多维度评分矩阵）

---

### 3. GitHub项目评估

**触发Agent**：project-assessor

```bash
# 项目评估
评估 facebook/react 的项目质量和安全风险

# 安全审计
审计项目的依赖包安全性和许可证合规

# 对比分析
对比 React 和 Vue 两个开源项目的健康度
```

**自动流程**：
1. 需求收集：项目URL、关注点（质量/安全）
2. Agent匹配：project-assessor（检测到github.com）
3. 项目分析：GitHub MCP工具（健康度+安全分）
4. 报告生成：技术报告（评估+风险+建议）

---

### 4. 架构评审

**触发Agent**：architecture-advisor

```bash
# 架构评审
评审 ./docs/architecture.md 的架构设计

# 重构建议
设计从单体到微服务的演进路径

# 模式识别
识别当前系统使用的架构模式并给出改进建议
```

**自动流程**：
1. 需求收集：设计文档路径、评审重点
2. Agent匹配：architecture-advisor（检测到"架构"关键词）
3. 架构分析：模式识别、原则验证（SOLID/DRY/KISS）
4. 报告生成：技术报告（评审+建议+演进路径）

## 高级功能

### 指定输出格式

```bash
# 执行摘要（高层决策者）
分析项目技术债，生成1页执行摘要用于管理层汇报

# 演示文稿（团队分享）
对比三个框架的优劣，生成PPT用于技术分享会

# 博客文章（技术传播）
解释DDD领域驱动设计，生成3000字博客文章
```

### 多Agent协作

```bash
# 代码质量 + 架构评审
全面评估项目的代码质量和架构设计，提供改进建议

# 技术选型 + 项目评估
对比 Next.js 和 Nuxt.js，并评估两个项目的健康度
```

**系统会自动调用多个Agent协作**。

### 指定质量标准

```bash
# A-B级来源（高质量）
研究微服务架构最佳实践，仅使用A-B级来源（论文+官方文档）

# 时效性要求
调研容器技术发展趋势，优先使用最近1年的资料
```

## 最佳实践

### ✅ 推荐做法

1. **提供具体目标**
   - ❌ "优化代码"
   - ✅ "分析 ./src 的代码质量，重点关注圈复杂度和技术债"

2. **明确输出格式和受众**
   - ❌ "研究微服务"
   - ✅ "研究微服务架构，生成技术报告给工程师团队"

3. **说明约束条件**
   - ❌ "对比框架"
   - ✅ "对比 React vs Vue，场景是企业级项目，团队规模20人"

### ❌ 避免做法

1. **过于模糊**：
   - ❌ "分析项目"（什么项目？分析什么？）

2. **缺少上下文**：
   - ❌ "性能优化"（哪个组件？当前指标？目标是多少？）

3. **混淆多个目标**：
   - ❌ "分析代码并对比框架还要评估GitHub项目"
   - ✅ 分为3个独立的研究任务

## 输出格式说明

系统会根据受众和场景自动选择最佳格式：

| 格式 | 适用场景 | 特点 |
|------|---------|------|
| **执行摘要** | 向管理层汇报 | 1-2页，强调业务价值 |
| **技术报告** | 技术评审 | 5-15页，深度分析+代码示例 |
| **对比表格** | 快速决策 | 1-2页，多维度评分矩阵 |
| **演示文稿** | 团队分享 | 10-20页PPT，可视化图表 |
| **博客文章** | 技术传播 | 1500-3000字，通俗易懂 |

你也可以**明确指定**输出格式：

```bash
生成【执行摘要】格式的报告
生成【技术报告】用于代码评审
生成【演示文稿】用于下周的技术分享会
```

| 场景 | 传统GoT成本 | DGoT成本 | 节省 |
|------|------------|---------|------|
| 代码质量分析 | $0.30 | $0.15 | 50% |

**节省来源**：
- 早停机制：平均减少30%迭代

- 智能缓存：减少重复检索

## 故障排除

### 问题：Agent匹配不准确

**症状**：系统选择了错误的Agent

**解决**：在任务描述中使用明确的关键词
- 代码分析 → "代码质量"、"性能瓶颈"
- 技术选型 → "对比"、"vs"、"选型"
- 项目评估 → "github.com"、"依赖安全"
- 架构评审 → "架构"、"设计"、"重构"

### 问题：输出格式不符合预期

**症状**：生成了技术报告但需要执行摘要

**解决**：明确指定输出格式
```bash
分析代码质量，生成【执行摘要】格式用于管理层汇报
```

### 问题：研究结果质量不够

**症状**：引用来源质量低

**解决**：明确质量要求
```bash
研究微服务架构，仅使用A-B级来源（论文和官方文档）
```
