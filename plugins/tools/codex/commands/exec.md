## 任务拆分规范

### 拆分原则

1. **独立性原则**
   - 每个子任务必须能够独立执行和验证
   - 任务之间尽量减少耦合和依赖
   - 每个任务有明确的输入输出边界

2. **最小化原则**
   - 单个任务的工作量适中，避免过大或过小
   - 每个任务聚焦单一目标
   - 避免任务包含多个不相关的子目标

3. **可验证性原则**
   - 每个任务必须有明确的验收标准
   - 验收标准应具体、可量化、可测试
   - 明确任务完成的定义（Definition of Done）

4. **并行友好原则**
   - 识别可并行执行的任务
   - 明确任务间的依赖关系
   - 避免多个并行任务操作同一模块或文件

### 拆分步骤

1. **理解需求**
   - 分析任务目标和约束条件
   - 识别关键功能点和非功能需求
   - 澄清不明确的需求点

2. **识别模块边界**
   - 根据系统架构划分模块
   - 确定每个模块的职责范围
   - 识别模块间的接口和依赖

3. **拆分子任务**
   - 按模块或功能点拆分
   - 确保每个任务符合拆分原则
   - 为每个任务定义目标、约束、验收标准

4. **确定依赖关系**
   - 绘制任务依赖图（不使用 mermaid）
   - 识别可并行执行的任务组
   - 确定串行执行的顺序

5. **验证拆分结果**
   - 检查是否覆盖所有需求
   - 验证任务间的依赖是否合理
   - 确认每个任务的可执行性

### 任务描述模板

每个子任务应包含以下要素：

```
任务ID: [唯一标识]
任务名称: [简洁明确的名称]
目标: [任务要达成的目标]
约束条件:
  - [技术约束]
  - [时间约束]
  - [资源约束]
验收标准:
  - [具体的验收条件1]
  - [具体的验收条件2]
依赖任务: [依赖的任务ID列表，无依赖则为空]
预估复杂度: [低/中/高]
涉及模块: [相关模块或文件路径]
```

### 并行执行策略

1. **并行任务分组**
   - 将无依赖的任务分为一组
   - 每组最多5个并行任务
   - 同一组任务不应操作同一文件或模块

2. **执行顺序规划**
   - 按依赖关系确定执行批次
   - 每批次内的任务可并行执行
   - 前一批次全部完成后启动下一批次

3. **冲突避免**
   - 明确每个任务的文件操作范围
   - 建立文件锁定或协调机制
   - 并行任务间通过清晰的接口交互

### 常见拆分模式

1. **按功能模块拆分**
   - 适用于功能独立的模块开发
   - 每个模块作为一个独立任务
   - 模块间通过定义好的接口协作

2. **按开发阶段拆分**
   - 设计阶段 → 实现阶段 → 测试阶段
   - 每个阶段可进一步细分子任务
   - 适用于复杂功能的渐进式开发

3. **按技术层次拆分**
   - 前端任务、后端任务、数据库任务
   - 适用于全栈开发场景
   - 各层任务可并行，但需协调接口

4. **按优先级拆分**
   - 核心功能优先，扩展功能次之
   - 高优先级任务先执行
   - 适用于时间紧迫或资源有限的场景

### 拆分检查清单

在完成任务拆分后，使用以下清单验证：

- [ ] 每个任务都有明确的目标和验收标准
- [ ] 任务粒度适中，不过大也不过小
- [ ] 任务间依赖关系清晰且合理
- [ ] 可并行任务已识别并分组
- [ ] 没有任务操作冲突（同一文件/模块）
- [ ] 所有任务覆盖了完整的需求范围
- [ ] 每个任务都有明确的约束条件
- [ ] 任务描述清晰，无歧义

### 异常处理

1. **任务执行失败**
   - 立即停止相关并行任务
   - 分析失败原因和影响范围
   - 重新规划任务拆分和执行顺序
   - 通过 `AskUserQuestion` 与用户确认新方案

2. **需求变更**
   - 评估变更对现有任务的影响
   - 调整受影响的任务或新增任务
   - 通过 `AskUserQuestion` 与用户确认调整方案
   - 更新任务清单和依赖关系

3. **依赖阻塞**
   - 识别阻塞原因
   - 评估是否可调整任务顺序
   - 必要时重新拆分任务以减少依赖
   - 及时与用户沟通解决方案
