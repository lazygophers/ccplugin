# 管理线索系统

帮助作者设计和管理悬疑推理故事中的线索系统和伏笔网络。

## 适用场景

| 场景 | 示例 |
|------|------|
| **线索设计** | `/novel-mystery clue 为密室案设计5条核心线索` |
| **红鲱鱼** | `/novel-mystery clue 设计3条误导性线索` |
| **伏笔管理** | `/novel-mystery clue 梳理当前所有伏笔的回收状态` |
| **逻辑验证** | `/novel-mystery clue 检验推理链条的完整性` |

## 执行流程

### 1. 确认需求方向

通过提问明确：
- **设计目标**：线索设计/红鲱鱼/伏笔管理/逻辑验证
- **案件概况**：案件类型和核心谜题
- **当前进度**：故事写到哪里，已有哪些线索
- **已有设定**：真相和推理路径的设定

### 2. 生成设计方案

根据需求激活 `Skills(clue-system)` 并生成：

#### 线索设计
- 核心线索清单和获取方式
- 辅助线索和氛围线索
- 线索之间的关联关系
- 线索出现的章节规划

#### 红鲱鱼设计
- 红鲱鱼的内容和误导方向
- 红鲱鱼存在的合理理由
- 排除红鲱鱼的推理过程
- 与真线索的配合节奏

#### 伏笔管理
- 伏笔清单和埋设位置
- 回收时机和回收方式
- 未回收伏笔的提醒
- 伏笔之间的连锁关系

#### 逻辑验证
- 证据链完整性检查
- 时间线可行性验证
- 推理唯一性分析
- 公平性评估

### 3. 交叉验证

检查线索系统的完整性：
- 核心线索是否全部已设计出现时机
- 红鲱鱼是否有合理的存在理由
- 所有伏笔是否都有回收计划
- 推理链条是否无断裂

### 4. 输出管理文档

以结构化格式输出线索管理方案，包含追踪清单。

## 设计原则

- **公平至上**：所有解谜线索必须在揭晓前呈现
- **缺失妙用**："不存在的东西"往往是最好的线索
- **管理追踪**：使用清单工具确保每条伏笔都能回收
- **密度均衡**：线索释放要有节奏，不能过密或过疏
