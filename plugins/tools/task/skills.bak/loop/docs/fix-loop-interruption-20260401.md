# 修复记录：Loop 流程阶段间中断问题（系统性修复）

> **注意**：本文档为历史修复记录。文中提到的 `plan-formatter` 已于 v0.0.186 合并至 `planner` 内部，不再作为独立组件存在。

**日期**：2026-04-01
**问题ID**：loop-stage-interruption
**严重程度**：高（影响所有阶段间转换）
**关联**：fix-planning-interruption-20260330.md（前次修复仅覆盖阶段2内部）

## 问题描述

task:loop 在执行过程中，经常在某个阶段完成后直接停止回复，而不是继续执行下一阶段。具体表现：

1. 阶段2（计划设计）完成后，直接输出 planner 返回的原始 JSON 并停止
2. 阶段3（任务执行）完成后停止，未进入结果验证
3. 阶段4（结果验证）完成后停止，未进入完成清理或失败调整
4. 其他阶段间转换也存在类似问题

## 根本原因分析

### 直接原因

SKILL.md 及各 phase 文档中，阶段之间的状态转换描述**缺少强制性的连续性语言**。

**问题示例**：

| 文件 | 原描述 | 问题 |
|------|--------|------|
| SKILL.md 阶段2 | "进入阶段3（任务执行）" | 无强制性，Claude 可能在输出后停止 |
| phase-5-execution.md | "成功 → Phase 6(结果验证)" | 只是说明，非执行指令 |
| phase-6-verification.md | "Phase 8（完成）" | 无"必须立即"等强制词汇 |

### 根本原因

**SKILL.md 是执行指南，不是功能文档**。Claude 读取 SKILL.md 后将其作为指令执行。如果指令中没有明确说"不允许停止"、"必须继续"，Claude 会按照默认行为模式——完成一个动作后结束回复。

### 与前次修复的关系

2026-03-30 的修复（fix-planning-interruption-20260330.md）解决了**阶段2内部**的中断问题（planner → plan-formatter → AskUserQuestion），但未解决**阶段之间**的中断问题。本次修复覆盖了所有阶段间的转换。

## 解决方案

### 1. 添加全局流程控制规则（SKILL.md）

在 `<execution>` 部分最前面添加"全局流程控制规则"，建立4条最高优先级规则：
- 禁止提前停止
- 阶段必须连续
- 禁止输出原始 JSON
- 强制完整性

### 2. 强化输出格式规范（SKILL.md）

添加"进度输出规范"，明确正确/错误的输出方式：
- 正确：人类可读的进度摘要
- 错误：原始 JSON 数据

### 3. 强化所有阶段的状态转换（SKILL.md + 5个 phase 文件）

为每个阶段的状态转换添加：
- "**必须立即**在同一回复中继续"强制语言
- "**禁止**在本阶段后结束回复"警告
- 表格增加"强制要求"列

### 4. 标注唯一允许结束的阶段（phase-8-finalization.md）

明确只有阶段6（Phase 8: Cleanup）是唯一允许结束 loop 的阶段。

## 经验教训

### SKILL.md 编写的核心原则

1. **SKILL.md 是执行指南，不是功能文档**
   - 每一句话都会被 Claude 当作指令执行
   - 缺少强制性语言 = 允许停止

2. **必须明确执行边界**
   - 什么时候可以结束回复
   - 什么时候绝对不可以结束回复
   - 唯一的出口是什么

3. **使用强制性语言**
   - "必须"、"禁止"、"立即"、"绝对不允许"
   - 避免"应该"、"建议"、"可以"

4. **添加自检规则**
   - 让 Claude 在每个阶段结束时自检
   - "我是否已到达阶段6且 finalizer 已执行？"

### 类似问题的预防检查清单

审查 SKILL.md 时，检查以下内容：
- [ ] 是否有全局流程控制规则（定义何时允许停止）？
- [ ] 每个阶段的状态转换是否包含"必须立即"等强制语言？
- [ ] 是否明确了唯一允许结束的阶段？
- [ ] 是否禁止了原始 JSON 输出？
- [ ] 是否有自检规则让 Claude 在每个阶段结束时验证？
