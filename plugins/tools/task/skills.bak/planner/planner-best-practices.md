# Planner 最佳实践与避坑指南

## 最佳实践

1. **中等深度探索优先**：先快速了解项目结构（扫描目录、README、配置文件），信息不足时再深入。一开始就深度分析所有源文件通常不必要。

2. **功能已存在时及时报告**：返回空 tasks 数组，在 report 中说明原因和功能位置。

3. **一次只问一个问题**：通过 `questions` 字段逐步确认需求，避免一次性提出多个问题。

4. **验收标准必须可量化**：使用数值指标（≥90%、<200ms）和明确验证方式，避免主观判断。

5. **报告简洁**：report ≤200字，突出任务数、依赖关系、预计时间。

6. **专用 agent/skills 优先**：探索→task:explorer-*，Go开发→golang:dev@golang，测试→tester。专用比通用（coder）更高效。

## 常见错误

### 1. 过度拆分

简单操作不应拆成多任务。错误：创建配置/写入配置/保存配置 → 正确：创建并配置API配置文件(1个任务)

**粒度判断**：修改≥2文件→过粗需拆分 | 只改1-2行→过细需合并 | 任务数>20→检查可合并项 | 描述含"和"/"且"→按语义拆分

### 2. 验收标准模糊

错误：`["代码质量好"]` → 正确：`["覆盖率≥90%", "API返回正确状态码", "Lint 0错误"]`

### 3. 缺少中文注释

Agent和Skills必须带中文注释。错误：`"agent": "coder"` → 正确：`"agent": "coder（开发者）"`

### 4. 循环依赖

违反DAG原则导致死锁。用拓扑排序验证。最多2个并行任务。

### 5. 结构化验收标准错误

必需字段：`id`+`type`+`description`+`priority`。exact_match需`verification_method`，quantitative_threshold需`metric`+`operator`+`threshold`。

### 6. 缺少Agent/Skills

tasks非空时每个任务必须指定agent+至少1个skill(带中文注释)。

## 验证检查清单

- [ ] 所有任务遵循 MECE 原则，原子性，粒度10-30分钟
- [ ] 依赖关系形成 DAG（无循环），并行度≤2
- [ ] tasks非空时 agent/skills 带中文注释和来源标注
- [ ] 验收标准可量化、可验证
- [ ] report ≤200字，包含关键信息
- [ ] 优先使用专用 agent/skills
- [ ] 制定计划前已发现全部可用的 agents 和 skills

## 关键要点

- **任务粒度**：单任务10-30分钟，产生可验证交付物
- **空任务数组**：功能已存在时返回，report 说明原因和位置
- **任务冲突**：检查 files 字段无重复，冲突任务串行化
