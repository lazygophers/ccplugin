# Task - 任务管理插件

<overview>

Task 是一个任务管理框架插件，提供规划、执行、验证和迭代的完整流程。它不绑定任何具体的执行 agents，而是让用户灵活选择其他插件或自定义 agents 来完成实际工作。这种设计使得框架可以适配任意技术栈和项目类型。

核心理念是基于 PDCA 循环（计划-执行-检查-改进）的持续迭代。通过 /loop 命令，Task 以 team leader 角色统一管理所有调度和用户交互，agents 只需通过 SendMessage 向 leader 上报，leader 通过 AskUserQuestion 与用户沟通。

</overview>

## 安装与命令

```bash
/plugin install task@ccplugin-market
```

| 命令 | 说明 |
|------|------|
| `/loop [任务目标]` | 进入循环迭代模式，作为 team leader 执行完整流程 |
| `/add [补充内容]` | 补充任务说明、纠正方向或添加约束 |
| `/cancel` | 取消当前执行，保留已完成工作 |

## 典型工作流

```
用户：/loop 实现用户认证功能

→ 初始化（Initialization）
  - 初始化状态变量和执行环境
  - 列出可用资源（skills、agents）

→ 提示词优化（Prompt Optimization）[新增功能]
  - 仅在第一轮（iteration=0）执行
  - 评估提示词质量（清晰度、完整性、可执行性）
  - 质量≥8分：静默跳过（完全无感知）
  - 质量<8分：通过 5W1H 框架向用户提问澄清
  - 质量<6分：搜索最新的提示词工程最佳实践
  - 生成优化后的结构化提示词
  - 示例：
    输入："优化代码" (质量2.3分)
    → 提问："优化什么方面？性能/质量/架构？"
    → 提问："当前性能指标？目标是多少？"
    输出："优化API性能，将响应时间从500ms降到200ms" (质量8.7分)

→ 计划设计（Planning / Plan）
  - 分析项目结构和技术栈
  - 分解为原子子任务（MECE 原则）
  - 建立依赖关系（DAG）
  - 为每个任务分配 agent 和 skills（带中文注释）
  - 定义可量化的验收标准

→ 计划确认（Plan Confirmation）
  - 生成计划文档（Markdown）
  - 输出执行流程图和验收标准
  - 用户确认计划

→ 任务执行（Execution / Do）
  - 判断任务数量（1 个任务不创建 team，多个任务创建 team）
  - 按依赖顺序调度执行
  - 支持并行/串行编排（最多 2 个并行）
  - 实时输出进度
  - 执行完成后删除 team（如果创建了）

→ 结果验证（Verification / Check）
  - 检查所有验收标准（acceptance_criteria）
  - 验收标准失败 → 失败调整
  - 验收标准通过 + 有建议事项 → 自动继续迭代（回到计划设计）
  - 验收标准通过 + 无建议 → 完成

→ 失败调整（Adjustment / Act）
  - 分析失败原因和停滞模式
  - 应用分级升级策略（retry → debug → replan → ask_user）
  - 指数退避（0秒 → 2秒 → 4秒）
  - 回到相应阶段（任务执行 或 计划设计）

→ 全部完成（Completion / Finalization）
  - 调用 finalizer agent 清理资源
  - 输出最终报告和统计信息
  - （team 已在任务执行阶段结束时删除）
```

## 设计原则

Task 插件的架构围绕几个关键决策展开。框架与执行分离是最核心的设计——Task 只负责流程编排，具体任务由外部 agents 完成，这样同一个框架可以驱动 golang、python、flutter 等不同技术栈的 agents。

/loop 命令作为唯一的 team leader，统一管理所有调度和用户交互。agents 不能直接向用户提问，必须通过 SendMessage 上报给 leader，由 leader 决定是否需要用户介入。这种集中式通信避免了多 agent 同时向用户提问造成的混乱。

任务拆分遵循原子性原则，每个子任务必须不可再分。并行执行时最多 2 个任务同时运行，且不能修改同一文件或模块。Team 按需创建——单任务不创建 team，多任务在执行阶段创建，执行完成后立即删除。

## Agent 来源

Task 插件不提供执行 agents，而是使用以下来源：其他插件提供的专业 agents（如 golang、python、flutter 插件），用户或项目自定义的 agents，以及通用 agents（如 general-purpose、Explore）。

## 终止条件

| 条件 | 触发 | 行为 |
|------|------|------|
| 目标达成 | 结果验证全部通过 | 正常退出，输出报告 |
| 停滞过多 | 连续 3 次相同错误 | 请求用户指导后继续（不退出） |
| 用户中断 | 用户主动中断 | 根据用户指令处理 |

循环没有最大迭代次数限制。停滞时请求用户指导，但不退出循环；获得用户指导后重置停滞计数器，继续执行。

## 提示词优化功能（新增）

### 功能介绍

提示词优化是 Task 插件在 v0.0.181 中新增的功能，旨在提升用户输入的质量，减少因需求不清导致的返工。

**核心特性**：
- **自动质量评估**：使用三维度（清晰度、完整性、可执行性）评分 0-10 分
- **智能静默**：高质量输入（≥8 分）完全静默跳过，无感知
- **结构化提问**：基于 5W1H 框架（What/Why/Who/When/Where/How）系统性收集缺失信息
- **最佳实践融合**：质量低时（<6 分）搜索最新的提示词工程技巧
- **不限提问次数**：确保信息完整，直到质量≥8 分

### 使用示例

#### 示例1：模糊输入 → 自动优化

```bash
用户输入：/loop 优化代码

→ [MindFlow·优化代码·提示词优化/0·进行中]

AI 评估：
- 清晰度：3分（目标不明确）
- 完整性：2分（缺少上下文）
- 可执行性：2分（无法分解）
- 综合得分：2.3分 → 需要优化

AI 提问1："关于优化目标，请问："
  1. 性能优化（减少响应时间）
  2. 代码质量优化（重构、测试覆盖）
  3. 架构优化（设计模式、可维护性）

用户回答："性能优化"

AI 提问2："关于性能指标，请问："
  "当前响应时间是多少？目标是多少？"

用户回答："当前 500ms，希望降到 200ms 以下"

AI 搜索：WebSearch("prompt engineering best practices 2025")

AI 生成优化提示：
"# 任务目标
优化 API 响应性能，将平均响应时间从 500ms 降低到 200ms 以下

## 技术上下文
- 当前性能：500ms
- 目标性能：<200ms

## 验收标准
1. 平均响应时间 < 200ms
2. P95 响应时间 < 300ms
3. 性能测试通过（1000 req/s 负载）"

→ [MindFlow·优化代码·提示词优化/0·completed]
优化报告：优化完成：2.3分 → 8.7分。通过2次提问澄清了性能目标和验收标准。

→ [MindFlow·优化API性能·计划设计/1·进行中]
（使用优化后的提示词继续流程）
```

### 质量评分标准

| 得分范围 | 质量等级 | AI 行为 |
|---------|---------|---------|
| 9-10分 | 优秀 | 静默跳过，不输出任何信息 |
| 8-8.9分 | 良好 | 静默跳过，不输出任何信息 |
| 6-7.9分 | 中等 | 提问澄清，不触发 WebSearch |
| 4-5.9分 | 较低 | 提问澄清 + WebSearch 搜索最佳实践 |
| 0-3.9分 | 很低 | 多次提问 + WebSearch 搜索最佳实践 |

### 5W1H 框架

提示词优化使用 5W1H 框架系统性收集信息：

- **What（目标）**：要实现什么功能？预期交付成果？
- **Why（动机）**：为什么需要？解决什么问题？
- **Who（受众）**：目标用户是谁？对用户体验有什么要求？
- **When（时间）**：什么时候完成？优先级如何？
- **Where（范围）**：影响哪些模块？部署在哪个环境？
- **How（方式）**：技术方案偏好？使用什么技术栈？
