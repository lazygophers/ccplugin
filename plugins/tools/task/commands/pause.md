# 暂停当前任务

暂停当前正在执行的 loop 流程，保留所有状态，稍后可通过 `/resume` 恢复。

## ✋ 前置条件

**必须有正在执行的 task:loop**，否则无法使用 `/pause`：

- ✅ **可用场景**：Loop 执行过程中（步骤 1-6 任意阶段）
- ❌ **不可用**：无活跃 Loop 任务时
- ❌ **不可用**：Loop 已暂停（避免重复暂停）

**检查方式**：TaskList 查询状态为 `in_progress` 且类型为 `loop` 的主任务

## 📋 快速参考

| 步骤 | 核心操作 | 工具 | 输出 |
|------|---------|------|------|
| 1️⃣ 停止执行 | 暂停所有 Agents | TaskStop | - |
| 2️⃣ 状态保存 | 标记为暂停状态 | TaskUpdate | paused + 暂停时间/原因 |
| 3️⃣ 保留团队 | 不删除 Team 资源 | - | Team 保持活跃 |
| 4️⃣ 输出报告 | 汇总暂停结果 | - | 已完成/已暂停/未开始 |

**核心原则**：暂停保留所有状态，可随时恢复，与 cancel 完全不同

## ⚠️ 核心原则

1. **状态保留**：所有任务状态、Team、Agents 信息完整保留
2. **可恢复**：通过 `/resume` 从暂停点继续执行
3. **不删除 Team**：与 `/cancel` 不同，Team 资源保持活跃
4. **暂停顺序**：TaskStop → TaskUpdate(paused) → 保留 Team
5. **可追溯**：记录暂停时间、暂停原因、暂停时所处步骤

## 暂停流程

### 1️⃣ 停止执行

**工具**：TaskStop

**操作**：
- 暂停所有正在运行的 Agents（不终止）
- 中断当前迭代循环
- 不再调度新的子任务

**与 cancel 的区别**：
- `pause`：暂停 Agents，可恢复
- `cancel`：终止 Agents，不可恢复

### 2️⃣ 状态保存

**工具**：TaskUpdate

**状态分类**：
- **主任务** → 标记为 `paused`（记录暂停时间、原因、所处步骤）
- **执行中任务** → 保持 `in_progress` 状态，但暂停执行
- **已完成任务** → 保持 `completed` 状态
- **待执行任务** → 保持 `pending` 状态

**记录信息**：
```
metadata: {
  paused_at: "2026-03-13 19:30:00",
  paused_reason: "用户主动暂停",
  paused_at_step: 4,  // 暂停时所处的步骤
  paused_iteration: 2  // 暂停时的迭代次数
}
```

### 3️⃣ 保留团队

**操作**：不执行 TeamDelete，保留 Team 和 Agents 信息

**保留内容**：
- Team ID
- Agents 配置
- Agents 执行上下文
- 任务分配信息

**⚠️ 资源占用**：暂停状态会保留 Team 资源，建议暂停时间不超过 30 分钟

### 4️⃣ 输出暂停报告

**输出格式**：
```
⏸️  [暂停] 任务已暂停

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
主任务：实现用户认证系统
当前步骤：4️⃣ 任务执行 | 迭代：第 2 次
暂停时间：2026-03-13 19:30:00
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

✅ 已完成：T1 (数据库迁移)
⏸️  已暂停：T2 (实现用户模型, 45%), T3 (实现订单模型, 30%)
⏸️  未开始：T4 (实现支付模块), T5 (集成测试)

Team 保留：team_abc123 (2个 agents)

💡 提示：
  - 使用 /resume 从暂停点恢复执行
  - 使用 /status 查看暂停状态
  - 使用 /cancel 终止任务（将删除 Team）
```

## 📊 与其他 Commands 的对比

| 对比项 | pause | cancel | loop |
|--------|-------|--------|------|
| 意图 | 暂时中断，稍后恢复 | 终止任务，不再执行 | 开始新任务 |
| 主任务状态 | `paused` | 保持 `in_progress` | 创建新任务 |
| 子任务状态 | 保留原状态 | completed/cancelled/pending | - |
| Team 处理 | ✅ 保留 | ❌ 删除 | 新建（如需要） |
| 后续操作 | `/resume` 恢复 | `/loop` 重新开始 | - |
| 资源占用 | ⚠️  占用（保留 Team） | ✅ 释放 | - |

## 适用场景

| ✅ **应该使用 /pause** | 说明 |
|----------------------|------|
| 等待外部条件 | 等待代码审查、等待数据、等待环境 |
| 手动检查中间状态 | 想查看当前输出，决定是否继续 |
| 临时中断 | 需要处理其他紧急任务，稍后继续 |
| 调整计划 | 发现问题，需要思考调整策略 |

| ❌ **不应该使用 /pause** | 推荐操作 |
|------------------------|----------|
| 长时间中断（>30分钟） | 使用 `/cancel` 释放资源 |
| 不打算继续执行 | 使用 `/cancel` 终止 |
| 只需调整部分内容 | 使用 `/add` 补充说明 |

## 注意事项

- **资源占用**：暂停状态会保留 Team 资源，长时间暂停可能影响性能
- **建议时长**：暂停时间不超过 30 分钟，超过建议使用 `/cancel`
- **暂停后操作**：暂停后可使用 `/status` 查看状态，使用 `/add` 补充说明
- **恢复时机**：随时可通过 `/resume` 恢复，从暂停点继续执行
- **避免重复**：已暂停的任务不能再次暂停，需先 `/resume` 或 `/cancel`
