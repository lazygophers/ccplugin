# 恢复暂停任务

从暂停点恢复执行，继续之前暂停的 loop 流程。

## ✋ 前置条件

**必须有已暂停的 task:loop**，否则无法使用 `/resume`：

- ✅ **可用场景**：Loop 已暂停（状态为 `paused`）
- ❌ **不可用**：无暂停的 Loop 任务时
- ❌ **不可用**：Loop 正在执行中（避免重复恢复）

**检查方式**：TaskList 查询状态为 `paused` 且类型为 `loop` 的主任务

## 📋 快速参考

| 步骤 | 核心操作 | 工具 | 输出 |
|------|---------|------|------|
| 1️⃣ 检查状态 | 验证任务可恢复 | TaskGet | 暂停时间、步骤、迭代 |
| 2️⃣ 恢复状态 | 标记为执行中 | TaskUpdate | in_progress |
| 3️⃣ 重启 Agents | 恢复被暂停的 Agents | Agent调用 | 后台任务启动 |
| 4️⃣ 继续执行 | 从暂停点继续 | - | 进入暂停时所处步骤 |

**核心原则**：从暂停点无缝恢复，不重新规划，不丢失进度

## ⚠️ 核心原则

1. **无缝恢复**：从暂停点继续执行，不重新开始
2. **保持上下文**：保留暂停前的所有状态和进度
3. **不重新规划**：继续执行暂停前的计划，不进入步骤 2
4. **Agent 恢复**：重新启动被暂停的 Agents，继续执行任务
5. **可追溯**：记录恢复时间和恢复原因

## 恢复流程

### 1️⃣ 检查状态

**工具**：TaskGet

**操作**：
1. 获取主任务信息，验证状态为 `paused`
2. 提取暂停时的上下文：
   - 暂停时间（paused_at）
   - 暂停所处步骤（paused_at_step）
   - 暂停时迭代次数（paused_iteration）
   - Team ID
3. 检查 Team 是否仍然存在
4. 检查子任务状态

**如果检查失败**：
```
❌ 无法恢复任务

原因：Team 已被删除（可能因为超时清理）

建议：使用 /loop 重新开始
```

### 2️⃣ 恢复状态

**工具**：TaskUpdate

**操作**：
1. 更新主任务状态：`paused` → `in_progress`
2. 记录恢复信息：
   ```
   metadata: {
     resumed_at: "2026-03-13 19:45:00",
     resumed_reason: "用户恢复执行",
     paused_duration: "15分钟"  // 暂停时长
   }
   ```
3. 保留暂停前的迭代次数和步骤信息

### 3️⃣ 重启 Agents

**工具**：Agent 调用

**操作**：
1. 查找暂停前正在执行的任务（状态为 `in_progress`）
2. 重新启动对应的 Agents：
   - 使用原有的 Agent 配置
   - 传递原有的上下文（working_directory、skills等）
   - 后台模式启动（background=True）
3. 恢复任务分配

**示例**：
```
# 暂停前：
T2 (in_progress, 45%) ← coder_1
T3 (in_progress, 30%) ← coder_2

# 恢复后：
T2 (in_progress, 从45%继续) ← coder_1（重启）
T3 (in_progress, 从30%继续) ← coder_2（重启）
```

### 4️⃣ 继续执行

**操作**：根据暂停时所处的步骤，继续执行循环

**恢复点**：
- **步骤 1-3**（信息收集/计划设计/计划确认）→ 继续当前步骤
- **步骤 4**（任务执行）→ 重启 Agents，继续执行
- **步骤 5**（结果验证）→ 继续验证流程
- **步骤 6**（失败调整）→ 继续调整策略

**不会执行**：
- ❌ 不重新进入步骤 2（计划设计）
- ❌ 不重新创建 Team
- ❌ 不重新规划任务

### 5️⃣ 输出恢复报告

**输出格式**：
```
▶️  [恢复] 任务已恢复执行

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
主任务：实现用户认证系统
暂停时长：15分钟
恢复步骤：4️⃣ 任务执行 | 迭代：第 2 次
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

✅ 已完成：T1 (数据库迁移)
🔄 继续执行：T2 (实现用户模型, 从45%继续), T3 (实现订单模型, 从30%继续)
⏸️  待执行：T4 (实现支付模块), T5 (集成测试)

Agents 已重启：
  - coder_1: 继续执行 T2
  - coder_2: 继续执行 T3

💡 提示：
  - 使用 /status 查看当前状态
  - 使用 /add 补充说明或调整方向
  - 使用 /pause 再次暂停
```

## 📊 恢复与重新开始的区别

| 对比项 | resume (恢复) | loop (重新开始) |
|--------|--------------|----------------|
| 意图 | 从暂停点继续 | 开始新任务 |
| 前置条件 | 有暂停的 loop | 无要求 |
| 任务规划 | 使用暂停前的计划 | 重新规划（步骤2） |
| 进度保留 | ✅ 保留（从暂停点继续） | ❌ 从头开始 |
| Team 处理 | 使用现有 Team | 创建新 Team（如需要） |
| 迭代次数 | 保持暂停前的次数 | 从 1 开始 |

## 适用场景

| ✅ **应该使用 /resume** | 说明 |
|----------------------|------|
| 短时间暂停后恢复 | 暂停时间 ≤ 30 分钟，想继续执行 |
| 检查完中间状态 | 查看了中间输出，决定继续执行 |
| 外部条件已满足 | 等待的外部条件（审查、数据等）已就绪 |
| 临时中断后返回 | 处理完其他紧急任务，返回继续执行 |

| ❌ **不应该使用 /resume** | 推荐操作 |
|------------------------|----------|
| 长时间暂停（>30分钟） | Team 可能已被清理，使用 `/loop` 重新开始 |
| 想改变任务目标 | 使用 `/cancel` + `/loop` 重新开始 |
| 想重新规划任务 | 使用 `/cancel` + `/loop` 重新开始 |

## 注意事项

- **Team 存活**：恢复前检查 Team 是否仍然存在，超时可能被清理
- **进度连续性**：Agents 会从暂停点继续，不重新开始任务
- **上下文保持**：恢复后使用暂停前的 Anchor、user_additions 等上下文
- **迭代次数**：保持暂停前的迭代次数，不重置为 1
- **可多次暂停**：恢复后可以再次暂停，不限制次数
- **失败处理**：如果恢复失败（如 Team 丢失），提示用户使用 `/loop` 重新开始
