# 任务规划

**使用 `planner` Agent** 对任务进行结构化分解。

## 规划流程

### 步骤 1：需求理解

首先理解任务：
- 明确用户的核心目标
- 识别隐含约束（技术栈、规范、时间）
- 对不明确的部分**必须提问确认**

### 步骤 2：信息收集

收集项目上下文：
- 读取相关代码文件和配置
- 了解项目结构和技术栈
- 查看 CLAUDE.md 中的规范要求
- 检查现有测试和文档

### 步骤 3：任务分解

将任务分解为子任务列表，每个子任务包含：

| 字段 | 必填 | 说明 |
|------|------|------|
| 描述 | 是 | 做什么、为什么做 |
| 输入 | 是 | 依赖的文件、数据或前置任务输出 |
| 输出 | 是 | 产出的文件、代码变更 |
| 依赖 | 是 | 前置子任务（无则标"无"） |
| 验收标准 | 是 | 可自动验证的检查项 |
| 执行建议 | 否 | 推荐的 Agent 或工具 |

### 步骤 4：生成执行计划

输出结构化计划：

```markdown
# 执行计划：[任务标题]

## 目标
[1-2 句话]

## 子任务

### T1: [标题]
- 描述：...
- 输入：...
- 输出：...
- 依赖：无
- 验收标准：
  - [ ] ...

### T2: [标题]
- 描述：...
- 输入：T1 的输出
- 输出：...
- 依赖：T1
- 验收标准：
  - [ ] ...

## 执行策略
- 并行组 1：T1, T3（无依赖）
- 串行：T2（依赖 T1）
- 并行组 2：T4, T5（依赖 T2）

## 风险点
- ...
```

### 步骤 5：注册任务

1. 为每个子任务创建 `TaskCreate` 任务条目
2. 通过 `TaskUpdate` 建立依赖关系（`addBlockedBy`）
3. 附加 metadata（target_files、retry_count 等）
4. 展示完整任务清单给用户确认

### 步骤 6：确认计划

将计划展示给用户，确认后可通过 `/exec` 开始执行。

## 执行检查清单

### 需求理解检查

- [ ] 核心目标已明确（可用 1-2 句话描述）
- [ ] 技术栈约束已识别（语言、框架、库版本）
- [ ] 项目规范已确认（检查 CLAUDE.md、.claude/skills/）
- [ ] 不明确的需求已通过 `AskUserQuestion` 确认
- [ ] 隐含约束已识别（性能、安全、兼容性）

### 信息收集检查

- [ ] 已读取相关代码文件
- [ ] 已了解项目目录结构
- [ ] 已查看 CLAUDE.md 中的规范要求
- [ ] 已检查现有测试覆盖情况
- [ ] 已识别依赖的第三方库和服务

### 任务分解检查

每个子任务必须包含：
- [ ] 描述清晰（做什么、为什么做）
- [ ] 输入明确（依赖的文件、数据、前置任务输出）
- [ ] 输出明确（产出的文件、代码变更）
- [ ] 依赖关系明确（前置任务或标注"无"）
- [ ] 验收标准可自动验证（测试、lint、编译）
- [ ] 执行建议明确（推荐的 Agent 或工具）

### 任务分解质量检查

- [ ] 符合**单一职责**原则（每个子任务只做一件事）
- [ ] 符合**可验证**原则（每个子任务都有自动化验收标准）
- [ ] 符合**无文件冲突**原则（并行任务不操作同一文件）
- [ ] 符合**适度粒度**原则（不过度分解简单任务）
- [ ] 任务粒度适中（1-4 小时完成）
- [ ] 所有任务覆盖了完整需求（无遗漏）

### 依赖关系检查

- [ ] 依赖图无环（DAG）
- [ ] 并行任务已识别并分组
- [ ] 并行任务不操作同一文件或模块
- [ ] 串行任务顺序合理
- [ ] 关键路径已识别（影响总时长的任务链）

### 执行计划检查

- [ ] 执行策略清晰（并行组、串行顺序）
- [ ] 风险点已识别
- [ ] 并行组不超过 2 个任务
- [ ] 每个并行组的任务互不影响

### 任务注册检查

- [ ] 每个子任务已通过 `TaskCreate` 创建
- [ ] 依赖关系已通过 `TaskUpdate(addBlockedBy)` 建立
- [ ] metadata 包含必要信息（target_files、retry_count 等）
- [ ] 任务状态初始化为 `pending`

### 用户确认检查

- [ ] 计划已展示给用户
- [ ] 计划格式清晰易读
- [ ] 用户已确认计划合理
- [ ] 用户已确认可通过 `/exec` 开始执行

## 规划原则

- **单一职责**：每个子任务只做一件事
- **可验证**：每个子任务都有自动化验收标准
- **无文件冲突**：并行任务不操作同一文件
- **适度粒度**：不过度分解简单任务

## 注意事项

### 规划质量保证

- 不要过度分解简单任务（如只需修改 1-2 个文件）
- 验收标准必须可自动验证（避免主观判断）
- 依赖关系要准确（错误的依赖会导致执行失败）

### 常见错误

- ❌ 任务粒度过大（包含多个独立功能）
- ❌ 任务粒度过小（过度分解导致任务数量过多）
- ❌ 验收标准模糊（如"代码质量好"）
- ❌ 依赖关系错误（导致执行顺序混乱）
- ❌ 遗漏关键任务（需求覆盖不完整）

### 最佳实践

- ✅ 从用户视角定义验收标准
- ✅ 优先识别可并行的任务
- ✅ 为复杂任务预留缓冲时间
- ✅ 明确标注技术风险点
- ✅ 计划展示前先自我审查
