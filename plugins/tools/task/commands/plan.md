# 任务规划

使用 `planner` Agent 对任务进行结构化分解。

## 规划流程

### 步骤 1：需求理解

首先理解任务：
- 明确用户的核心目标
- 识别隐含约束（技术栈、规范、时间）
- 对不明确的部分**必须提问确认**

### 步骤 2：信息收集

收集项目上下文：
- 读取相关代码文件和配置
- 了解项目结构和技术栈
- 查看 CLAUDE.md 中的规范要求
- 检查现有测试和文档

### 步骤 3：任务分解

将任务分解为子任务列表，每个子任务包含：

| 字段 | 必填 | 说明 |
|------|------|------|
| 描述 | 是 | 做什么、为什么做 |
| 输入 | 是 | 依赖的文件、数据或前置任务输出 |
| 输出 | 是 | 产出的文件、代码变更 |
| 依赖 | 是 | 前置子任务（无则标"无"） |
| 验收标准 | 是 | 可自动验证的检查项 |
| 执行建议 | 否 | 推荐的 Agent 或工具 |

### 步骤 4：生成执行计划

输出结构化计划：

```markdown
# 执行计划：[任务标题]

## 目标
[1-2 句话]

## 子任务

### T1: [标题]
- 描述：...
- 输入：...
- 输出：...
- 依赖：无
- 验收标准：
  - [ ] ...

### T2: [标题]
- 描述：...
- 输入：T1 的输出
- 输出：...
- 依赖：T1
- 验收标准：
  - [ ] ...

## 执行策略
- 并行组 1：T1, T3（无依赖）
- 串行：T2（依赖 T1）
- 并行组 2：T4, T5（依赖 T2）

## 风险点
- ...
```

### 步骤 5：注册任务

1. 为每个子任务创建 `TaskCreate` 任务条目
2. 通过 `TaskUpdate` 建立依赖关系（`addBlockedBy`）
3. 附加 metadata（target_files、retry_count 等）
4. 展示完整任务清单给用户确认

### 步骤 6：确认计划

将计划展示给用户，确认后可通过 `/exec` 开始执行。

## 规划原则

- **单一职责**：每个子任务只做一件事
- **可验证**：每个子任务都有自动化验收标准
- **无文件冲突**：并行任务不操作同一文件
- **适度粒度**：不过度分解简单任务
