# 审查任务完成质量

**使用 `reviewer` Agent** 对已完成的任务进行全面审查。

## 审查流程

### 1. 确定审查范围

- 指定任务名称：审查特定子任务
- "全部"：审查所有已完成的子任务

### 2. 执行审查

对每个待审查的子任务：

**功能验证**
- 运行所有相关测试
- 验证输出是否符合预期
- 检查边界条件处理

**代码质量**
- 运行 lint/format 工具
- 检查安全性（OWASP Top 10）
- 检查性能隐患
- 检查代码规范遵循度

**验收标准**
- 逐项检查每个验收条件
- 标注通过/未通过及原因

### 3. 输出审查报告

```markdown
# 审查报告

## 总体：[通过/有条件通过/不通过]

### T1: [标题] - 通过
- [x] 验收标准 1
- [x] 验收标准 2
- 代码质量：良好

### T2: [标题] - 不通过
- [x] 验收标准 1
- [ ] 验收标准 2：[失败原因]
- 代码质量：需修复
- 问题：[具体问题描述，附文件路径和行号]

## 必须修复
1. [问题描述]

## 建议改进
1. [改进建议]
```

### 4. 后续行动

- **通过**：输出完成总结，清理 TaskList 中的已完成条目
- **有条件通过**：列出建议改进项，由用户决定是否修复
- **不通过**：列出必须修复项，建议使用 `/loop` 迭代修复

### 5. 清理

审查通过后自动执行清理：
- 清理 TaskList 中已完成的任务条目
- 确保不留残余的临时状态

## 执行检查清单

### 审查范围确定检查

- [ ] 明确审查范围（特定任务或全部任务）
- [ ] 通过 `TaskList` 获取待审查任务列表
- [ ] 通过 `TaskGet` 获取每个任务的详情和验收标准
- [ ] 确认任务状态为 `completed`（已完成）

### 功能验证检查

- [ ] 运行所有相关测试（单元测试、集成测试）
- [ ] 验证输出是否符合预期
- [ ] 检查边界条件处理（空值、极值、异常输入）
- [ ] 验证错误处理逻辑
- [ ] 确认功能符合需求规格

### 代码质量检查

- [ ] 运行 lint 工具（如 golangci-lint, eslint, ruff）
- [ ] 运行 format 检查（如 gofmt, prettier, black）
- [ ] 检查代码安全性（OWASP Top 10）
- [ ] 检查性能隐患（N+1 查询、内存泄漏、低效算法）
- [ ] 检查代码规范遵循度（命名、结构、注释）

### 验收标准检查

- [ ] 逐项检查每个验收条件
- [ ] 标注每项通过/未通过及具体原因
- [ ] 确认自动化验证结果（测试通过率、覆盖率）
- [ ] 确认手动验证结果（UI 交互、用户体验）

### 审查报告生成检查

- [ ] 报告包含总体评价（通过/有条件通过/不通过）
- [ ] 每个任务的审查结果明确
- [ ] 失败项包含具体问题描述、文件路径和行号
- [ ] 区分**必须修复**和**建议改进**
- [ ] 提供后续行动建议

### 后续行动检查

**通过**：
- [ ] 输出完成总结
- [ ] 清理 TaskList 中的已完成条目
- [ ] 确认无残余临时状态

**有条件通过**：
- [ ] 列出建议改进项
- [ ] 通过 `AskUserQuestion` 询问用户是否修复
- [ ] 记录用户决策

**不通过**：
- [ ] 列出必须修复项
- [ ] 建议使用 `/loop` 迭代修复
- [ ] 不清理 TaskList（保留任务状态以便修复）

### 清理检查（仅审查通过时）

- [ ] 使用 `TaskUpdate` 标记所有任务为 `archived`
- [ ] 清理 TaskList 中的已完成条目
- [ ] 确认无残余的临时状态
- [ ] 记录完成时间戳

## 工具调用说明

### 必需工具

| 工具 | 用途 | 调用时机 |
|------|------|---------|
| `TaskList` | 获取待审查任务列表 | 审查开始时 |
| `TaskGet` | 获取任务详情和验收标准 | 审查每个任务时 |
| `TaskUpdate` | 更新任务状态和审查结果 | 审查完成后 |
| `Bash` | 运行测试和 lint 命令 | 功能验证和代码质量检查时 |

### 可选工具

| 工具 | 用途 | 调用时机 |
|------|------|---------|
| `AskUserQuestion` | 询问用户是否修复建议改进项 | 有条件通过时 |
| `Read` | 读取代码文件进行人工审查 | 需要深入审查代码时 |

### Agent 调用

| Agent | 用途 | 触发条件 |
|------|------|---------|
| `reviewer` | 执行代码审查 | **推荐**作为 preferred_subagent |
| `tester` | 运行测试验证 | 需要额外测试验证时 |

### Skills 引用

| Skill | 用途 | 必需性 |
|------|------|--------|
| `Skills(task:verification)` | 任务验收规范 | 推荐 |
| `Skills(task:core)` | 任务管理核心规范 | 推荐 |
| 语言特定 Skills | 代码质量检查标准 | 按语言选择 |

```markdown
### T1: [任务标题] - 通过
- [x] 验收标准 1
- [x] 验收标准 2
- 代码质量：良好
- 测试覆盖率：95%

### T2: [任务标题] - 不通过
- [x] 验收标准 1
- [ ] 验收标准 2：[失败原因]
- 代码质量：需修复
- 问题：
  - `file.go:123` - [具体问题描述]
  - `api.ts:45` - [具体问题描述]

## 必须修复
1. [问题描述] - 位置：file.go:123

## 建议改进
1. [改进建议] - 位置：utils.py:67
```

## 注意事项

### 审查质量保证

- 严格对照验收标准逐项检查
- 不通过的项目必须附具体问题和位置
- 区分**必须修复**（阻塞发布）和**建议改进**（不阻塞）
- 审查结果必须客观可复现

### 常见错误

- ❌ 未运行测试就标记通过
- ❌ 问题描述模糊（缺少文件路径和行号）
- ❌ 混淆**必须修复**和**建议改进**
- ❌ 审查通过后忘记清理 TaskList
- ❌ 主观判断代码质量（未运行 lint）

### 最佳实践

- ✅ 使用自动化工具进行客观验证
- ✅ 问题描述包含具体位置和修复建议
- ✅ 审查报告结构清晰易读
- ✅ 审查通过后及时清理临时状态
- ✅ 不通过时提供明确的后续行动建议
