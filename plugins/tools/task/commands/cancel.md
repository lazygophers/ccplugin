# 取消当前任务

立即终止当前正在执行的 Loop 或 Exec 流程。

## 取消流程

### 1. 停止执行

- 终止所有正在运行的 Agent（通过 TaskStop）
- 中断当前迭代循环
- 不再调度新的子任务

### 2. 状态保存

取消不等于丢弃，保留已完成的工作：
- 已完成的子任务状态保持 `completed`
- 执行中的子任务标记为 `cancelled`
- 待执行的子任务保持 `pending`
- 通过 `TaskUpdate` 更新所有任务状态

### 3. 输出取消报告

```
[取消] 任务已终止

已完成：T1, T2
已取消：T3（执行中被终止）
未开始：T4, T5

可通过 /exec 恢复执行未完成的任务
```

### 4. 恢复选项

取消后用户可以：
- `/exec` — 恢复执行剩余的子任务
- `/plan` — 重新规划未完成的部分
- `/loop` — 用新的目标重新开始迭代

## 执行检查清单

### 取消前检查

- [ ] 确认当前有活跃的 Loop 或 Exec 任务正在执行
- [ ] 确认取消操作不会导致数据丢失
- [ ] 确认已保存的代码变更可以保留

### 取消执行检查

- [ ] 通过 `TaskStop` 终止所有正在运行的 Agent
- [ ] 中断当前迭代循环
- [ ] 停止调度新的子任务
- [ ] 更新所有任务状态（completed/cancelled/pending）
- [ ] 通过 `TaskUpdate` 记录取消原因和时间戳

### 状态保存检查

- [ ] 已完成任务状态保持为 `completed`
- [ ] 执行中任务标记为 `cancelled`
- [ ] 待执行任务保持为 `pending`
- [ ] 任务 metadata 记录取消时的上下文信息

### 输出检查

- [ ] 输出包含已完成任务列表
- [ ] 输出包含已取消任务列表（附取消原因）
- [ ] 输出包含未开始任务列表
- [ ] 提示用户后续恢复选项（/exec, /plan, /loop）

### 清理检查

- [ ] TaskList 中保留所有任务状态（不删除）
- [ ] 团队资源未提前释放（支持后续恢复）
- [ ] 临时状态已记录到 metadata 中

## 工具调用说明

### 必需工具

| 工具 | 用途 | 调用时机 |
|------|------|---------|
| `TaskStop` | 终止正在运行的 Agent | 取消操作开始时 |
| `TaskUpdate` | 更新任务状态 | 每个任务状态变更时 |
| `TaskList` | 列出所有任务 | 生成取消报告时 |

### 可选工具

| 工具 | 用途 | 调用时机 |
|------|------|---------|
| `AskUserQuestion` | 确认取消操作 | 可能导致数据丢失时 |

## 注意事项

### 安全保证

- 取消操作是安全的，不会丢弃已完成的工作
- 已提交的代码变更不会被回退
- 任务状态保留在 TaskList 中，支持后续恢复

### 取消时机

- 用户主动中断（Ctrl+C 或明确取消指令）
- 发现任务规划错误，需要重新规划
- 遇到无法解决的阻塞，需要用户介入

### 不适用场景

- 任务即将完成（建议等待完成）
- 只需要调整部分子任务（使用 `/add` 补充说明）
- 需要修复失败任务（使用 `/loop` 迭代修复）
