# 执行任务计划（/exec）

按照已确认的执行计划，调度并执行所有子任务。

## 执行流程

### 1. 加载执行计划

- 通过 `TaskList` 加载所有子任务
- 确认所有任务状态为 `pending`（即用户已确认的规划）
- 解析依赖图，确定执行顺序

### 2. 按组执行

```
对于每个执行组（按依赖顺序）：
    1. 确认前置依赖已全部完成
    2. 更新当前任务状态为 in_progress
    3. 组建 Agent 团队，分发组内任务
       - 独立任务可并行（最多 2 个并行）
       - 并行任务不得修改同一文件或同一包
       - 有依赖的任务串行
    4. 收集执行结果
    5. 验证每个子任务的验收标准
    6. 更新任务状态（completed/failed）
    7. 失败的任务交给 debugger Agent 分析修复
    8. 进入下一组
```

### 3. 执行监控

每完成一个子任务，输出进度：

```
[进度] T1 创建数据模型 ......... 已完成
[进度] T2 实现 API 接口 ......... 执行中
[进度] T3 编写单元测试 ......... 待执行
[进度] T4 集成测试 ............. 待执行

总进度：1/4 (25%)
```

### 4. 失败处理策略

遵循统一的失败升级链：
1. executor 自行调整重试
2. debugger 介入诊断根因
3. reviewer 审查任务定义，必要时 planner 重新规划
4. 仍然失败则向用户报告，请求指导

特殊情况直接升级：
- 依赖缺失 → 暂停执行，向用户报告
- 设计问题 → 暂停执行，建议重新规划

### 5. 完成汇总

所有子任务完成后：
1. 触发 reviewer Agent 进行整体审查（Oracle 验证）
2. 输出完成摘要
3. 列出所有变更的文件
4. **不在此阶段清理** — 清理由 `/review` 审查通过后执行

## 执行原则

- 严格按照计划执行，不擅自增加或跳过子任务
- 每个子任务状态变更时通过 `TaskUpdate` 更新状态和 metadata
- 并行任务数不超过 2，并行任务不得修改同一文件或包
- 遇到阻塞时及时向用户报告
