# 项目改进建议

深度分析项目，提供架构、代码质量、性能、安全等方面的系统性改进建议。

## ✋ 前置条件

**无前置条件**：可以在任何时候使用 `/suggest`

- ✅ **可用场景**：任何时候（无需活跃的 loop）
- ✅ **典型用途**：项目进入维护期，想找改进点；代码质量下降，需系统性优化

## 📋 快速参考

| 步骤 | 核心操作 | 分析维度 | 输出 |
|------|---------|---------|------|
| 1️⃣ 深度扫描 | 全面分析项目 | 代码质量、架构、性能、安全 | 问题清单 |
| 2️⃣ 问题分类 | 按严重程度分类 | 严重/重要/一般 | 分类报告 |
| 3️⃣ 生成建议 | 提供改进方案 | 短期/中期/长期 | 改进计划 |
| 4️⃣ 输出报告 | 格式化输出 | - | Markdown 完整报告 |

**核心原则**：全面分析，系统性改进，长期规划

## ⚠️ 核心原则

1. **全面覆盖**：从代码质量、架构、性能、安全等多维度分析
2. **分级建议**：按严重程度（严重/重要/一般）和时间跨度（短期/中期/长期）分类
3. **可执行**：每个建议都提供具体的改进方案和执行路径
4. **数据驱动**：基于实际代码分析，不是泛泛而谈
5. **平衡取舍**：考虑改进成本和收益，优先高性价比的改进

## 执行流程

### 1️⃣ 深度扫描

**工具**：Read, Glob, Grep, Bash, Agent(Explore)

**扫描内容**：

1. **架构分析**
   - 分析目录结构
   - 识别模块和依赖关系
   - 检测循环依赖
   - 评估耦合度

2. **代码质量分析**
   - 计算圈复杂度（CC）
   - 检测重复代码
   - 检查代码风格（如果有 lint 配置）
   - 统计测试覆盖率
   - 评估文档覆盖率

3. **性能分析**
   - 识别大文件（> 500 行）
   - 检测 N+1 查询问题
   - 检查缓存使用
   - 分析并发处理

4. **安全分析**
   - 扫描依赖漏洞（如果有 lock 文件）
   - 检测敏感信息（密钥、密码、token）
   - 检查输入验证
   - 审查认证授权逻辑

5. **工程实践分析**
   - 检查 CI/CD 配置
   - 检查日志实践
   - 评估错误处理
   - 检查部署配置

### 2️⃣ 问题分类

**分类维度**：

1. **按严重程度**
   - 🔴 **严重**：影响安全、稳定性、核心功能
   - 🟡 **重要**：影响性能、可维护性、用户体验
   - 🟢 **一般**：影响代码风格、文档、工具链

2. **按时间跨度**
   - **短期**（1-2 周）：立即可执行，影响明显
   - **中期**（1-2 月）：需要规划，影响长期质量
   - **长期**（3-6 月）：战略性改进，架构级变化

### 3️⃣ 生成改进建议

**建议格式**：

- **问题描述**：是什么问题
- **影响分析**：为什么要改进
- **改进方案**：具体怎么做
- **预估工作量**：需要多长时间
- **优先级**：严重程度 + 时间跨度
- **参考资料**：最佳实践、文档链接

## 📝 代码质量改进建议 (60/100)

### ⚠️  问题

#### 🔴 问题 1：高复杂度模块
**描述**：3 个文件圈复杂度 > 10

| 文件 | CC | 问题函数 |
|------|----|---------|
| `models/session.py` | 15 | `validate_session()` (CC=8) |
| `api/auth.py` | 12 | `login()` (CC=7) |
| `utils/token.py` | 11 | `verify_token()` (CC=6) |

**影响**：
- 难以理解和维护
- 测试困难（需要更多测试用例）
- 容易引入 Bug

**改进方案**：
1. 重构 `validate_session()`：拆分为多个小函数
2. 使用卫语句（Guard Clauses）减少嵌套
3. 提取复杂条件为有意义的变量

**示例**（重构前 → 后）：
```python
# 重构前（CC=8）
def validate_session(session):
    if session:
        if session.is_active:
            if session.expires_at > now():
                if session.user_id:
                    return True
    return False

# 重构后（CC=2）
def validate_session(session):
    if not session:
        return False
    if not session.is_active:
        return False
    if session.expires_at <= now():
        return False
    return bool(session.user_id)
```

**预估工作量**：1-2 天

**优先级**：🔴 高优先级（短期改进）

**执行命令**：`/loop 重构 session/auth/token 模块，降低复杂度到 CC<10`

---

#### 🟡 问题 2：重复代码
**描述**：检测到 5 处重复代码（> 10 行）

**位置**：
- `api/user.py:45-60` ≈ `api/session.py:30-45`（用户验证逻辑）
- `models/user.py:80-95` ≈ `models/session.py:60-75`（时间戳处理）

**影响**：
- 维护成本高（修改需要同步多处）
- 容易遗漏同步，导致不一致

**改进方案**：
1. 提取公共逻辑到 `utils/validation.py`

**预估工作量**：半天

**优先级**：🟡 中优先级（短期改进）

#### 🟡 问题 3：测试覆盖率不足

**未覆盖代码**：

**影响**：

## ⚡ 性能改进建议 (70/100)

### ⚠️  问题

#### 🟡 问题 1：数据库查询未优化
**描述**：Session 查询未使用索引

**位置**：`models/session.py:get_by_token()`

**影响**：

**预估工作量**：1-2 小时

## 🔒 安全改进建议 (80/100)

### ⚠️  问题

#### 🔴 问题 1：敏感信息硬编码
**描述**：JWT 密钥硬编码在 `api/auth.py:10`

```python
SECRET_KEY = "my-secret-key-123"  # ❌ 硬编码
```

**影响**：
- 密钥泄露风险（代码仓库、日志）
- 生产环境安全隐患

**改进方案**：
1. 使用环境变量管理密钥
2. 使用 `python-dotenv` 加载 `.env` 文件
3. 生产环境使用密钥管理服务（如 AWS Secrets Manager）

**示例**：
```python
import os
from dotenv import load_dotenv

load_dotenv()
SECRET_KEY = os.getenv("JWT_SECRET_KEY")  # ✅ 环境变量
```

**预估工作量**：1-2 小时

**优先级**：🔴 高优先级（立即修复）

**执行命令**：`/loop 将 JWT 密钥迁移到环境变量`

## 📊 与其他 Commands 的关系

| Command | 关系 | 说明 |
|---------|------|------|
| **next** | 互补 | next 聚焦当前紧急任务，suggest 聚焦长期改进 |
| **plan** | 后续 | suggest 识别问题，plan 规划改进方案 |
| **loop** | 后续 | suggest → plan → loop（完整改进流程） |

**典型工作流**：
```

   ↓

```

## 注意事项

- **全面分析**：suggest 会深度扫描项目，可能耗时较长（取决于项目大小）
- **建议性质**：suggest 只提供建议，不执行任何代码修改
- **优先级权衡**：建议按优先级逐步执行，不要一次性改太多
- **成本收益**：考虑改进成本和收益，优先高性价比的改进
- **定期执行**：建议每季度或半年执行一次，跟踪项目健康度
- **数据依赖**：suggest 的质量取决于项目信息的完整性（测试报告、lint 配置等）
- **长期视角**：suggest 侧重长期改进，与 next 的短期视角互补
