# 补充当前任务

在任务执行（Loop 或 Exec）过程中，向当前任务注入额外的上下文信息。

## 适用场景

| 场景 | 示例 |
|------|------|
| **补充说明** | `/add 数据库用 PostgreSQL，不要用 SQLite` |
| **纠正方向** | `/add 不要改 auth 模块，只改 session 模块` |
| **添加约束** | `/add 必须兼容 Python 3.11` |
| **调整目标** | `/add 先不管性能，优先保证功能正确` |
| **提供线索** | `/add 错误可能在 utils/token.py 的 verify 方法中` |

## 处理流程

### 1. 接收补充信息

将用户的补充内容标记为高优先级上下文，分类为：

- **约束追加**：新的技术约束或限制条件
- **方向纠正**：修改当前的执行方向
- **信息补充**：提供额外的背景信息或线索
- **目标调整**：修改或细化任务目标
- **范围变更**：扩大或缩小任务范围

### 2. 注入当前任务

将补充信息注入到当前执行上下文：
- 如果在 Loop 中：下一次 Gather 阶段会读取此信息，并据此调整 Act 策略
- 如果在 Execute 中：当前或下一个子任务的执行会考虑此信息
- 如果在 Plan 中：更新计划中受影响的子任务定义

### 3. 确认反馈

输出简要确认：

```
[补充已注入] 类型：方向纠正
内容：不要改 auth 模块，只改 session 模块
影响：当前 Loop 的后续迭代将跳过 auth 相关修改
```

## 与其他命令的交互

| 当前状态 | `/add` 的效果 |
|---------|--------------|
| `/loop` 执行中 | 下一次迭代的 Gather 阶段读取并应用 |
| `/exec` 执行中 | 下一个子任务执行时考虑 |
| `/plan` 规划中 | 更新规划上下文，影响后续分解 |
| 无活跃任务 | 记录为待用上下文，下次启动任务时加载 |

## 注意事项

### 补充信息生效规则

- 补充信息**立即生效**，但不会回滚已完成的步骤
- 如果补充内容与已完成的工作矛盾，会提示用户确认是否需要回退
- 多次 `/add` 的内容会**累积**，不会互相覆盖

### 补充信息优先级

- **约束追加**：高优先级，必须遵守
- **方向纠正**：高优先级，立即调整
- **目标调整**：中优先级，影响后续规划
- **信息补充**：低优先级，作为参考
- **范围变更**：高优先级，需重新评估计划

### 常见使用场景

| 场景 | 补充内容示例 | 影响范围 |
|------|------------|---------|
| 发现技术限制 | `/add 数据库用 PostgreSQL，不要用 SQLite` | 后续所有数据库相关任务 |
| 纠正执行方向 | `/add 不要改 auth 模块，只改 session 模块` | 当前 Loop 迭代 |
| 添加技术约束 | `/add 必须兼容 Python 3.11` | 所有代码实现任务 |
| 调整优先级 | `/add 先不管性能，优先保证功能正确` | 验收标准和执行策略 |
| 提供调试线索 | `/add 错误可能在 utils/token.py 的 verify 方法中` | 当前失败任务 |

### 常见错误

- ❌ 补充内容模糊不清（如"改好一点"）
- ❌ 补充内容与已完成工作严重冲突但未确认
- ❌ 忘记记录补充信息到任务 metadata
- ❌ 补充信息未明确影响范围

### 最佳实践

- ✅ 补充内容清晰具体
- ✅ 明确补充信息的类型和优先级
- ✅ 说明补充信息的影响范围
- ✅ 检测并处理与已完成工作的冲突
- ✅ 及时更新任务 metadata 记录补充信息
