# Verifier 验证检查清单

本文档包含验证前后的检查清单、注意事项和最佳实践。

## 验证前检查清单

### 任务完成性检查

- [ ] 所有任务已执行完成（状态为 completed）
- [ ] 没有遗漏或跳过的任务
- [ ] 所有任务都有明确的验收标准
- [ ] 任务之间的依赖关系已满足

### 验收标准检查

- [ ] 每个任务的验收标准已明确定义
- [ ] 验收标准可量化且可验证
- [ ] 验收标准避免使用绝对词汇（all, always, never）
- [ ] 验收标准符合 SMART 原则：
  - **S**pecific（具体的）
  - **M**easurable（可度量的）
  - **A**chievable（可达成的）
  - **R**elevant（相关的）
  - **T**ime-bound（有时限的）

### 环境和工具检查

- [ ] 测试环境已准备就绪
- [ ] 必要的测试工具可用
- [ ] 数据库和服务已启动
- [ ] CI/CD 流水线状态正常

## 注意事项

### Do's ✓

1. **✓ 始终使用正确的 agent 调用方式**
   ```python
   Agent(agent="task:verifier", prompt="...")
   ```

2. **✓ 检查并处理所有三种状态**
   - `passed` - 正常退出
   - `suggestions` - 询问用户
   - `failed` - 进入失败调整

3. **✓ 验证所有任务，不要遗漏**
   ```python
   # 确保所有任务都在 verified_tasks 中
   assert len(verification_result["verified_tasks"]) == len(all_tasks)
   ```

4. **✓ 对于 suggestions，必须询问用户**
   ```python
   if status == "suggestions":
       user_decision = AskUserQuestion("是否继续优化？")
   ```

5. **✓ 记录详细的验证日志**
   ```python
   log_verification_result(verification_result)
   ```

6. **✓ 生成可追溯的验证报告**
   ```python
   generate_verification_report(verification_result, timestamp)
   ```

### Don'ts ✗

1. **✗ 不要接受模糊的验收标准**
   ```python
   # ❌ 错误
   "代码质量好"  # 无法量化

   # ✓ 正确
   "Lint 检查 0 错误 0 警告"  # 可量化
   ```

2. **✗ 不要忽略 suggestions 中的建议**
   ```python
   # ❌ 错误
   if status == "suggestions":
       return "exit"  # 直接退出，忽略建议

   # ✓ 正确
   if status == "suggestions":
       ask_user_about_suggestions()  # 询问用户
   ```

3. **✗ 不要跳过回归测试检查**
   ```python
   # ❌ 错误
   # 只验证新功能，不检查回归

   # ✓ 正确
   run_regression_tests()  # 确保无影响已有功能
   ```

4. **✗ 不要修改 verifier 返回的 JSON 结构**
   ```python
   # ❌ 错误
   verification_result["status"] = "custom_status"

   # ✓ 正确
   # 保持原始结构，不做修改
   ```

5. **✗ 不要在验证失败时自动重试所有任务**
   ```python
   # ❌ 错误
   if status == "failed":
       retry_all_tasks()  # 盲目重试

   # ✓ 正确
   if status == "failed":
       analyze_failures()  # 分析后再决定
   ```

## 验收标准最佳实践

### 可测试性（Testability）

**好的标准**：
- "所有单元测试通过"
- "API 返回正确的 HTTP 状态码"
- "函数返回预期的输出"

**不好的标准**：
- "代码写得好"
- "功能运行正常"
- "用户满意"

### 可度量性（Measurability）

**好的标准**：
- "测试覆盖率 ≥ 90%"
- "响应时间 < 200ms"
- "内存使用 < 100MB"

**不好的标准**：
- "测试覆盖率高"
- "响应速度快"
- "内存使用合理"

### 独立性（Independence）

**好的标准**：
```json
{
  "acceptance_criteria": [
    "单元测试覆盖率 ≥ 90%",  // 可独立验证
    "所有 API 返回正确状态码",  // 可独立验证
    "Lint 检查 0 错误"  // 可独立验证
  ]
}
```

**不好的标准**：
```json
{
  "acceptance_criteria": [
    "测试通过且代码质量好",  // 两个标准混在一起
    "功能完整并且性能达标"  // 无法独立验证
  ]
}
```

### 关注结果（Outcome-Focused）

**好的标准**：
- "用户可以成功登录"
- "订单创建后发送确认邮件"
- "数据保存到数据库"

**不好的标准**：
- "实现了登录函数"
- "写了发送邮件的代码"
- "使用了 ORM 框架"

### 避免绝对词汇

**好的标准**：
- "≥ 95% 的测试用例通过"
- "常见场景下响应时间 < 200ms"
- "主要功能覆盖率 ≥ 90%"

**不好的标准**：
- "所有测试用例通过"（all）
- "总是在 200ms 内响应"（always）
- "从不发生错误"（never）

## 终止条件决策表

| 验收状态 | 条件 | 询问用户？ | Loop 行为 |
|---------|------|-----------|----------|
| `passed` | 所有标准通过，无建议 | 否 | 正常退出 |
| `suggestions` | 标准通过，有优化建议 | 是 | 用户决定 |
| `suggestions` + 用户选择继续 | - | - | 继续迭代 |
| `suggestions` + 用户选择完成 | - | - | 正常退出 |
| `failed` | 部分标准未满足 | 否 | 失败调整 |

## 常见问题 FAQ

### Q1: 如何判断验收标准是否合理？

**A**: 使用 SMART 原则检查：
- **S**pecific（具体）：标准描述清晰具体
- **M**easurable（可度量）：可以用数值或明确的方式验证
- **A**chievable（可达成）：在当前条件下可实现
- **R**elevant（相关）：与任务目标直接相关
- **T**ime-bound（有时限）：有明确的完成时间要求

### Q2: suggestions 状态什么时候使用？

**A**: 当满足以下条件时使用 `suggestions`：
- 所有必需的验收标准都已满足
- 存在可以改进但非必需的地方
- 改进属于"锦上添花"而非"雪中送炭"

**示例**：
- 必需：测试覆盖率 ≥ 90% ✓
- 建议：可以添加更多边界测试（已达标，但可以更好）

### Q3: 如何处理部分任务失败的情况？

**A**: 返回 `failed` 状态，并在 `failures` 字段中详细说明：
```json
{
  "status": "failed",
  "failures": [
    {
      "task_id": "T2",
      "criterion": "测试覆盖率 ≥ 90%",
      "actual": "75%",
      "reason": "错误处理分支未覆盖"
    }
  ]
}
```

Loop 将进入失败调整阶段，分析原因并修复。

### Q4: 回归测试失败怎么办？

**A**: 回归测试失败应该返回 `failed` 状态：
```json
{
  "status": "failed",
  "report": "回归测试失败：2 个原有测试用例不通过",
  "summary": {
    "regression_tests_passed": false
  }
}
```

这表示新改动影响了已有功能，必须修复。

### Q5: 验证时间过长怎么优化？

**A**: 可以采用以下策略：
1. **并行验证**：独立的验证任务可以并行执行
2. **增量验证**：只验证变更相关的部分
3. **缓存结果**：缓存不变部分的验证结果
4. **分层验证**：先快速验证，再深度验证
