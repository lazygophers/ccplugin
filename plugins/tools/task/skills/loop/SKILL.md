# Agentic Loop 规范

## 循环模型

```
┌─────────────────────────────────────┐
│           Agentic Loop              │
│                                     │
│  ┌────────┐  ┌─────┐  ┌────────┐  │
│  │ Gather │→│ Act  │→│ Verify │  │
│  └────────┘  └─────┘  └───┬────┘  │
│       ↑                    │       │
│       │    ┌────────┐      │       │
│       └────│ Adjust │←─────┘       │
│            └────────┘   (if fail)  │
│                                     │
│         ──── break (if pass) ────  │
└─────────────────────────────────────┘
```

## 四阶段详解

### Gather（收集）
**目标**：了解当前状态，确定本次迭代要做什么。

执行内容：
- 读取当前代码/测试/错误状态
- 收集可用的 skills 和 agents，评估本次迭代需要的能力
- 如果是首次迭代，收集完整上下文
- 如果是后续迭代，聚焦上次失败的反馈
- 通过 `TaskList` / `TaskGet` 确认当前任务状态
- 确定本次迭代的目标，识别可并行的独立子目标
- 选择合适的 agents 分配任务

**输出**：当前状态摘要 + 本次迭代目标 + Agent 分配方案

### Act（执行）
**目标**：做出原子性的变更，支持多 Agent 并行。

执行原则：
- 将独立子目标分配给不同 agents 并行执行
- **并行数最大不超过 2 个**，超过时排队等待
- **并行任务不得修改同一文件或同一包**
- 每个 agent 每次只做一个逻辑变更
- 变更必须是可观测的（有新代码、新配置等产出）
- 基于 Gather 收集的信息和 Agent 分配方案决策

**输出**：变更描述 + 修改的文件列表 + 各 Agent 执行结果

### Verify（验证）
**目标**：检查 Act 的效果是否达成目标。

验证方法：
- 运行测试命令
- 检查输出是否符合预期
- 对比错误数量的变化
- 检查是否引入新问题

**输出**：通过/失败 + 具体结果

### Adjust（调整）
**目标**：分析失败原因，调整下次策略。

仅在 Verify 失败时执行：
- 分析失败的具体原因
- 总结本次尝试的教训
- 制定不同于上次的策略
- 如果连续相同错误，标记为"无进展"

**输出**：失败分析 + 下次策略

## Oracle 验证

Loop 的 Verify 阶段采用 Oracle 验证模式：

1. executor 完成变更并自评"通过"
2. **不直接信任自评结果**，调用 reviewer 进行独立验证
3. reviewer 独立运行测试、检查代码，给出独立判断
4. 只有 reviewer 也确认通过，才真正标记目标达成
5. reviewer 否决 → 将否决原因作为反馈注入下一次迭代

## 三次失败升级

同一问题连续失败时的升级链：

```
第 1 次失败 → executor 自行调整重试
第 2 次失败 → debugger 介入诊断根因
第 3 次失败 → reviewer 审查任务定义 → planner 重新规划
仍然失败   → AskUserQuestion 请求用户指导
```

## 终止条件

**Loop 模式设计原则**：必须完成全部工作直到符合预期，不存在迭代次数上限。

| 条件 | 触发 | 行为 |
|------|------|------|
| 目标达成 | Verify 通过 + Oracle 验证通过 | 正常退出，输出成功报告 |
| 停滞过多 | 连续 3 次相同错误，已尝试多种恢复策略 | 请求用户指导后继续（不退出） |
| 用户中断 | 用户主动发送中断信号 | 根据用户指令处理 |

**重要**：
- 无最大迭代次数限制，Loop 会持续执行直到所有验收标准通过
- 停滞时请求用户指导，但**不会退出循环**
- 获得用户指导后重置停滞计数器，继续执行

## 防无限循环机制

### 进展检测
每次迭代记录：
- 错误信息的哈希值
- 测试通过/失败数量
- 修改的文件和行数

### 停滞判断
- 连续 3 次错误信息完全相同 → 停滞
- 连续 3 次测试结果无变化 → 停滞
- 变更量为零（Act 没做任何修改） → 停滞

### 停滞处理
1. **不停止循环**，继续执行
2. 输出所有迭代的摘要
3. 分析可能的根本原因
4. 向用户提供 2-3 个可选方向和已尝试的策略
5. 等待用户指导
6. **应用用户指导后重置停滞计数器**
7. 继续循环执行

## 迭代日志格式

```
=== 迭代 1 ===
[Gather] 首次执行，3 个测试失败：test_login, test_auth, test_session
[Act] 修复 auth.py 中的 token 验证逻辑
[Verify] 失败 - 2 个测试仍失败：test_auth, test_session（进展：3→2）
[Adjust] token 验证已修复，剩余问题在 session 处理，下次聚焦 session.py

=== 迭代 2 ===
[Gather] 2 个测试失败：test_auth, test_session，聚焦 session 处理
[Act] 修复 session.py 中的过期检查逻辑
[Verify] 通过 - 所有测试通过

=== Loop 完成 ===
状态：成功
迭代次数：2
变更文件：auth.py, session.py
```

## 适用场景指南

| 场景 | 是否适合 Loop | 原因 |
|------|-------------|------|
| 修复测试失败 | 适合 | 有明确的验证条件 |
| 性能优化到目标值 | 适合 | 有可量化的目标 |
| lint 错误修复 | 适合 | 有明确的通过条件 |
| 实现新功能 | 不太适合 | 用 plan+execute 更好 |
| 大规模重构 | 不适合 | 用 plan+execute |
| 探索性研究 | 不适合 | 用 deepresearch |
