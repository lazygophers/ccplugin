# 任务执行最佳实践

## Gather-Act-Verify 模式

每个子任务的执行遵循三步循环：

### Gather（收集）
在动手前收集足够的上下文：
- 读取要修改的文件，理解现有代码
- 查看相关测试，了解预期行为
- 检查项目规范和约定
- 确认前置任务的输出可用

**反模式**：不读代码就开始写 —— 这会导致与现有代码风格不一致。

### Act（执行）
基于收集的信息执行变更：
- 每次只做一个逻辑变更
- 遵循项目的代码规范
- 保持变更最小化
- 不做任务范围外的"改进"

**反模式**：一次改太多东西 —— 出错时难以定位问题。

### Verify（验证）
变更后立即验证：
- 运行相关测试
- 检查编译/lint 是否通过
- 对比输出与预期
- 逐项检查验收标准

**反模式**：改完不验证就进入下一个任务 —— 问题会在后面放大。

## 最小变更原则

### 做什么
- 只修改任务定义中要求的内容
- 只添加实现功能所必需的代码
- 只修改必要的文件

### 不做什么
- 不重构周围的代码
- 不添加"以后可能用到"的功能
- 不修改不相关的文件
- 不升级不相关的依赖
- 不改变现有的代码风格

## 安全执行规范

### 操作前检查
- 确认目标文件存在且状态正确
- 确认没有未提交的冲突变更
- 确认依赖的服务/工具可用

### 变更保护
- 不执行不可逆的操作（除非任务明确要求）
- 大批量变更前先验证小范围
- 保留原始状态的恢复能力

### 失败处理
- 失败后不暴力重试
- 分析失败原因后再决定策略
- 超过 2 次重试仍失败则上报

## 完成信号（Completion Promise）

子任务执行完成时，必须输出明确的完成信号：

```markdown
## T[N]: [标题] - 完成

[PROMISE] 所有验收标准已通过，变更已验证 [/PROMISE]
```

完成信号的作用：
- 为 orchestrator 提供明确的完成判断依据
- 触发 Oracle 验证流程
- 防止 agent 在任务完成后继续不必要的操作

**只有当所有验收标准都通过时才能输出 PROMISE 标记。**

## 失败升级路径

执行失败时的升级策略：

| 阶段 | 行为 |
|------|------|
| 第 1 次失败 | 自行分析错误，调整实现方式重试 |
| 第 2 次失败 | 输出详细的失败上下文，请求 debugger 介入诊断根因 |
| 第 3 次失败 | 上报给 orchestrator，reviewer 审查任务定义，必要时 planner 重新规划 |
| 仍然失败 | orchestrator 通过 AskUserQuestion 请求用户指导 |

每次失败都必须通过 `TaskUpdate` 记录失败原因到任务的 metadata 中。

## 执行报告规范

每个子任务完成后输出：

```markdown
## T[N]: [标题] - [成功/失败]

### 变更
- `path/to/file.py`: [变更描述]

### 验收
- [x] 标准 1
- [ ] 标准 2：[失败原因]

### 备注
[需要后续任务注意的信息]
```
