# Planner 最佳实践

本文档包含 Planner 的最佳实践、验证检查清单和常见问题。

## 最佳实践

### 1. 优先使用中等深度探索

**策略**：
- 先快速了解项目结构（中等深度）
- 发现信息不足时再深入探索
- 避免一开始就深度分析（耗时）

**示例**：
```
中等深度探索：
- 扫描目录结构
- 读取 README 和主要配置文件
- 了解技术栈和架构

深度探索（仅在必要时）：
- 分析所有源文件
- 理解详细实现
- 追踪所有依赖
```

### 2. 发现功能已存在时及时报告

**策略**：
- 分析时检查功能是否已实现
- 如果已存在，返回空 tasks 数组
- 在 report 中说明原因

**示例**：
```json
{
  "status": "completed",
  "report": "分析结果：用户认证功能已在 internal/auth 模块完整实现，包含 JWT 生成/验证、中间件和完整测试。无需额外开发。",
  "tasks": [],
  "dependencies": {},
  "parallel_groups": []
}
```

### 3. 一次只问一个问题

**策略**：
- 通过 `questions` 字段返回问题
- 等待用户回答后再继续
- 避免一次性提出多个问题

**示例**：
```json
{
  "status": "questions",
  "questions": [
    "是否需要添加 OAuth 集成功能？"
  ]
}

// 收到用户回答后继续
```

### 4. 验收标准必须可量化

**策略**：
- 使用数值指标（≥ 90%、< 200ms）
- 使用明确的验证方式（所有测试通过）
- 避免主观判断

**示例**：
```json
{
  "acceptance_criteria": [
    "单元测试覆盖率 ≥ 90%",
    "所有 API 返回正确状态码",
    "Lint 检查 0 错误 0 警告",
    "性能测试通过（响应时间 < 200ms）"
  ]
}
```

### 5. 保持报告简洁

**策略**：
- report 字段 ≤ 200 字
- 突出关键信息：任务数、依赖关系、预计时间
- 避免冗余描述

**好的报告**：
```json
{
  "report": "计划：3个子任务。T1：JWT 工具（coder）→ T2：认证中间件（coder）→ T3：测试覆盖（tester）。依赖：T2→T3。预计完成时间：2小时。"
}
```

**不好的报告**：
```json
{
  "report": "经过深入分析项目结构和技术栈，我们发现需要实现用户认证功能。首先需要创建 JWT 工具函数来处理 token 的生成和验证，然后基于这个工具函数实现认证中间件，最后编写完整的测试用例来验证功能的正确性。整个过程预计需要大约 2 个小时左右的时间。"
  // 过于冗长，超过 200 字
}
```

## 验证检查清单

在生成计划后，使用以下检查清单验证质量：

### 任务分解
- [ ] 所有任务遵循 MECE 原则
- [ ] 每个任务都是原子性的
- [ ] 任务粒度合理（10-30 分钟）

### 依赖关系
- [ ] 依赖关系形成 DAG（无循环）
- [ ] 所有依赖都是必要的
- [ ] 并行度 ≤ 2

### Agent/Skills
- [ ] 所有 Agent 带中文注释
- [ ] 所有 Skills 带中文注释
- [ ] Agent 和 Skills 分配合理

### 验收标准
- [ ] 所有任务都有验收标准
- [ ] 验收标准可量化
- [ ] 验收标准可验证

### 报告质量
- [ ] report 字段 ≤ 200 字
- [ ] 包含关键信息（任务数、依赖、时间）
- [ ] 语言简洁清晰

## 常见问题 FAQ

### Q1: 如何判断任务粒度是否合适？

**A**: 合适的任务粒度应该满足：
- 单个任务执行时间：10-30 分钟
- 任务产生可验证的交付物
- 任务可以独立测试和验证

**判断标准**：
- 太大：执行时间 > 1 小时，难以并行
- 太小：执行时间 < 5 分钟，管理成本高
- 合适：10-30 分钟，原子性强

### Q2: 什么时候应该返回空 tasks 数组？

**A**: 以下情况应该返回空 tasks 数组：
- 功能已存在且满足需求
- 没有找到需要改动的地方
- 用户要求已被满足

**注意**：
- 必须在 report 中说明原因
- 提供功能位置和现状
- 确认无需额外开发

### Q3: 如何处理任务之间的冲突？

**A**: 避免任务冲突的方法：
- 确保任务之间文件不冲突
- 使用依赖关系控制执行顺序
- 避免同时修改同一文件

**冲突检测**：
- 检查 files 字段是否重复
- 检查并行任务是否访问同一资源
- 使用依赖关系串行化冲突任务
