# Planner 最佳实践

本文档包含 Planner 的最佳实践、验证检查清单和常见问题。

<best_practices>

## 最佳实践

优先使用中等深度探索。先快速了解项目结构（扫描目录、读取 README 和主要配置文件、了解技术栈和架构），发现信息不足时再深入探索。一开始就深度分析所有源文件和依赖关系通常是不必要的，只会浪费时间。

发现功能已存在时及时报告。分析时如果发现用户请求的功能已实现，返回空 tasks 数组并在 report 中说明原因和功能位置，避免不必要的重复开发。

```json
{
  "status": "completed",
  "report": "分析结果：用户认证功能已在 internal/auth 模块完整实现。无需额外开发。",
  "tasks": [],
  "dependencies": {},
  "parallel_groups": []
}
```

一次只问一个问题。通过 `questions` 字段返回问题，等待用户回答后再继续。一次性提出多个问题会让用户感到困惑，逐步确认能更好地理解需求。

```json
{
  "status": "questions",
  "questions": ["是否需要添加 OAuth 集成功能？"]
}
```

验收标准必须可量化。使用数值指标（>= 90%、< 200ms）和明确的验证方式（所有测试通过），避免主观判断。可量化的标准消除了"是否完成"的歧义。

保持报告简洁。report 字段不超过 200 字，突出关键信息（任务数、依赖关系、预计时间），避免冗余描述。简洁的报告让 Team Leader 和用户能快速掌握计划要点。

好的报告：
```json
{"report": "计划：3个子任务。T1：JWT 工具（coder）→ T2：认证中间件（coder）→ T3：测试覆盖（tester）。依赖：T2→T3。预计完成时间：2小时。"}
```

## 验证检查清单

任务分解：所有任务遵循 MECE 原则、每个任务都是原子性的、任务粒度合理（10-30 分钟）。

依赖关系：依赖关系形成 DAG（无循环）、所有依赖都是必要的、并行度不超过 2。

Agent/Skills（强制要求）：
  - [ ] 当 tasks 不为空时，每个任务都有 agent 字段
  - [ ] 当 tasks 不为空时，每个任务都有非空 skills 数组
  - [ ] 所有 Agent 带中文注释（格式：name（中文注释）或 name（中文注释）@source）
  - [ ] 所有 Skills 带中文注释（格式同上）
  - [ ] Agent 来源灵活（task/其他插件/项目自定义/通用）
  - [ ] Skills 来源灵活（语言插件/通用技能/项目技能）
  - [ ] Loop 内部调用明确指定 @task 来源
  - [ ] 探索类任务使用 task:explorer-* agents
  - [ ] 实现类任务根据技术栈选择合适 agent（golang:dev/python:dev/coder）

验收标准：所有任务都有验收标准、验收标准可量化、验收标准可验证。

报告质量：report 字段不超过 200 字、包含关键信息（任务数、依赖、时间）、语言简洁清晰。

</checklist>

## 关键要点

**任务粒度判断**：合适的任务粒度应该满足：单个任务执行时间 10-30 分钟、任务产生可验证的交付物、任务可以独立测试和验证。太大（超过 1 小时）难以并行，太小（不到 5 分钟）管理成本过高。

**空任务数组**：功能已存在且满足需求、没有找到需要改动的地方、或用户要求已被满足时，应返回空 tasks 数组。必须在 report 中说明原因，提供功能位置和现状。

**任务冲突处理**：确保任务之间文件不冲突（检查 files 字段是否重复），使用依赖关系控制执行顺序，避免并行任务访问同一资源。如果检测到文件冲突，使用依赖关系将冲突任务串行化。

</faq>
