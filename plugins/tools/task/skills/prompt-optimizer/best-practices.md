# 提示词工程最佳实践

## 质量评分标准

三维评分（各0-10）：**清晰度**（目标/术语/歧义）、**完整性**（5W1H覆盖/上下文/边界）、**可执行性**（可分解/可量化验收/依赖明确）。综合=三项均值。评分仅作为质量参考，不影响是否执行优化（每次迭代必须优化）。

## 核心原则

| 原则 | 要求 | 好的示例 | 差的示例 |
|------|------|---------|---------|
| 明确性 | 具体动词+精确术语+量化指标 | "实现JWT认证" "响应从500ms降到200ms" | "添加认证" "让API更快" |
| 上下文 | 项目类型+技术栈+当前状态+依赖 | 完整的技术上下文描述 | 缺少技术栈信息 |
| 范围边界 | In Scope/Out of Scope/影响范围 | 明确包含和排除列表 | 无边界定义 |
| 验收标准 | 功能(exact_match)+性能(threshold)+质量+安全 | "覆盖率≥90%" "P95<200ms" | "代码质量要好" |

## 5W1H 框架

| 维度 | 核心问题 | 关键信息 |
|------|---------|---------|
| What | 实现什么/交付什么/成功标准 | 功能列表、交付物 |
| Why | 解决什么问题/带来什么价值 | 业务背景、期望收益 |
| Who | 目标用户/体验要求 | 用户类型、UX标准 |
| When | 完成时间/优先级/里程碑 | 截止日期、阶段划分 |
| Where | 影响模块/部署环境/交互组件 | 修改范围、不影响范围 |
| How | 技术方案/工具选型/架构约束 | 技术栈、兼容性要求 |

## 优化模板

```markdown
# 任务目标
[具体动词] + [精确对象] + [可量化目标]

## 技术上下文
- 项目/技术栈/当前状态/痛点

## 具体要求
1. [可验证的功能点]

## 范围边界
- 包含/不包含/影响范围

## 验收标准
- 功能/性能/质量/安全（均可量化）
```

## 反模式

避免：过于简单（缺上下文/验收标准）、过于复杂（30+功能应拆分）、隐含假设（未说明具体技术）、模糊验收（"好/快/高"不可量化）。

## 检查清单

清晰度：具体动词+精确术语+无歧义。完整性：5W1H全覆盖。可执行性：可分解+可量化+依赖明确。格式：Markdown+层次清晰+关键前置+≤500字。
