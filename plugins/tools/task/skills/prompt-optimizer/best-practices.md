# 提示词工程最佳实践（2025版）

本文档提供详细的提示词优化指南，帮助用户编写清晰、可执行的任务描述。

---

## 一、核心原则

### 1. 明确性原则（Clarity）

**定义**：提示词必须无歧义，使用精确的术语和具体的动词。

**为什么重要**：模糊的描述会导致理解偏差，浪费时间在错误的方向上。

**如何实践**：

- **使用具体动词**：
  - ✅ 实现、添加、修复、优化、重构、迁移
  - ❌ 处理、改进、优化（未说明方向）、做、弄

- **提供精确术语**：
  - ✅ "实现 JWT 认证"（明确是 JWT）
  - ❌ "添加认证"（未说明认证方式）

- **量化指标**：
  - ✅ "优化 API 响应时间，从 500ms 降到 200ms"
  - ❌ "让 API 更快"

**示例对比**：

| 模糊表述 | 明确表述 |
|---------|---------|
| "优化代码" | "优化 API 性能，将平均响应时间从 500ms 降低到 200ms 以下" |
| "添加功能" | "添加用户登录功能，支持邮箱+密码认证，包含密码重置和 remember me" |
| "修复 bug" | "修复用户登录时 token 过期导致的 401 错误" |
| "改进系统" | "重构认证模块，使用 JWT 替换 Session，支持无状态认证" |

---

### 2. 上下文原则（Context）

**定义**：提供充分的背景信息，包括项目类型、技术栈、当前状态、依赖关系。

**为什么重要**：缺乏上下文会导致假设错误，技术方案可能与现有架构不兼容。

**必需的上下文信息**：

1. **项目信息**：
   - 项目类型（Web API、移动应用、桌面软件、CLI 工具）
   - 规模（个人项目、团队项目、企业级）
   - 部署环境（开发、测试、生产）

2. **技术栈**：
   - 编程语言和版本（Node.js 18、Python 3.11、Go 1.21）
   - 框架（Express 4.18、Django 4.2、Gin 1.9）
   - 数据库（PostgreSQL 15、MongoDB 6.0、Redis 7.0）
   - 其他关键依赖

3. **当前状态**：
   - 已有什么功能
   - 正在进行什么工作
   - 痛点和问题

4. **依赖关系**：
   - 需要哪些外部服务
   - API 集成
   - 第三方库

**示例**：

```markdown
# 任务目标
在现有的电商 API 中添加用户认证功能

## 技术上下文
- 项目类型：REST API（电商平台）
- 技术栈：
  - 后端：Node.js 18 + Express 4.18 + TypeScript 5.0
  - 数据库：PostgreSQL 15 + Prisma ORM
  - 缓存：Redis 7.0
  - 部署：Docker + AWS ECS
- 当前状态：
  - 已有用户注册功能（邮箱+密码）
  - 使用 Express Session（存储在 Redis）
  - 没有 token 机制，每次请求都查询 session
- 痛点：
  - Session 不支持分布式（多实例部署时有问题）
  - 无法支持移动端 App（需要无状态认证）
```

### 3. 范围边界原则（Scope）

**定义**：明确任务包含什么、不包含什么，定义清晰的边界。

**为什么重要**：防止范围蔓延，避免过度设计，确保按时交付。

**如何定义边界**：

1. **包含（In Scope）**：
   - 核心功能列表
   - 必须实现的特性
   - 关键交付物

2. **不包含（Out of Scope）**：
   - 未来可能添加的功能
   - 不需要的特性
   - 明确排除的内容

3. **影响范围**：
   - 修改哪些模块/文件
   - 不影响哪些模块
   - 向后兼容性

**示例**：

```markdown
## 范围边界

包含（In Scope）：
- ✅ 邮箱+密码登录
- ✅ JWT token 生成和验证
- ✅ Refresh token 机制（token 有效期 24h，refresh token 7天）
- ✅ 密码加密（bcrypt，salt rounds=10）
- ✅ 密码重置（发送邮件验证）
- ✅ Remember me 功能（延长 token 有效期到 30 天）

不包含（Out of Scope）：
- ❌ 社交登录（Google、Facebook、GitHub）
- ❌ 双因素认证（2FA）
- ❌ 生物识别（指纹、面部识别）
- ❌ 单点登录（SSO）
- ❌ 旧 Session 机制的迁移（保留兼容，新用户使用 JWT）

影响范围：
- 仅修改 `src/auth/` 模块
- 添加新的中间件 `src/middleware/jwt.ts`
- 不影响现有的业务逻辑（订单、支付、商品）
```

### 4. 可量化验收标准（Quantifiable Acceptance Criteria）

**定义**：每个任务必须有可验证的、可量化的成功标准。

**为什么重要**：模糊的标准无法判断是否完成，导致反复修改和验收失败。

**验收标准的类型**：

1. **功能标准**（exact_match）：
   - ✅ "登录成功返回 JWT token（格式：Bearer xxx）"
   - ✅ "密码错误返回 401，错误码 INVALID_CREDENTIALS"
   - ✅ "token 过期返回 401，错误码 TOKEN_EXPIRED"

2. **性能标准**（quantitative_threshold）：
   - ✅ "登录接口响应时间 < 200ms（P95）"
   - ✅ "支持并发 1000 req/s"
   - ✅ "数据库查询次数 ≤ 2 次"

3. **质量标准**（quantitative_threshold）：
   - ✅ "单元测试覆盖率 ≥ 90%"
   - ✅ "代码检查无 error，warning ≤ 5 个"
   - ✅ "所有 API 有 OpenAPI 文档"

4. **安全标准**（exact_match）：
   - ✅ "密码使用 bcrypt 加密（salt rounds=10）"
   - ✅ "token 使用 RS256 算法签名"
   - ✅ "敏感信息不记录到日志"

**验收标准模板**：

```markdown
## 验收标准

### 功能验收
1. **登录成功**：
   - 输入：正确的邮箱+密码
   - 输出：`{token: "xxx", refreshToken: "yyy", user: {...}, expiresAt: "2025-03-20T10:00:00Z"}`
   - 验证方式：手动测试 + 单元测试

2. **登录失败**：
   - 密码错误 → 401，错误码 `INVALID_CREDENTIALS`
   - 账号锁定 → 403，错误码 `ACCOUNT_LOCKED`
   - 账号不存在 → 404，错误码 `USER_NOT_FOUND`

### 性能验收
- 登录接口响应时间 < 200ms（P95），< 100ms（P50）
- 验证 token 响应时间 < 50ms
- 支持并发 1000 req/s 无错误

### 质量验收
- 单元测试覆盖率 ≥ 90%
- 集成测试覆盖所有 API 端点
- ESLint 检查 0 error，warning ≤ 3
- 所有函数有 JSDoc 注释

### 安全验收
- 密码使用 bcrypt 加密（salt rounds=10）
- Token 使用 RS256 签名
- 敏感字段（password、token）不记录到日志
- 通过 OWASP Top 10 检查
```

### 5. 结构化输出要求

**定义**：明确指定输出格式、数据结构、错误处理方式。

**为什么重要**：统一的输出格式便于前端/客户端处理，减少集成问题。

**如何指定输出**：

1. **成功响应格式**：
```json
{
  "success": true,
  "data": {
    "token": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...",
    "refreshToken": "xxx",
    "user": {
      "id": "123",
      "email": "user@example.com",
      "role": "user",
      "createdAt": "2025-01-01T00:00:00Z"
    },
    "expiresAt": "2025-03-20T10:00:00Z"
  }
}
```

2. **错误响应格式**：
```json
{
  "success": false,
  "error": {
    "code": "INVALID_CREDENTIALS",
    "message": "邮箱或密码错误",
    "details": null
  }
}
```

3. **错误码规范**：
```markdown
| 错误码 | HTTP 状态 | 说明 |
|--------|----------|------|
| INVALID_CREDENTIALS | 401 | 邮箱或密码错误 |
| TOKEN_EXPIRED | 401 | Token 已过期 |
| INVALID_TOKEN | 401 | Token 格式错误 |
| ACCOUNT_LOCKED | 403 | 账号已锁定 |
| USER_NOT_FOUND | 404 | 用户不存在 |
| INTERNAL_ERROR | 500 | 服务器内部错误 |
```

## 二、5W1H 框架详解

### What（目标）- 最关键

**要回答的问题**：
- 要实现什么功能？
- 预期交付成果是什么？
- 成功的标准是什么？

**示例**：
```markdown
# 任务目标（What）
实现基于 JWT 的用户认证系统，包括：
- 用户登录（邮箱+密码）
- Token 生成和验证
- Refresh token 机制
- 密码重置功能
```

---

### Why（动机）- 理解背景

**要回答的问题**：
- 为什么需要这个功能？
- 解决什么业务问题？
- 带来什么价值？

**示例**：
```markdown
## 背景和动机（Why）
当前使用 Session 认证存在以下问题：
1. 不支持分布式部署（Session 存储在单个实例）
2. 无法支持移动端 App（需要无状态认证）
3. 扩展性差（每次请求都查询 Redis）

期望收益：
- 支持水平扩展（无状态认证）
- 降低服务器负载（减少 Redis 查询）
- 统一 Web 和 App 的认证方式
```

---

### Who（受众）- 质量标准

**要回答的问题**：
- 目标用户是谁？
- 谁会使用这个功能？
- 对用户体验有什么要求？

**示例**：
```markdown
## 目标用户（Who）
- C端用户：普通消费者（对性能和易用性要求高）
- B端用户：企业客户（对安全性和稳定性要求高）
- 移动端用户：iOS/Android App 用户（需要无状态认证）

用户体验要求：
- 登录响应时间 < 200ms（用户无感知）
- 错误提示清晰友好（不暴露内部细节）
- 支持 remember me（减少重复登录）
```

---

### When（时间）- 优先级

**要回答的问题**：
- 什么时候需要完成？
- 优先级如何？
- 是否有里程碑节点？

**示例**：
```markdown
## 时间和优先级（When）
- 优先级：高（阻塞移动端 App 上线）
- 期望完成时间：2025-03-25
- 里程碑：
  - 2025-03-20：完成开发和单元测试
  - 2025-03-22：完成集成测试
  - 2025-03-24：完成安全审计
  - 2025-03-25：发布到生产环境
```

---

### Where（范围）- 定义边界

**要回答的问题**：
- 影响哪些模块/文件/系统？
- 部署在哪个环境？
- 与哪些组件交互？

**示例**：
```markdown
## 影响范围（Where）
修改的模块：
- `src/auth/`：认证核心逻辑
- `src/middleware/`：添加 JWT 中间件
- `src/routes/auth.ts`：认证相关路由

不影响的模块：
- `src/orders/`：订单模块
- `src/products/`：商品模块
- `src/payments/`：支付模块

部署环境：
- 开发环境：localhost
- 测试环境：staging.example.com
- 生产环境：api.example.com

依赖的组件：
- PostgreSQL（用户数据）
- Redis（黑名单 token）
- SMTP 服务（密码重置邮件）
```

---

### How（方式）- 技术偏好

**要回答的问题**：
- 技术方案偏好（如有）？
- 使用什么技术栈和工具？
- 有什么架构约束？

**示例**：
```markdown
## 技术方案（How）
认证方式：
- JWT（JSON Web Token）
- 算法：RS256（RSA 签名）
- Token 有效期：24 小时
- Refresh token 有效期：7 天

技术选型：
- jsonwebtoken 库（v9.0.0）
- bcrypt 库（v5.1.0）用于密码加密
- Prisma ORM 用于数据库操作

架构约束：
- 必须向后兼容现有 Session 认证（渐进式迁移）
- 不能修改现有的数据库表结构
- 必须通过 API Gateway 的安全检查
```

## 三、常见场景的最佳实践

### 场景1：代码实现任务

**模糊描述**："实现一个登录功能"

**优化后**：
```markdown
# 任务目标
实现基于 JWT 的用户登录功能

## 技术上下文
- 项目：电商 API（Node.js + Express + TypeScript）
- 数据库：PostgreSQL（已有 users 表）
- 当前状态：已有用户注册，缺少登录

## 具体要求
1. POST /api/auth/login 接口
2. 输入：{email, password}
3. 输出：{token, refreshToken, user, expiresAt}
4. 使用 bcrypt 验证密码
5. 返回 JWT token（RS256 签名）

## 验收标准
- 正确的邮箱密码返回 token
- 错误的密码返回 401
- 响应时间 < 200ms
- 单元测试覆盖率 ≥ 90%
```

---

### 场景2：性能优化任务

**模糊描述**："优化 API 性能"

**优化后**：
```markdown
# 任务目标
优化商品列表 API 的响应性能

## 背景和动机
当前商品列表接口响应时间 500-800ms，用户体验差。
目标：将响应时间降低到 200ms 以下。

## 技术上下文
- 接口：GET /api/products
- 当前瓶颈：
  - 数据库查询慢（N+1 问题）
  - 未使用缓存
  - 返回字段过多（包含完整商品详情）

## 优化方案
1. 修复 N+1 查询（使用 JOIN）
2. 添加 Redis 缓存（TTL 5分钟）
3. 简化返回字段（列表只返回摘要）
4. 添加数据库索引（product_id, category_id）

## 验收标准
- 平均响应时间 < 200ms（P50）
- P95 响应时间 < 300ms
- P99 响应时间 < 500ms
- 缓存命中率 > 80%
- 数据库查询次数从 N+1 降到 1
```

---

### 场景3：Bug 修复任务

**模糊描述**："修复登录 bug"

**优化后**：
```markdown
# 任务目标
修复用户登录时 token 过期导致的 500 错误

## 问题描述
- 复现步骤：
  1. 用户登录成功（获得 token）
  2. 等待 25 小时（token 过期）
  3. 使用过期 token 访问 /api/profile
  4. 返回 500 错误（预期应该返回 401）

## 根本原因
JWT 验证中间件未正确处理 TokenExpiredError，
导致抛出未捕获的异常。

## 修复方案
1. 在 jwt.middleware.ts 中添加错误处理
2. 捕获 TokenExpiredError 并返回 401
3. 错误响应：{code: "TOKEN_EXPIRED", message: "..."}

## 验收标准
- 过期 token 返回 401（不是 500）
- 错误码正确（TOKEN_EXPIRED）
- 添加单元测试覆盖此场景
- 不影响正常 token 的验证逻辑
```

---

### 场景4：架构重构任务

**模糊描述**："重构认证模块"

**优化后**：
```markdown
# 任务目标
重构认证模块，从 Session 迁移到 JWT

## 背景和动机
当前 Session 认证不支持分布式，需要迁移到无状态的 JWT。

## 技术上下文
- 当前架构：Express Session + Redis
- 目标架构：JWT + Refresh Token
- 用户量：10 万活跃用户
- 部署：3 个实例（负载均衡）

## 迁移策略（渐进式）
阶段1（本次任务）：
- 实现 JWT 认证模块（不影响现有 Session）
- 新用户使用 JWT，老用户继续用 Session
- 添加配置开关控制使用哪种认证

阶段2（未来）：
- 引导老用户重新登录（迁移到 JWT）
- 逐步下线 Session 支持

## 范围边界
包含：
- ✅ JWT 核心模块
- ✅ Refresh token 机制
- ✅ 兼容层（支持 Session 和 JWT 并存）

不包含：
- ❌ 老用户数据迁移（留到阶段2）
- ❌ Session 模块删除（留到阶段2）

## 验收标准
- 新用户注册自动使用 JWT
- 老用户登录仍使用 Session（不影响）
- 配置开关可动态切换
- 性能无退化（响应时间不增加）
- 所有现有测试通过
```

## 四、提示词检查清单

使用以下检查清单评估你的提示词质量：

### 清晰度检查 ✓

- [ ] 使用了具体动词（实现/添加/修复）而非模糊动词（处理/改进）
- [ ] 术语精确（如 "JWT" 而非 "认证"）
- [ ] 没有歧义表述（如 "优化" 已说明优化方向）
- [ ] 目标明确（知道要做什么）

### 完整性检查 ✓

- [ ] What（目标）已说明
- [ ] Why（动机）已说明
- [ ] Who（受众）已说明
- [ ] When（时间）已说明
- [ ] Where（范围）已说明
- [ ] How（技术方案）已说明

### 可执行性检查 ✓

- [ ] 可以分解为子任务
- [ ] 验收标准可量化（有具体数值）
- [ ] 验收标准可验证（可测试）
- [ ] 依赖关系明确
- [ ] 影响范围清晰

### 格式检查 ✓

- [ ] 使用 Markdown 格式
- [ ] 结构清晰（有标题层级）
- [ ] 关键信息前置（目标在最前面）
- [ ] 长度适中（≤500 字）

## 五、反模式（避免这样做）

### ❌ 反模式1：过于简单

**问题**："添加登录功能"

**为什么不好**：
- 没有技术方案（Session？JWT？OAuth？）
- 没有验收标准（如何判断完成？）
- 没有上下文（什么项目？什么技术栈？）

---

### ❌ 反模式2：过于复杂

**问题**：
```markdown
# 任务目标
实现一个完整的用户管理系统，包括注册、登录、权限管理、角色管理、
用户组管理、审计日志、单点登录、双因素认证、生物识别、社交登录、
邮箱验证、手机验证、实名认证、密码策略、账号锁定、IP 白名单...
（共 30 个功能）
```

**为什么不好**：
- 范围过大，无法在一个任务中完成
- 功能过多，难以聚焦
- 应该拆分为多个独立任务

---

### ❌ 反模式3：隐含假设

**问题**："优化数据库查询"

**隐含假设**：
- 假设是 MySQL（可能是 PostgreSQL、MongoDB）
- 假设是查询慢（可能是写入慢）
- 假设是性能问题（可能是逻辑错误）

**改进**：明确说明当前数据库、具体问题、优化目标。

---

### ❌ 反模式4：模糊的验收标准

**问题**：
```markdown
验收标准：
- 代码质量要好
- 性能要快
- 安全性要高
```

**为什么不好**：
- "好"、"快"、"高" 是主观的，无法量化
- 无法验证是否满足

**改进**：
```markdown
验收标准：
- 单元测试覆盖率 ≥ 90%
- 响应时间 < 200ms（P95）
- 通过 OWASP Top 10 安全检查
```

---

## 六、总结

### 好的提示词具备的特征

1. ✅ **清晰**：目标明确，术语精确，无歧义
2. ✅ **完整**：5W1H 全覆盖，上下文充分
3. ✅ **可执行**：可分解为任务，验收标准可量化
4. ✅ **结构化**：Markdown 格式，层次清晰
5. ✅ **聚焦**：范围明确，不过度设计

### 从模糊到明确的转换流程

```
模糊输入："优化代码"
    ↓
5W1H 分析：
  - What：优化什么方面？→ 性能
  - Why：为什么需要？→ 响应时间过长
  - Where：哪个模块？→ 商品列表 API
  - How：如何优化？→ 缓存 + 索引
  - When：什么时候完成？→ 本周五
  - Who：谁使用？→ C 端用户
    ↓
明确输出：
"优化商品列表 API 性能，将平均响应时间从 500ms 降低到 200ms 以下。
使用 Redis 缓存 + 数据库索引优化。目标用户：C 端消费者。
完成时间：本周五。验收标准：P95 响应时间 < 200ms，缓存命中率 > 80%。"
```

---

**最后的建议**：当不确定如何描述时，问自己：
- 如果我是执行者，这个描述能让我立即开始工作吗？
- 如果我完成了，能明确判断是否符合预期吗？

如果答案是"否"，说明提示词还需要进一步优化。
