# 任务规划方法论

## 需求分析框架

### 信息收集清单

在规划前必须收集：
- [ ] 用户的核心目标（做什么）
- [ ] 成功标准（怎样算完成）
- [ ] 技术约束（技术栈、版本、规范）
- [ ] 项目上下文（现有代码、架构）
- [ ] 时间和范围约束

### 需求澄清原则

- 明确的需求直接规划
- 模糊的需求必须提问确认
- 隐含的约束需主动识别（从 CLAUDE.md、代码规范等推断）
- 不假设用户意图，宁可多问一句

## 任务分解策略

### 按功能分解
适用于功能开发任务：
```
新功能 → 数据层 + 逻辑层 + 接口层 + 测试
```

### 按阶段分解
适用于重构和迁移任务：
```
迁移 → 准备 → 核心迁移 → 适配 → 验证 → 清理
```

### 按文件分解
适用于大范围代码变更：
```
变更 → 文件A修改 + 文件B修改 + 文件C新建 + 测试更新
```

### 分解粒度判断

| 条件 | 粒度 |
|------|------|
| 可在一次交互中完成 | 合适 |
| 需要多次交互 | 需要进一步分解 |
| 只需改 1-2 行代码 | 过细，可合并 |
| 涉及多个不相关文件 | 需要进一步分解 |

## DAG 依赖建模

### 依赖类型

- **数据依赖**：B 需要 A 的输出作为输入
- **顺序依赖**：B 必须在 A 之后执行（如先建表再插数据）
- **资源依赖**：A 和 B 操作同一文件，不能并行

### 并行识别

两个任务可并行的条件：
1. 无数据依赖
2. 无顺序依赖
3. 不操作同一文件
4. 不修改同一模块的公共接口

### 关键路径

- 最长的依赖链决定最短完成时间
- 关键路径上的任务应优先执行
- 非关键路径的任务有灵活的执行时机

## 执行策略设计

### Agent 分配

| 任务类型 | 推荐 Agent |
|---------|-----------|
| 代码实现 | executor |
| 测试编写 | executor |
| 问题修复 | debugger |
| 质量审查 | reviewer |
| 多任务调度 | orchestrator |

### 并行分组

```
组 1（并行）：无依赖的任务
    ↓ 等待全部完成
组 2（并行）：依赖组 1 的任务
    ↓ 等待全部完成
组 3（串行）：最终集成和测试
```

## 意图分类框架

规划前先对任务进行分类，决定规划深度：

### 意图类型判断

通过以下信号判断任务意图：

| 信号 | 判断 |
|------|------|
| "改一下"、"修个 typo" | → `trivial`，跳过规划 |
| "这个 bug"、"报错了" | → `fix`，简单规划 |
| "重构"、"优化"、"整理" | → `refactor`，标准规划 |
| "新功能"、"添加"、"实现" | → `feature`，完整规划 |
| "架构"、"迁移"、"重新设计" | → `architecture`，深度规划 |
| "为什么"、"调查"、"分析" | → `investigation`，探索性规划 |

### 复杂度评估

评估维度：
- 涉及多少文件？（1-2 / 3-5 / 5+）
- 是否跨模块/跨包？
- 是否有外部依赖？
- 是否需要修改公共接口？
- 是否需要数据迁移？

### 规划深度决策

```
trivial + L1 → 不规划，直接执行
fix + L1/L2  → 简单规划（1-2 个子任务）
refactor/feature + L2 → 标准规划
feature/architecture + L3/L4 → 完整规划 + reviewer 预审
investigation → 探索性规划（可能需要多轮信息收集）
```

## 规划输出

规划完成后通过 `TaskCreate` 注册所有子任务，通过 `TaskUpdate` 建立依赖关系和附加 metadata。不将计划写入文件，TaskList 是唯一的任务状态来源。

## 风险评估

每个计划应识别：
- **高风险子任务**：可能失败或耗时长的任务
- **依赖风险**：外部依赖不可控
- **备选方案**：高风险任务的替代路径
