# 任务规划方法论

## 需求分析框架

### 信息收集清单

在规划前必须收集：
- [ ] 用户的核心目标（做什么）
- [ ] 成功标准（怎样算完成）
- [ ] 技术约束（技术栈、版本、规范）
- [ ] 项目上下文（现有代码、架构）
- [ ] 时间和范围约束

### 需求澄清原则

- 明确的需求直接规划
- 模糊的需求必须提问确认
- 隐含的约束需主动识别（从 CLAUDE.md、代码规范等推断）
- 不假设用户意图，宁可多问一句

## 任务分解策略

### 按功能分解
适用于功能开发任务：
```
新功能 → 数据层 + 逻辑层 + 接口层 + 测试
```

### 按阶段分解
适用于重构和迁移任务：
```
迁移 → 准备 → 核心迁移 → 适配 → 验证 → 清理
```

### 按文件分解
适用于大范围代码变更：
```
变更 → 文件A修改 + 文件B修改 + 文件C新建 + 测试更新
```

### 分解粒度判断

| 条件 | 粒度 |
|------|------|
| 可在一次交互中完成 | 合适 |
| 需要多次交互 | 需要进一步分解 |
| 只需改 1-2 行代码 | 过细，可合并 |
| 涉及多个不相关文件 | 需要进一步分解 |

## DAG 依赖建模

### 依赖类型

- **数据依赖**：B 需要 A 的输出作为输入
- **顺序依赖**：B 必须在 A 之后执行（如先建表再插数据）
- **资源依赖**：A 和 B 操作同一文件，不能并行

### 并行识别

两个任务可并行的条件：
1. 无数据依赖
2. 无顺序依赖
3. 不操作同一文件
4. 不修改同一模块的公共接口

### 关键路径

- 最长的依赖链决定最短完成时间
- 关键路径上的任务应优先执行
- 非关键路径的任务有灵活的执行时机

## 执行策略设计

### Agent 分配

| 任务类型 | 推荐 Agent |
|---------|-----------|
| 代码实现 | executor |
| 测试编写 | executor |
| 问题修复 | debugger |
| 质量审查 | reviewer |
| 多任务调度 | orchestrator |

### 并行分组

```
组 1（并行）：无依赖的任务
    ↓ 等待全部完成
组 2（并行）：依赖组 1 的任务
    ↓ 等待全部完成
组 3（串行）：最终集成和测试
```

## 意图分类框架

规划前先对任务进行分类，决定规划深度：

### 意图类型判断

通过以下信号判断任务意图：

| 信号 | 判断 |
|------|------|
| "改一下"、"修个 typo" | → `trivial`，跳过规划 |
| "这个 bug"、"报错了" | → `fix`，简单规划 |
| "重构"、"优化"、"整理" | → `refactor`，标准规划 |
| "新功能"、"添加"、"实现" | → `feature`，完整规划 |
| "架构"、"迁移"、"重新设计" | → `architecture`，深度规划 |
| "为什么"、"调查"、"分析" | → `investigation`，探索性规划 |

### 复杂度评估

评估维度：
- 涉及多少文件？（1-2 / 3-5 / 5+）
- 是否跨模块/跨包？
- 是否有外部依赖？
- 是否需要修改公共接口？
- 是否需要数据迁移？

### 规划深度决策

```
trivial + L1 → 不规划，直接执行
fix + L1/L2  → 简单规划（1-2 个子任务）
refactor/feature + L2 → 标准规划
feature/architecture + L3/L4 → 完整规划 + reviewer 预审
investigation → 探索性规划（可能需要多轮信息收集）
```

## 规划输出

规划完成后通过 `TaskCreate` 注册所有子任务，通过 `TaskUpdate` 建立依赖关系和附加 metadata。不将计划写入文件，TaskList 是唯一的任务状态来源。

## 风险评估

每个计划应识别：
- **高风险子任务**：可能失败或耗时长的任务
- **依赖风险**：外部依赖不可控
- **备选方案**：高风险任务的替代路径

## 执行过程检查清单

### 需求分析阶段
- [ ] 收集用户核心目标（做什么）
- [ ] 明确成功标准（怎样算完成）
- [ ] 识别技术约束（技术栈、版本、规范）
- [ ] 了解项目上下文（现有代码、架构）
- [ ] 确认时间和范围约束
- [ ] 模糊需求已通过提问确认
- [ ] 隐含约束已从 CLAUDE.md/代码规范推断

### 任务分解阶段
- [ ] 选择合适的分解策略（功能/阶段/文件）
- [ ] 任务粒度适中（可在一次交互完成）
- [ ] 每个子任务符合单一职责原则
- [ ] 每个子任务有明确的输入和输出
- [ ] 验收标准可自动验证（优先级：测试 > 输出检查 > 人工确认）

### DAG 依赖建模阶段
- [ ] 识别所有依赖关系（数据/顺序/资源）
- [ ] 确认依赖图无环（DAG）
- [ ] 识别可并行任务（无依赖、不操作同一文件）
- [ ] 并行任务数不超过 2
- [ ] 关键路径已识别

### 执行策略设计阶段
- [ ] 为每个任务分配合适的 Agent
- [ ] 并行任务已分组
- [ ] 串行任务顺序合理
- [ ] 每个任务的 metadata 包含 target_files

### 风险评估阶段
- [ ] 识别高风险子任务
- [ ] 识别依赖风险
- [ ] 准备备选方案

### 意图分类阶段
- [ ] 正确判断任务意图（trivial/fix/refactor/feature/architecture/investigation）
- [ ] 复杂度评估准确（L1-L4）
- [ ] 规划深度决策合理

## 完成后检查清单

### 规划输出检查
- [ ] 所有子任务已通过 `TaskCreate` 注册
- [ ] 依赖关系已通过 `TaskUpdate` 建立
- [ ] 每个任务的 metadata 包含必要信息（target_files、retry_count）
- [ ] 任务状态初始化为 `pending`
- [ ] 未将计划写入文件（TaskList 是唯一状态来源）

### 规划质量检查
- [ ] 所有子任务覆盖完整需求（无遗漏）
- [ ] 任务粒度适中（不过大不过小）
- [ ] 验收标准清晰可验证
- [ ] 依赖关系准确无误
- [ ] 并行任务不操作同一文件或模块

### 用户确认检查
- [ ] 规划已展示给用户
- [ ] 格式清晰易读（子任务列表、依赖图、执行策略）
- [ ] 用户已确认规划合理
- [ ] 用户已确认可通过 `/exec` 开始执行

### 风险识别检查
- [ ] 高风险子任务已标注
- [ ] 依赖风险已说明
- [ ] 备选方案已准备（如适用）
