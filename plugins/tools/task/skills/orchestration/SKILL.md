# 多任务编排规范

## 编排模式

### 模式 1：顺序执行
```
T1 → T2 → T3
```
**适用**：任务间有强数据依赖。
**调度**：逐个执行，前一个完成才开始下一个。

### 模式 2：并行分组
```
     ┌→ T1 ─┐
Start┤       ├→ T3 → End
     └→ T2 ─┘
```
**适用**：存在独立子任务可并行。
**调度**：同一组内并行执行（最多 2 个），组间串行。

### 模式 3：流水线
```
Plan → [Execute Group 1] → Review → [Execute Group 2] → Review → Final Review
```
**适用**：大型任务需要分阶段交付和审查。
**调度**：每个阶段完成后审查，确认无问题再进入下一阶段。

### 模式 4：迭代（Loop）
```
while !done:
    Gather → Act → Verify → Adjust
```
**适用**：目标明确但路径不确定（修 bug、优化性能）。
**调度**：单线程循环，每次迭代一个逻辑步骤。

## 并行调度规则

### 硬性约束
1. **最大并行数：2** — 不可超过
2. **文件隔离** — 并行任务不操作同一文件
3. **模块隔离** — 并行任务不修改同一模块的公共接口
4. **依赖前置** — 所有前置依赖必须完成才能开始

### 并行分配策略
- 将无依赖的任务归入同一并行组
- 每组最多 2 个任务
- 优先将关键路径上的任务排在前面
- 短任务和长任务搭配，避免等待浪费

## 状态追踪

### 状态表格式
```
| # | 子任务 | 状态 | 依赖 | 验收 |
|---|--------|------|------|------|
| T1 | 标题 | pending/in_progress/completed/failed/blocked | - | 0/N |
```

### 状态转换规则
- `pending → in_progress`：所有依赖已 completed
- `in_progress → completed`：所有验收标准通过
- `in_progress → failed`：执行出错或验收不通过
- `failed → in_progress`：重试（最多 2 次）
- `blocked → in_progress`：阻塞原因已解除

## 异常处理

### 子任务失败
1. 记录失败原因
2. 评估是否可重试（代码错误 → 可重试，设计问题 → 不可重试）
3. 可重试：交给 debugger 修复后重试（最多 2 次）
4. 不可重试：暂停执行，向用户报告

### 阻塞处理
1. 识别阻塞原因（依赖未完成、外部资源不可用）
2. 检查是否有可绕过的替代方案
3. 无替代方案：标记为 blocked，继续执行其他不受影响的任务
4. 向用户报告阻塞情况

### 级联失败
- 子任务失败后，检查依赖它的后续任务
- 后续任务标记为 blocked
- 修复失败任务后，自动解除 blocked 状态

## 进度报告

每完成一个执行组，输出进度摘要：

```
[进度] ████████░░░░░░░░ 50%  (3/6 子任务完成)
当前：执行组 2 (T4, T5 并行执行中)
阻塞：无
预计剩余：2 个执行组
```
