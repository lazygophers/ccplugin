# 任务验收规范

## 验收标准定义原则

好的验收标准必须是：
- **可自动化**：能通过运行命令或脚本验证
- **可观测**：有明确的通过/失败判断
- **独立**：不依赖其他标准的结果
- **具体**：描述具体的检查项，而非模糊的"质量好"

### 验收标准示例

| 好的验收标准 | 差的验收标准 |
|------------|------------|
| `pytest tests/test_user.py` 全部通过 | 测试通过 |
| `ruff check src/` 无错误 | 代码规范 |
| API 返回 200 且包含 `user_id` 字段 | API 正常工作 |
| 文件 `config.yaml` 包含 `database` 配置块 | 配置已添加 |

## 验证方法优先级

1. **自动化测试**（最优）：运行 `pytest`/`go test`/`vitest` 等
2. **命令输出检查**：运行 lint、build 等命令检查输出
3. **文件状态检查**：检查文件是否存在、内容是否正确
4. **手动确认**（最后手段）：需要用户确认的主观标准

## 质量检查清单

### 代码质量
- [ ] lint 检查通过
- [ ] 格式化检查通过
- [ ] 无编译/类型错误
- [ ] 无安全漏洞（注入、XSS 等）
- [ ] 无明显性能问题

### 测试覆盖
- [ ] 正常路径测试
- [ ] 边界条件测试
- [ ] 异常/错误路径测试
- [ ] 测试遵循 AAA 模式（Arrange-Act-Assert）

### 功能完整性
- [ ] 所有要求的功能点已实现
- [ ] 输入验证完备
- [ ] 错误处理合理
- [ ] 与现有代码集成无冲突

## 审查报告格式

```markdown
# 审查报告：[任务标题]

## 总体评价：[通过/有条件通过/不通过]

## 详细审查

### T[N]: [标题]
**状态**：通过/不通过
**验收标准**：
- [x] 标准描述
- [ ] 标准描述 - 失败原因

**代码质量**：
- lint: 通过/N 个问题
- 测试: N/M 通过
- 安全: 无问题/N 个风险

**改进建议**：
- [建议]

## 汇总
- 必须修复：N 项
- 建议改进：N 项
- 总体结论：[通过/需修复]
```

## 验收决策

| 结果 | 定义 | 后续动作 |
|------|------|---------|
| **通过** | 所有必选标准通过 | 标记完成 |
| **有条件通过** | 必选通过，可选有未通过 | 用户决定 |
| **不通过** | 有必选标准未通过 | 修复后重新审查 |
